<TLDR>Agent S is an MLLM-based desktop GUI agent that layers retrieval over self-evaluated, abstraction-stratified memory plus live web search onto a bounded language-level <Term def="Agent-Computer Interface: an abstraction layer that gives an MLLM agent a bounded action set and a grounding-friendly observation, instead of raw coordinates and arbitrary code">Agent-Computer Interface (ACI)</Term>, nearly doubling OSWorld success over the OSWorld-agent baseline.</TLDR>

<WhyItMatters>A foundational OSWorld-era computer-use agent and the first to port SWE-agent's "ACI" abstraction into the desktop GUI setting. Its Manager/Worker/Self-Evaluator decomposition, narrative-vs-episodic memory split, and planning/grounding/execution error taxonomy are directly reusable harness patterns.</WhyItMatters>

## Motivation
Automating real desktop tasks via direct mouse/keyboard control faces three pressures (§1): <Problem>domain knowledge</Problem> — apps and websites change constantly, so the agent needs up-to-date knowledge it cannot hold statically in weights; <Problem>long task horizons</Problem> — desktop tasks are multi-step with order-sensitive, interdependent actions; and <Problem>dynamic, non-uniform interfaces</Problem> — the agent must parse large visual+textual state and pick the right element from a vast action space. Underneath: GUIs are built for humans (real-time perception) or scripts (APIs), but an MLLM is a third user type that acts in slow discrete steps, lacks an internal coordinate system, and cannot process per-pixel feedback (§3.3). Prior GUI agents are reported bottlenecked specifically on <Problem>grounding</Problem> (§3.3, citing Xie et al. 2024; Zheng et al. 2024a).

## Contribution (what's new)
The genuine delta: a GUI agent improves most not from a smarter single policy but from <Novelty>layering retrieval of self-evaluated, abstraction-stratified experience onto a bounded language-level action interface</Novelty>. Two design commitments follow mechanistically:
- <Novelty>Memory is stratified by abstraction and matched to stage</Novelty>: high-level full-task "narrative" memory (actions stripped) feeds the planner; low-level grounded "episodic" memory feeds the executor. A single memory would either pollute planning with brittle coordinates or starve execution of step detail (§3.1.1–3.1.2).
- <Novelty>Grounding is delegated to the accessibility tree, not asked of the MLLM as coordinates</Novelty>: elements get integer IDs, and rather than augmenting the image with tree marks (prior Set-of-Marks work), Agent S augments the *tree* with OCR text blocks so missing elements become referenceable (§3.3).

This is distinct from prior RAG-for-agents (RAP/AutoGuide/RADA: single-level exemplars or guidelines) and from SWE-agent's ACI (code, not GUI dual-input). Note the abstract's framing of "extensive experiments / SOTA" as a contribution is evidence, not novelty.

## Research questions
Can a frozen MLLM control a full desktop GUI well by (1) retrieving stratified self-built experience and live web knowledge rather than fine-tuning, and (2) acting through a bounded, ID-grounded interface — and which components actually drive the gains?

## Methodology
A closed-loop hierarchical architecture (Fig.3): a **Manager** plans by fusing web search and narrative memory into a subtask queue; sequential **Workers** execute each subtask using episodic memory plus in-episode reflection, emitting one grounded action per step; a **Self-Evaluator** summarizes completed episodes into textual "rewards" written back to memory. Backbones are frozen — "learning" is retrieval over a continually-updated memory, bootstrapped by self-supervised exploration on synthetic tasks.

## Key findings
- <Finding src="Tab.2, p.9">Web knowledge is the single most impactful experience component</Finding>: on the 65-task `testsub`, removing it drops Agent S from 26.15% to 16.80% (vs w/o Episodic 18.46%, w/o Narrative 21.41%, w/o All 13.85%, baseline 10.77%). But on OS, "w/o Web Knowledge" falls *below* baseline (16.60 vs 33.33) — web guidance can mislead in some categories.
- <Finding src="Fig.6, p.9">Retrieval and ACI are synergistic, not additive</Finding>: baseline 10.77% → +Retrieval 12.31% → ACI-only 12.31% → ACI+Retrieval 20.00% → full Agent S (adds hierarchical planning) 26.15%. Retrieval barely helps without ACI but jumps once ACI is present; hierarchical planning adds the final 20.00→26.15.
- <Finding src="Fig.7, p.9">Self-supervised exploration is the most load-bearing memory mechanism, and summaries beat raw trajectories</Finding>: full 26.15% → w/o Continual Update 23.18% → w/o Self-Evaluator 18.46% → w/o Exploration 15.38%. Replacing summaries with full trajectories (w/o Self-Evaluator) costs ~8 pts.
- <Finding src="Tab.3, p.10">Execution and grounding errors dominate failures</Finding>: of failed `testsub` tasks, execution errors 79.59%, grounding 53.06%, planning 34.69% (overlapping). Grounding stays the practical bottleneck the ACI only partially fixes.
- <Finding src="Tab.4, p.10">Generalization to Windows is real but uneven</Finding>: 18.2% vs NAVI 13.3% with no adaptation, driven by Windows System (29.2→45.8) and Coding (9.1→29.2), but Agent S is *worse* on Web Browser (20.0→13.3) and Media&Video (25.3→19.1), and 0.0% on Office.
- <Finding src="Tab.8, p.16">Per-app gains are concentrated, and one app regresses</Finding>: large jumps in VS Code (30.43→52.17), Thunderbird (6.67→40.00), GIMP (0.00→23.08), and OS, but LibreOffice Calc *regresses* (4.26→2.13) — the OS-category headline is dominated by one app's leap.

## Implications
Establishes that retrieval-as-learning over stratified memory plus a bounded ACI is a strong, fine-tuning-free recipe for desktop control, extending the <Related>SWE-agent ACI line (Yang et al. 2024)</Related> from code to GUIs and the <Related>RAG-for-agents line (RAP, AutoGuide, RADA)</Related> with two-level, self-evaluated experience. It also recombines the broader cognitive-architecture lineage (<Related>Reflexion, ReAct, Voyager, Generative Agents; contemporaneous OS-Copilot, Synapse, Cradle</Related>) rather than introducing a new learning mechanism. Worth chasing: <FollowUp>cost/step-count Pareto-optimality, explicitly flagged as future work</FollowUp>; <FollowUp>whether the grounding bottleneck is fixed by later grounding-specialized models or Agent S2</FollowUp>; <FollowUp>head-to-head against the contemporaneous GUI agents it names but never compares to</FollowUp>.

## Limitations
- <Caveat>No variance, seeds, or significance anywhere</Caveat> — all numbers appear single-run at low absolute success (~20%), so a few tasks can swing a category.
- <Caveat>Cost, latency, and step-count are unmeasured</Caveat> (authors flag this, p.10) — a major omission for practical computer-use given the slow one-action-per-step loop.
- <Caveat>No head-to-head with the contemporaneous GUI agents</Caveat> (Synapse, OS-Copilot, Cradle) it names as closest work.
- <Caveat>Heavy dependence on accessibility-tree quality</Caveat> limits applicability to canvas-heavy / custom-rendered apps (consistent with GIMP, Calc, and Windows web-browser weaknesses).
- <Caveat>Abstract and intro miscite OSWorld as "(OpenAI, 2023)"</Caveat> — it is Xie et al. 2024 (minor but notable).

## Critiques and Questions
1. Self-evaluation labels success/failure with no ground truth; mislabeled successes would silently poison memory, and this is never measured.
2. Self-supervised exploration tasks are generated from the benchmarks' own initial environments — possible leakage between exploration and test tasks.
3. The `testsub` of 65 is the authors' own resampling (the official 39-task "small" set is called too imbalanced, footnote p.8); ablations run only on GPT-4o and only on this subset.
4. The Trajectory Reflector does not reliably break documented grounding→repeated-action→execution-error loops (Figs.14,16).

## Glossary
- <Term def="A continually-updated store of high-level, abstractive full-task experience summaries (success and failure, with specific actions removed), keyed by an observation-aware query; feeds the Manager's planning.">Narrative Memory</Term>
- <Term def="A store of complete subtask plans with grounded actions, only from subtasks marked DONE, keyed by query+subtask+context; feeds the Worker's execution.">Episodic Memory</Term>
- <Term def="A live web-search engine (Perplexica) the Manager queries in a 'How to do X' format for up-to-date, app-specific knowledge.">Online Web Knowledge</Term>
- <Term def="A prompting technique that overlays numbered marks on a screenshot so the model can refer to elements by mark; Agent S inverts this by augmenting the accessibility tree with OCR text instead.">Set-of-Marks</Term>
- <Term def="Executable benchmarks of real desktop apps with success-rate checkers: OSWorld (Ubuntu, 369 tasks) and WindowsAgentArena (Windows, 154 tasks).">OSWorld / WindowsAgentArena</Term>
