<TLDR>A compositional computer-use agent that routes each action's <Term def="mapping a UI element description to exact pixel coordinates">grounding</Term> to a specialist expert (visual / OCR / spreadsheet-API) and re-plans subgoals after every step, setting new SOTA on OSWorld, WindowsAgentArena, and AndroidWorld.</TLDR>

<WhyItMatters>The lineage successor to Agent S from an open-code lab; its "compose cheap specialists to beat the best monolith" thesis is directly relevant to GUI-agent harness design (planner/worker/grounder separation and routing).</WhyItMatters>

## Motivation
Computer-use agents must do three cognitively distinct things at once — long-horizon planning, action generation, and pixel-precise grounding — yet prior work pushes all three through a single model. That monolith fails on two fronts: it <Claim>imprecisely grounds GUI elements to coordinates</Claim> and <Claim>struggles on long-horizon tasks amid pop-ups and evolving screens</Claim>. A structural reason: fine-tuning one model to specialize on a sub-skill erodes its other abilities (catastrophic forgetting), so a generalist is bottlenecked by its weakest sub-skill. The best agents still trail humans by ~40% on OSWorld (p.1). (Problem only — not the fix.)

## Contribution (what's new)
Two transferable mechanisms layered on the inherited Agent S Manager/Worker + knowledge base:
- <Claim>**Mixture-of-Grounding**: treat grounding as a routing problem</Claim> — the Worker gates each action to a specialist tool because a single visual grounder misses fine targets (exact word edges, resized cells) that non-MLLM tools handle better (OCR for spans, a spreadsheet API for cells).
- <Claim>**Proactive Hierarchical Planning**: re-plan after every subgoal, not just on failure</Claim> — a reactive plan cannot absorb information learned from a *successful* subgoal, so proactive re-planning enables backward correction and adaptive exploration as the partial screen is revealed.

The core empirical claim: <Claim>composing individually-suboptimal specialists can beat the best monolithic model</Claim>.

## Research questions
Can splitting cognition across a generalist planner/worker and specialist grounders — plus per-subgoal re-planning — outperform monolithic native agents on real computer-use tasks, and does each component contribute?

## Methodology
A **Manager** decomposes the instruction into subgoals; a **Worker** executes the top subgoal as atomic actions, each tagged with a language descriptor of its target, and routes that descriptor to a grounding expert rather than localizing itself. After each subgoal returns SUCCESS/FAILURE, the Manager regenerates the remaining subgoals from the latest observation. Screenshot-only input; no fine-tuning (off-the-shelf backbones + pretrained grounders).

<Pass title="Method details">
Three grounding experts:
- **Visual** — description → ⟨x,y⟩ pixel coordinate, screenshot-only (UI-TARS-72B-DPO in their setup).
- **Textual** — Tesseract OCR returns start/end coordinates of a word span given two boundary phrases (`highlight_text_span`).
- **Structural** — UNO interface programmatically writes spreadsheet cells from a `{cell:value}` dict (`set_cell_values`), assuming a UNO-controllable app (LibreOffice).

## Implications
A testable case that modular composition of off-the-shelf models beats monolithic native agents on computer use, and that grounding is better framed as per-action routing than as one skill. For harness design: separate planner/worker/grounder, route grounding to the cheapest tool that works, and re-plan per subgoal. Lowers the adoption barrier by reusing commodity OCR/UNO and open code.

## Limitations
- <Limit>No variance/error bars; headline and ablation numbers appear single-run, so small SOTA margins (e.g. +1.9 pts on 50-step OSWorld) could be within noise.</Limit>
- <Limit>Ablations on a 65-task subset, not the full 369.</Limit>
- <Limit>Cost/latency of per-subgoal re-planning and per-action routing is never quantified.</Limit>
- <Limit>Structural grounding is tied to UNO-controllable office apps — not a general mechanism.</Limit>
- <Limit>By the authors' own error analysis, planning is now the dominant failure (41.0%) — the framework moves the bottleneck rather than removing it; still ~40% below humans.</Limit>

## Critiques and Questions
1. With single-run numbers and a 1.9-pt 50-step margin, how much of the SOTA is noise? Re-run with seeds.
2. Routing is itself an LLM decision — what is the misroute rate, and how often does a wrong expert choice cause failure?
3. What is the latency/cost multiplier of re-planning after every subgoal vs reactive? The obvious price of the design is unmeasured.
4. Does the compositional advantage survive when grounders are also weak, or does it depend on a strong UI-TARS visual expert?
5. How does the v1→S2 delta isolate against the original Agent S? Check the repo for Manager/Worker/router prompts and the 65-task subset selection.

## Glossary
<Term def="autonomous agent that completes tasks by driving a GUI with keyboard/mouse, no app-specific APIs">Computer-use agent</Term> ·
<Term def="Worker acts as a gate routing each action's localization to a specialist grounder (visual / OCR-textual / spreadsheet-structural), analogous to Mixture-of-Experts">Mixture-of-Grounding (MoG)</Term> ·
<Term def="Manager re-generates remaining subgoals after every completed subgoal using the latest observation, enabling backward correction; vs reactive plans that revise only on failure">Proactive Hierarchical Planning (PHP)</Term> ·
<Term def="hierarchical roles — Manager generates high-level subgoals; Worker executes the current subgoal as atomic actions and routes their grounding">Manager / Worker</Term> ·
<Term def="single model post-trained to do planning + grounding + execution end-to-end, e.g. UI-TARS, Claude Computer Use, OpenAI CUA">Native / monolithic agent</Term> ·
<Term def="Universal Network Objects — a programmatic interface to LibreOffice used by the structural grounder to set cell values directly">UNO</Term> ·
<Term def="structured machine-readable representation of on-screen UI elements; some baselines feed it, Agent S2 avoids it">Accessibility (a11y) tree</Term> ·
<Term def="benchmark of 369 real Ubuntu desktop tasks, functional success scoring; run at 15- and 50-step caps">OSWorld</Term>
