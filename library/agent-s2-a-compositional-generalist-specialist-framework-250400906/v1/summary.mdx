## Motivation
Computer-use agents must do three cognitively distinct things at once — long-horizon planning, action generation, and pixel-precise grounding — yet prior work pushes all three through a single model. That monolith fails on two fronts: it <Problem>imprecisely grounds GUI elements to coordinates</Problem> and <Problem>struggles on long-horizon tasks amid pop-ups and evolving screens</Problem>. A structural reason: <Problem>fine-tuning one model to specialize on a sub-skill erodes its other abilities (catastrophic forgetting)</Problem>, so a generalist is bottlenecked by its weakest sub-skill. The best agents still trail humans by ~40% on OSWorld (p.1). (Problem only — not the fix.)

## Contribution (what's new)
Two transferable mechanisms layered on the inherited Agent S Manager/Worker + knowledge base:
- <Novelty>**Mixture-of-Grounding**: treat grounding as a routing problem</Novelty> — the Worker gates each action to a specialist tool because a single visual grounder misses fine targets (exact word edges, resized cells) that non-MLLM tools handle better (OCR for spans, a spreadsheet API for cells).
- <Novelty>**Proactive Hierarchical Planning**: re-plan after every subgoal, not just on failure</Novelty> — a reactive plan cannot absorb information learned from a *successful* subgoal, so proactive re-planning enables backward correction and adaptive exploration as the partial screen is revealed.

The core empirical claim: <Novelty>composing individually-suboptimal specialists can beat the best monolithic model</Novelty>.

## Research questions
Can splitting cognition across a generalist planner/worker and specialist grounders — plus per-subgoal re-planning — outperform monolithic native agents on real computer-use tasks, and does each component contribute?

## Methodology
A **Manager** decomposes the instruction into subgoals; a **Worker** executes the top subgoal as atomic actions, each tagged with a language descriptor of its target, and routes that descriptor to a grounding expert rather than localizing itself. After each subgoal returns SUCCESS/FAILURE, the Manager regenerates the remaining subgoals from the latest observation. Screenshot-only input; no fine-tuning (off-the-shelf backbones + pretrained grounders).

<Pass title="Method details">
Three grounding experts:
- **Visual** — description → ⟨x,y⟩ pixel coordinate, screenshot-only (UI-TARS-72B-DPO in their setup).
- **Textual** — Tesseract OCR returns start/end coordinates of a word span given two boundary phrases (`highlight_text_span`).
- **Structural** — UNO interface programmatically writes spreadsheet cells from a `{cell:value}` dict (`set_cell_values`), assuming a UNO-controllable app (LibreOffice).

## Key findings
- <Finding src="Fig.6, p.8">A small open-source specialist grounder beats the best closed generalist as the grounder</Finding>: in-framework UI-TARS-7B-DPO scores 29.23% vs Claude-3.7 24.61% (UGround-V1-7B 24.61%, UI-TARS-72B-DPO 30.77%) — direct evidence for the "suboptimal specialists beat the monolith" thesis.
- <Finding src="Fig.8, p.9">MoG shifts the bottleneck off grounding</Finding>: planning is now the top failure (41.0%) while grounding is only 20.5% — a reversal of prior work where grounding dominated, so the framework moves rather than removes the limit.
- <Finding src="Fig.5, p.8">Both mechanisms help more at longer horizons</Finding>: MoG adds 27.69→30.77% @15 and 33.85→38.46% @50; proactive vs reactive planning adds +4.62% @15 and +6.15% @50 (65-task subset).
- <Finding src="§4.3, p.8">Each grounding expert measurably lifts its own subtasks</Finding>: removing textual grounding drops subtask success 70.6→65.2%; removing structural grounding drops it 73.7→69.4%.
- <Finding src="§4.2, p.7">Specialist routing explains a backbone surprise</Finding>: Claude-3.5 (new) beats Claude-3.7 in the Office category because it routes to Textual/Structural experts almost twice as often.
- <Finding src="Fig.7, p.9">Test-time scaling (15→50 steps) works via four behaviors</Finding>: of tasks fixed by more steps, gains come from Adaptive Navigation (34.5%), Adaptive Interaction (31.0%), Backward Correction (13.8%), and Task Complexity (10.3%) — all enabled by proactive replanning.

## Implications
A testable case that modular composition of off-the-shelf models beats monolithic native agents on computer use, and that grounding is better framed as per-action routing than as one skill. For harness design: separate planner/worker/grounder, route grounding to the cheapest tool that works, and re-plan per subgoal. Lowers the adoption barrier by reusing commodity OCR/UNO and open code. <Related>Direct successor to Agent S (Agashe et al. 2024); reuses its Manager/Worker + knowledge base, and competes with native monoliths UI-TARS / Claude Computer Use / OpenAI CUA.</Related> <FollowUp>The repo (simular-ai/Agent-S) holds the Manager/Worker/router prompts and the 65-task subset selection worth inspecting.</FollowUp>

## Limitations
- <Caveat>No variance/error bars; headline and ablation numbers appear single-run, so small SOTA margins (e.g. +1.9 pts on 50-step OSWorld) could be within noise.</Caveat>
- <Caveat>Ablations on a 65-task subset, not the full 369.</Caveat>
- <Caveat>Cost/latency of per-subgoal re-planning and per-action routing is never quantified.</Caveat>
- <Caveat>Structural grounding is tied to UNO-controllable office apps — not a general mechanism.</Caveat>
- <Caveat>By the authors' own error analysis, planning is now the dominant failure (41.0%) — the framework moves the bottleneck rather than removing it; still ~40% below humans.</Caveat>

## Critiques and Questions
1. With single-run numbers and a 1.9-pt 50-step margin, how much of the SOTA is noise? Re-run with seeds.
2. Routing is itself an LLM decision — what is the misroute rate, and how often does a wrong expert choice cause failure?
3. What is the latency/cost multiplier of re-planning after every subgoal vs reactive? The obvious price of the design is unmeasured.
4. Does the compositional advantage survive when grounders are also weak, or does it depend on a strong UI-TARS visual expert?
5. How does the v1→S2 delta isolate against the original Agent S? Check the repo for Manager/Worker/router prompts and the 65-task subset selection.

<Term def="Worker acts as a gate routing each action's localization to a specialist grounder (visual / OCR-textual / spreadsheet-structural), analogous to Mixture-of-Experts">Mixture-of-Grounding (MoG)</Term> ·
<Term def="Manager re-generates remaining subgoals after every completed subgoal using the latest observation, enabling backward correction; vs reactive plans that revise only on failure">Proactive Hierarchical Planning (PHP)</Term> ·
<Term def="hierarchical roles — Manager generates high-level subgoals; Worker executes the current subgoal as atomic actions and routes their grounding">Manager / Worker</Term> ·
<Term def="single model post-trained to do planning + grounding + execution end-to-end, e.g. UI-TARS, Claude Computer Use, OpenAI CUA">Native / monolithic agent</Term> ·
<Term def="Universal Network Objects — a programmatic interface to LibreOffice used by the structural grounder to set cell values directly">UNO</Term> ·
