A live, runnable Android environment that gives reward signals for 116 programmatically parameterized tasks across 20 real apps, so agents can be scored on whether they truly completed a task — under endlessly varying parameters — rather than matched against one frozen human demonstration.

Why this matters to me now

This is a foundational evaluation substrate for computer-use / mobile GUI agents — exactly the harness-relevant tooling worth tracking. Its central methodological move (read durable reward from OS/app state, and parameterize tasks so a single fixed seed cannot flatter an agent) is reusable far beyond Android and directly informs how to build trustworthy agent benchmarks.

PASS 2 Content

Motivation — the problem (NOT the novelty)

Most agent evaluation compares an agent's per-step action to a previously collected human demonstration (Deng et al. 2023; Rawles et al. 2023). This is misleading online: agents can reach a goal by multiple paths, real environments behave non-deterministically, and agents can recover from mistakes — none of which a fixed-trajectory match rewards (§1, p.1). The gold standard is online evaluation that rewards the outcome, but real apps and websites expose no built-in reward signal; human or LLM judges scale poorly or are unreliable respectively (§1, p.2). Execution-based environments that do provide ground-truth reward exist, but the paper identifies two gaps: they are confined to desktop (overlooking mobile, used by billions), and they support only static task specifications — when task parameters deviate, the reward check is likely to break (§1, p.2). Mobile is also intrinsically harder: smaller screens but a richer action space (carousels, long-press, multi-finger zoom) and more steps per task (§3.1, p.5). This is the problem space — no comprehensive, reproducible, outcome-rewarding, parameterizable environment existed for Android.

Novelty — the genuine delta ★ (this is a benchmark — novelty = what it can measure that priors could not)

  • Delta in one sentence (survives deleting "we propose"): a live Android environment where every one of 116 tasks across 20 real apps carries programmatic, parameterized setup + success-check + teardown, so an agent can be rewarded on actual task completion across a practically unbounded space of randomized goals/initial states — not just one frozen instance.
  • What it can measure that prior benchmarks could not: (1) outcome success on real apps (vs MiniWoB++'s synthetic pages, which were parameterizable but not real); (2) robustness across task parameters — Table 1 lists "avg # task instances" as ∞ for AndroidWorld and MiniWoB++, but 1 for almost every other real-app benchmark, including OSWorld (1), WebArena (3.3), VisualWebArena (2.9) (Table 1, p.3); (3) cross-platform agent transfer, by also exposing 92 MiniWoB++ web tasks (MobileMiniWoB++) through the same interface.
  • Mechanistic reason the design must take this form: real apps emit no reward, and naive parameter randomization breaks brittle UI-screen-matching checks. AndroidWorld instead reads/writes durable reward from system state via adb, using the same caching mechanisms the apps themselves use. Because the check inspects ground-truth state (e.g., "does this calendar event exist in the DB?"), it stays valid when parameters change and even when app source code is unavailable (§3.4, p.6).
  • Closest prior work & precise difference: OSWorld / WindowsAgentArena — comparable task complexity and execution-based reward, but desktop-only and static (1 instance/task); AndroidWorld is mobile and dynamically randomizes start state and parameters (§2.1, p.4). AndroidEnv — same emulator bridge, but its reward needs modifying app source and per-task logging (suited to games); AndroidWorld's reward is non-invasive and reusable across apps (§2.1, p.4). MiniWoB++ — also ∞ parameterized, but synthetic websites; AndroidWorld uses real Android apps (§1, p.2). B-MoCA (6 tasks/4 apps), Mobile-Env (13 templates/1 app) — same mobile focus but far smaller and lower-complexity (§2.1, p.4).
  • Motivation-vs-novelty check: the motivation ends at "prior reward-bearing environments are desktop-only and static, so reward breaks under parameter variation"; the novelty is the concrete artifact — durable state-based reward + per-task parameter generation across 20 real apps — that makes outcome-and-robustness evaluation possible. Distinct.
  • 30-second test: "The first live Android benchmark that rewards real task completion in real apps and can re-roll each task's parameters indefinitely, so you can measure an agent's robustness, not just its luck on one fixed instance."

Core idea / method

Each task subclasses a TaskEval contract with five pieces: a natural-language goal template, generate_random_params, initialize_task (seeds device state), is_successful (returns reward, often a 0–1 score; composite tasks give partial credit), and teardown (§3.3, Table 2, p.6). Parameters are drawn from a controlled random seed, so the same task code instantiates endless variants with consistent, valid reward logic. Reproducibility is pinned hard: a Pixel 6 emulator on Android 13, device clock reset to 2023-10-15 15:34 UTC, and fixed app versions from F-Droid that need no login and store data on-device (§3.3, p.6). Reward is read primarily from system state (SQLite rows, files, settings) via adb, falling back to on-screen UI-element checks where state inspection is impractical (§3.4, p.6). State-based checking also makes tasks composable — two standalone tasks merge into a composite with partial reward (§3.5, p.7). The web benchmark MiniWoB++ is wrapped as MobileMiniWoB++ inside a WebView app (92 of the original tasks; 12 excluded for rendering/touch/real-time issues) so the same observation/action interface covers web tasks too (§3.6, p.7; App. C.3).

Experimental setup

Benchmarks: AndroidWorld (116 tasks / 20 apps) and MobileMiniWoB++ (evaluated on a 62-task subset, per prior work). Agents: M3A (multimodal and text-only variants), M3A-Simple (minimal prompting, no guidelines/reflection), and a SeeAct baseline (a strong web agent adapted to Android — DOM replaced by a11y tree, ranker dropped, mobile actions added). Base models: GPT-4 Turbo, Gemini 1.5 Pro, open-source Gemma 2 27B. Inputs: a11y tree (text-only) vs SoM (screen + a11y tree). Metric: task success rate (SR). Determinism: seed 30, temperature 0. Human reference: two software engineers, one attempt per task. Robustness: 20 trials per task on a representative subset, comparing same-seed (fixed parameters) vs different-seed (varied parameters); also an OS-variation test on Pixel 5 / Android 12.

Qualitatively, agents show basic mobile UI competence (long-press, scroll-to-find, replanning) but fail on perception (missing a visual cue), reasoning (mis-tracking which field is active), missing UI knowledge (hunting for a non-existent "delete-all"), grounding (cursor placement / sliders), and memory-heavy tasks (transcription, multiplication) (§4.3, Figs. 6–8). The abstract and Table 3 report 30.6%, but §5 shows the three-seed mean is 29.0% and the authors keep the single-seed figure in the main table "for consistency with existing literature" — the robustness section argues the mean is the better estimate, so the headline number is mildly optimistic. Figure 3 also shows the agent appears "completely incapable" of some tasks under an unlucky fixed seed yet solves them often under other seeds — direct evidence that any single-seed leaderboard number is fragile. "Millions of unique goals" / "∞ instances" is a by-construction claim about the parameter space, not an empirically measured count.

Hidden assumptions & failure modes

Reproducibility rests on apps that need no login and store data locally — which forces the suite toward open-source F-Droid apps and built-in system apps, excluding logged-in, cloud-synced, or trending commercial apps (App. A). The authors note open-source apps can actually be harder (less-optimized UIs, fewer shortcuts), so difficulty is skewed by the reproducibility constraint, not chosen freely. State-based reward assumes the relevant outcome is observable in a DB/file/setting; where it is not, the fallback is UI-element matching, which is exactly the brittle mechanism the paper criticizes. Information-retrieval tasks are LLM-authored protobufs requiring human review (LLM tasks suffer missing/hallucinated fields), so the task set's correctness depends on manual curation (App. D.3). Finally, single-seed leaderboard numbers are demonstrably fragile (Fig. 3) — a real failure mode for anyone who reports one seed.

Strengths

  • Genuinely fills a gap: first comprehensive, live, outcome-rewarding, parameterized benchmark on real Android apps (Table 1 makes the gap explicit and quantitative).
  • The robustness analysis is a methodological contribution in its own right — it shows single-seed agent numbers can be illusory, which should change how the field reports results.
  • Engineering is honest and transparent: validator code, full task list, per-app counts, and detailed error taxonomy are all shown.
  • Lightweight and reusable: runs on a laptop; state-based validators reused across unrelated apps; composable tasks; web tasks unified under one interface.

Weaknesses & limitations

  • App scope skewed to open-source/no-login apps for reproducibility; trending commercial apps and authenticated/cloud workflows are out of scope (App. A).
  • Headline 30.6% is a single seed; the 3-seed mean (29.0%) and high intra-task variance (Fig. 3) make any single number a weak summary — and the paper still puts the single-seed figure in the main table.
  • Human baseline is only two engineers, one attempt each — a thin reference for the 80.0% upper bound.
  • "Millions of unique goals / ∞ instances" is a by-construction assertion, not an enumerated or sampled measurement.
  • Reward fallback to UI-element matching for non-inspectable tasks reintroduces the brittleness the paper otherwise avoids.

Relationship to prior work

Genuinely new as an artifact: it ports OSWorld-style execution-based reward to mobile and adds MiniWoB++-style unlimited parameterization to real apps — a combination no prior benchmark offered (Table 1). The agent (M3A) and base models are off-the-shelf compositions (ReAct + Reflexion + SoM), so the agent itself is incremental; the value is the environment and the robustness methodology.

Glossary / key terms

Reward signal
A quantitative metric of functional correctness — did the agent achieve the stated goal? Here often a 0–1 score (1 = success; composites give fractional partial credit). Real apps emit none natively, which is the core obstacle AndroidWorld solves.
Durable reward
A reward derived by inspecting persisted OS/app state (SQLite databases, files, system settings) rather than matching surface UI text/pixels. It stays valid when task parameters change and works even without the app's source code.
Parameterized task
A task whose goal, initial state, and success criterion are filled from randomly generated parameters (controlled by a seed). One task template thus yields many distinct instances — the property that lets AndroidWorld test robustness rather than one fixed case.
Task success rate (SR)
Fraction of tasks an agent completes correctly per the environment's reward check. The primary metric in Table 3.
TaskEval
The base class every task inherits, defining goal, generate_random_params, initialize_task, is_successful, and teardown — the contract that makes setup/check/cleanup hermetic and reproducible.
adb (Android Debug Bridge)
Command-line tool giving full access to a device's file system, app databases, and settings; AndroidWorld uses it both to act and to read ground-truth state for reward.
Accessibility (a11y) tree
A structured snapshot of all on-screen UI elements (text, descriptions, bounding boxes, state flags), exposed for accessibility. Used as the text observation in lieu of a web DOM.
Set-of-Mark (SoM)
A prompting technique (Yang et al., 2023) that overlays numbered bounding boxes on a screenshot so a multimodal model can refer to elements by index. The multimodal input mode for M3A.
AndroidEnv
Python library (Toyama et al., 2021) bridging an agent and the Android emulator over adb/gRPC; originally an RL platform for Android games. AndroidWorld reuses its I/O but replaces its invasive, source-code-dependent reward with non-invasive state checks.
MiniWoB++
A benchmark of small synthetic HTML/JS pages with parameterizable tasks and programmatic success checks — the canonical "∞ instances" web benchmark. AndroidWorld borrows its parameterization idea but applies it to real apps.
MobileMiniWoB++
AndroidWorld's port of MiniWoB++ into a WebView app (92 of the original tasks; 12 excluded for rendering/touch/real-time issues), exposed through the same observation/action interface so web tasks can be run alongside Android tasks.
M3A (Multimodal Autonomous Agent for Android)
The paper's zero-shot baseline agent: ReAct-style action+reasoning plus a Reflexion-style self-reflection step over before/after states. Has a multimodal (SoM) and a text-only (a11y) variant.
M3A-Simple
An ablated M3A with minimal prompting (no operating guidelines, no reflection), used to quantify how much advanced prompting helps (19.8% vs 30.6% on AndroidWorld with GPT-4).
SeeAct
A strong web-navigation agent (Zheng et al., 2024) adapted here to Android (a11y tree instead of DOM, mobile actions added) as a cross-platform baseline; reaches only 15.5% on AndroidWorld.
ReAct / Reflexion
ReAct interleaves reasoning and acting; Reflexion adds self-evaluation/self-correction memory after each action. M3A combines both.
OSWorld / WindowsAgentArena
Execution-based desktop computer-use benchmarks with ground-truth reward but static (1 instance/task) and desktop-only — the closest comparators AndroidWorld extends to mobile and to unlimited parameterization.
