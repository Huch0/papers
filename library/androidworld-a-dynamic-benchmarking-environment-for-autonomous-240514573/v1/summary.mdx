<TLDR>A live, runnable Android environment giving programmatic reward signals for 116 parameterized tasks across 20 real apps, so agents are scored on actual task completion across endlessly varying goals — not on matching one frozen human demonstration.</TLDR>

<WhyItMatters>Foundational evaluation substrate for mobile / computer-use agents. Its core moves — read durable reward from OS state, and randomize task parameters so one fixed seed can't flatter an agent — generalize well beyond Android.</WhyItMatters>

## Motivation
Most agent evaluation compares an agent's per-step action to a previously collected human demonstration, which is <Problem>misleading online: goals are reachable by many paths, environments are non-deterministic, and agents recover from mistakes — none of which a fixed-trajectory match rewards</Problem>. The gold standard is outcome-based online evaluation, but <Problem>real apps expose no built-in reward, and human / LLM judges scale poorly or are unreliable</Problem>. Execution-based environments that *do* give ground-truth reward existed, but <Problem>only on desktop (ignoring mobile, used by billions) and only for static task specs — when parameters change, the reward check breaks</Problem>. No reproducible, outcome-rewarding, *parameterizable* environment existed for Android.

## Contribution (what's new)
<Novelty>The first live Android environment where every one of 116 tasks across 20 real apps carries programmatic, parameterized setup + success-check + teardown</Novelty>, so agents are rewarded on real task completion across a practically unbounded space of randomized goals/states. The **mechanistic why**: real apps emit no reward and naive parameter randomization breaks brittle UI-matching checks, so AndroidWorld instead reads <Term def="reward derived by inspecting persisted OS/app state (SQLite DBs, files, settings) rather than matching surface UI; survives parameter changes and needs no app source code">durable reward</Term> from system state via <Term def="Android Debug Bridge — CLI giving full access to the device file system, app SQLite DBs, and settings">adb</Term> — <Novelty>a check that stays valid when parameters change</Novelty>. This lets it measure what priors could not: outcome success on real apps, and *robustness across task parameters* (Tab.1 lists avg instances/task as ∞ for AndroidWorld vs 1 for OSWorld, 3.3 for WebArena). The baseline agent (M3A) and base models are off-the-shelf compositions — the value is the environment and the robustness methodology.

## Research questions
Can a reproducible mobile environment provide durable, outcome-based reward across endlessly parameterized real-app tasks? How well do current LLM agents perform on it, do desktop web agents transfer to mobile, and how much does task-parameter variation move measured performance?

## Methodology
Each task subclasses a <Term def="base class every task inherits; defines goal, generate_random_params, initialize_task, is_successful, teardown">TaskEval</Term> contract; parameters are drawn from a controlled seed so one task template yields endless valid variants. Reward is read primarily from system state (SQLite rows, files, settings) via adb, falling back to UI-element checks where state is not inspectable. Reproducibility is pinned (Pixel 6 / Android 13 emulator, clock fixed to 2023-10-15, fixed F-Droid app versions needing no login). MiniWoB++ is wrapped as MobileMiniWoB++ (92 web tasks in a WebView) so web tasks run through the same interface.

<Pass title="Method details">
**Task contract (TaskEval):** NL goal template, `generate_random_params`, `initialize_task` (seeds device state), `is_successful` (returns reward, often a 0–1 score; composites give partial credit), `teardown` (Tab.2, §3.3). State-based checking makes tasks *composable*: two standalone tasks merge into a composite with partial reward (§3.5).

**Observation:** full-res screenshot (2400×1080×3), raw <Term def="structured snapshot of on-screen UI elements — text, descriptions, bounding boxes, state flags; used in lieu of a web DOM">accessibility (a11y) tree</Term>, processed UIElement list; optional <Term def="overlay of numbered boxes on a screenshot so a multimodal model refers to elements by index (Yang et al. 2023)">Set-of-Mark (SoM)</Term> annotated screenshot. No DOM (§3.2).

**Action:** human-like gestures (tap, long-press, swipe), text input, navigation, open-app, plus task-control (status/answer/wait) and limited high-level APIs; executed asynchronously via adb.

**Baseline agent M3A:** zero-shot <Term def="interleaves reasoning and acting">ReAct</Term>-style action+reasoning, then a <Term def="self-evaluation/self-correction memory after each action">Reflexion</Term>-style self-reflection over before/after states. Multimodal (SoM) and text-only (a11y) variants. Verifier = the environment's own reward, not the agent.

**Setup:** base models GPT-4 Turbo, Gemini 1.5 Pro, open Gemma 2 27B. Metric = task success rate (SR), seed 30, temperature 0. Human reference = two software engineers, one attempt each. Robustness = 20 trials/task on a subset, same-seed (fixed params) vs different-seed (varied params); plus an OS-variation test on Pixel 5 / Android 12.

Environment is lightweight: 2 GB RAM, 8 GB disk; runs on a laptop. Code released at github.com/google-research/android_world.

## Main result
<Finding src="Tab.3, p.8">Best agent (M3A, GPT-4 Turbo, a11y tree) completes 30.6% of tasks vs human 80.0%</Finding>, leaving ample headroom. The adapted desktop web agent SeeAct reaches only 15.5%, evidence that web agents transfer poorly to mobile. Multimodal SoM input does *not* help here (25.4% < 30.6% text-only on GPT-4). The 30.6% headline is a single seed; the <Finding src="§5, p.9">3-seed mean is 29.0% (27.6 / 26.3 / 33.2)</Finding>, which the paper keeps out of the main table "for consistency with literature" — so the headline is mildly optimistic.

## Key findings
- <Finding src="§5, Fig.3, p.10">Task-parameter variation materially changes measured performance</Finding>: with a fixed seed an agent can look "completely incapable" on a task, yet under different parameterizations solves it fairly often; different-seed groups show significantly higher variance, with `p<0.05` for add-expense and edit-note — the paper's most original empirical result.
- <Finding src="Tab.3, §4.2, p.9">Multimodal (SoM) does not generally beat text-only on real apps</Finding>: 25.4% < 30.6% for GPT-4 on AndroidWorld, though SoM does help on MobileMiniWoB++ (whose a11y trees are often incomplete) — so the benefit of vision is interface-dependent.
- <Finding src="Tab.3, p.8">A strong desktop web agent transfers poorly to mobile</Finding>: SeeAct reaches only 15.5% vs M3A's 30.6%, struggling with mobile-specific gestures (long-press, swipe) and lacking memory of action outcomes — motivating dedicated cross-platform agents.
- <Finding src="Tab.3, §4.2, p.9">Prompting scaffolding matters a lot on hard mobile tasks</Finding>: stripping guidelines and reflection (M3A-Simple) drops GPT-4 from 30.6% to 19.8% on AndroidWorld, but barely moves MobileMiniWoB++ (67.7%) — sophisticated prompting pays off only on complex real-app interactions.
- <Finding src="§5, p.10">Reward stays stable across hardware/OS variation</Finding>: re-running on a Pixel 5 / Android 12 yields 28.4%, comparable to the Pixel 6 baseline, evidence the state-based reward generalizes across device layouts.
- <Finding src="§4.3, p.9">Foundation-model agents are slow</Finding>: M3A averages 3.9 min (multimodal) / 2.5 min (text-only) per task, roughly 3x slower than humans — a practical cost beyond raw success rate.

## Implications
Ports OSWorld-style execution-based reward to mobile and adds MiniWoB++-style unlimited parameterization to *real* apps — a combination no prior benchmark offered. The robustness analysis is itself a methodological lesson: single-seed agent leaderboard numbers can be illusory; report the multi-seed mean. The state-based-reward design and mandatory parameter variation are reusable patterns for trustworthy agent benchmarks far beyond Android. <Related>Extends OSWorld / WindowsAgentArena's execution-based desktop reward to mobile, and reuses AndroidEnv's I/O with a non-invasive reward.</Related> <FollowUp>The dynamic task generation is pitched as a hook for online RL and supervised data generation; non-zero rewards under some seeds hint at RL-style improvement.</FollowUp>

## Limitations
<Caveat>App scope skewed to open-source / no-login F-Droid apps for reproducibility; trending commercial and authenticated/cloud workflows are out of scope (and open-source UIs can be harder, skewing difficulty).</Caveat>
<Caveat>Headline 30.6% is a single seed; 3-seed mean is 29.0% with high intra-task variance (Fig.3), yet the single-seed figure stays in the main table.</Caveat>
<Caveat>Human baseline is only two engineers, one attempt each — a thin 80.0% upper bound.</Caveat>
<Caveat>"Millions of goals / ∞ instances" is a by-construction assertion, not an enumerated or sampled count.</Caveat>
<Caveat>Reward fallback to UI-element matching for non-inspectable tasks reintroduces the brittleness the paper otherwise avoids.</Caveat>

## Critiques and Questions
1. The agent (M3A) is incremental (ReAct + Reflexion + SoM); the contribution is the environment, so don't over-read the absolute SR numbers.
2. If single-seed numbers are demonstrably fragile (Fig.3), why keep 30.6% in the main table at all? A leaderboard that reports the mean ± variance would be the honest default the paper itself argues for.
3. How much of the "real apps are hard" signal is the reproducibility constraint (open-source, no-login, possibly less-polished UIs) rather than intrinsic mobile difficulty?
4. The UI-matching fallback reintroduces the brittleness the paper criticizes — how many of the 116 tasks rely on it, and does their reward survive parameter re-rolls?
5. What I'd test next: compare AndroidWorld's state checkers against OSWorld's device/cloud-state checkers; and check the live repo for task/app counts beyond 116/20.

## Glossary
- <Term def="fraction of tasks an agent completes correctly per the environment's reward check; primary metric in Tab.3">Task success rate (SR)</Term>
- <Term def="a task whose goal, initial state, and success criterion are filled from seed-controlled random parameters, yielding many distinct instances from one template">Parameterized task</Term>
- <Term def="Python library (Toyama et al. 2021) bridging an agent and the Android emulator over adb; AndroidWorld reuses its I/O but replaces its invasive, source-dependent reward with non-invasive state checks">AndroidEnv</Term>
- <Term def="benchmark of small synthetic HTML/JS pages with parameterizable tasks and programmatic checks — the canonical ∞-instance web benchmark; AndroidWorld borrows its parameterization for real apps">MiniWoB++</Term>
- <Term def="AndroidWorld's port of MiniWoB++ into a WebView app (92 of the original tasks; 12 excluded for rendering/touch/real-time issues), run through the same interface">MobileMiniWoB++</Term>
- <Term def="the paper's zero-shot baseline: ReAct action+reasoning plus Reflexion self-reflection; multimodal (SoM) and text-only (a11y) variants">M3A</Term>
- <Term def="a strong web-navigation agent (Zheng et al. 2024) adapted to Android (a11y tree instead of DOM, mobile actions added) as a cross-platform baseline">SeeAct</Term>
- <Term def="execution-based desktop computer-use benchmarks with ground-truth reward but static (1 instance/task) and desktop-only — the closest comparators AndroidWorld extends to mobile">OSWorld / WindowsAgentArena</Term>
