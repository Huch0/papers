<TLDR>AssistantBench is a 214-task benchmark of realistic, time-consuming open-web tasks on which every current system scores below 26% accuracy, alongside SeePlanAct (SPA), a planning- and memory-augmented web agent that roughly doubles the answer rate of prior agents.</TLDR>

<WhyItMatters>Directly relevant to the computer-use / web-agent track: it stresses agents on the *whole* web (multi-site, multi-hop, no sandbox) rather than a single site, exposing that open-web navigation is the dominant bottleneck for today's agents.</WhyItMatters>

## Motivation
Realistic information-seeking tasks — monitoring a real-estate market, finding a nearby early-bird fitness class — take a human several minutes of browsing across multiple sites, yet <Problem>closed-book LMs rely on parametric knowledge and hallucinate</Problem> while <Problem>retrieval-augmented LMs are capped by retrieval quality and can be hurt by irrelevant evidence</Problem>. Existing agent benchmarks <Problem>test only single-website interaction or operate inside a sandbox</Problem>, so they do not measure whether agents can plan and reason over the *entire* open web.

## Contribution (what's new)
Two deltas. First, <Novelty>a benchmark of realistic, time-consuming, automatically verifiable tasks that require autonomous open-web navigation across many real sites</Novelty> — the design forces multi-site browsing precisely because single-site/sandbox benchmarks let agents succeed without true web navigation. Second, <Novelty>SeePlanAct (SPA): SeeAct extended with an explicit planning/re-planning component and a memory buffer, plus open-web actions (go back, navigate to URL, query a search engine)</Novelty>, because the baseline SeeAct lacks state to carry information across the long trajectories these tasks demand.

## Research questions
Can language agents (closed-book LMs, retrieval-augmented LMs, and web agents) solve realistic, time-consuming, multi-site web tasks? Where do they fail, and does adding planning + memory to a SOTA agent help?

## Methodology
A three-stage human data-collection pipeline builds the benchmark; tasks have closed-form answers scored by a type-aware automatic metric. The agent SPA augments SeeAct with planning and memory and is compared against closed-book, retrieval-augmented, and ensemble baselines.

## Main result

## Key findings
- <Finding src="Tab.1">Closed-book LMs beat web agents and retrieval LMs on raw accuracy, mainly via higher answer rate, but at the cost of precision</Finding>: CB-1S hits 22.2% accuracy by answering 89.5% of tasks, while exhibiting low precision and a tendency to hallucinate.
- <Finding src="Fig.5">Agents "fail fast or slow": SPA's accuracy is near zero for very short (`<5` steps) or very long (`>15` actions) trajectories and peaks around ten steps</Finding> — and AssistantBench tasks demand long interactions, explaining the difficulty.
- <Finding src="Tab.3">Expert tasks are harder for closed-book models but *easier* for SeeAct than general tasks</Finding>: 70% of expert answers sit on a single URL (vs ~20% for general tasks), so they need less multi-site navigation.
- <Finding src="Tab.4">For web agents, navigation errors dominate (36.7% for SPA, 63.6% for SeeAct), followed by grounding failures (~20%)</Finding>; 80% (SPA) / 97% (SeeAct) of errors are simply not producing an answer.
- <Finding src="§6.2">Closed-book errors are mostly hallucinated answers (85%) or outdated answers (15%); retrieval-LM errors are dominated by retrieval failures (80%, split across tool-related 38.5%, irrelevant context 50%, partial info 11.5%)</Finding>.
- <Finding src="§6.3">Proprietary ChatGPT with web search + code interpreter also errs on >90% of dev tasks</Finding>, most often by over-relying on search results to produce a wrong answer.
- <Finding src="Tab.2">On a 31-task FanOutQA subset (Wikipedia-only, multi-hop), SPA beats SeeAct by 22.5 points and is the only system to reach exact answers (3/31)</Finding>, though closed-book CB-1S still leads on accuracy (40.9%).

## Implications
AssistantBench reframes web-agent evaluation around <Related>open-web, multi-site navigation rather than single-site (Mind2Web, WebVoyager) or sandboxed (WebArena, VisualWebArena) settings</Related>, and is positioned closest to <Related>GAIA, but scoped purely to web navigation without audio/video tools</Related>. SPA continues the <Related>SeeAct multimodal-agent line</Related> by adding planning + memory. <FollowUp>Whether open-weight agents (Llama-3.2, in-progress) can close the gap</FollowUp> and <FollowUp>training data / methods to fix open-web navigation, the dominant failure mode</FollowUp> are flagged as the key next threads; <FollowUp>extending to time-dependent tasks via reference-free evaluation models</FollowUp> is another.

## Limitations
- <Caveat>Coverage: realistic but non-auto-verifiable, time-dependent tasks (e.g. live ticket availability) had to be excluded; tasks are limited to reachable participants</Caveat>.
- <Caveat>Size: 214 tasks is smaller than some agent benchmarks</Caveat> — defended by appeal to small high-quality benchmarks (HumanEval 164, Bamboogle 125) and lower cost/footprint.
- <Caveat>All systems use only GPT-4-Turbo and Claude-3.5-Sonnet, limiting reproducibility</Caveat>; open-source agents were still in progress.
- <Caveat>~56.5% of tasks can still change over a longer horizon</Caveat> (only 43.5% are fully static with date constraints, §6.4), so the benchmark will age.

## Critiques and Questions
1. The "best score" headline (25.2%) is an *ensemble* whose accuracy comes largely from the closed-book fallback (CB-1S alone is 22.2%); the standalone agent contribution is modest (SPA 11.1%). How much of the win is genuine web navigation vs. parametric recall plus a fallback?
2. Precision when answering is high for agents (SPA 30.9%) but answer rate is low (35.9%) — is the real problem capability, or over-conservative abstention/grounding?
3. With answers spread across 258 sites, how stable is the benchmark to link rot and content drift, given >56% of tasks are not strictly static?

## Glossary
- <Term def="A web agent that picks an action in two stages: describe the next action in natural language from a screenshot, then ground it to an HTML element. AssistantBench's baseline SOTA agent.">SeeAct</Term>
- <Term def="SeePlanAct — this paper's agent: SeeAct plus an explicit planning/re-planning step, a memory buffer to carry info across steps, and open-web actions (back, go-to-URL, search).">SPA</Term>
- <Term def="Fraction of tasks for which the model produced an answer instead of abstaining.">Answer rate</Term>
- <Term def="Average accuracy computed only over tasks the model chose to answer (i.e. excluding abstentions).">Precision</Term>
- <Term def="A model answering from its internal parametric knowledge with no web/retrieval access; here prompted with chain-of-thought.">Closed-book LM</Term>
- <Term def="Number metric that gives partial credit by how many orders of magnitude a prediction is off, scoring zero once a full order of magnitude away.">Order-of-magnitude metric</Term>
