<TLDR>Ferret-UI adapts the Ferret <Term def="multimodal LLM that can refer to and ground regions in an image">referring-and-grounding MLLM</Term> to mobile screens with an "any-resolution" 2-way sub-image split plus curated UI instruction data over 11 tasks, producing a raw-pixel model that beats GPT-4V on every elementary UI task category.</TLDR>

<WhyItMatters>An early, foundational <Problem>perception backbone</Problem> for mobile/GUI agents: it fixes the low-level capabilities (OCR, icon/widget recognition, find-X, grounding) that any downstream agent needs before it can plan or act, and its anyres trick and GPT-generated data recipe are widely reused.</WhyItMatters>

## Motivation
General-domain MLLMs <Problem>cannot reliably comprehend or act on UI screens</Problem>: a UI system must both read the whole screen and focus on specific elements, mapping language to actions and locations. Prior approaches each fall short: natural-image grounding MLLMs (Ferret, Shikra, Kosmos2) are <Problem>restricted to natural images and lose detail when a UI screen is resized to one low-resolution global image</Problem>, while UI-targeted works either <Problem>process whole screens as a single input</Problem> (Pix2Struct, ILuvUI, CogAgent), support only one input box (Spotlight), or lean on GPT-4V navigation — and none covers all dimensions of UI understanding. Two measured properties make this hard: UI screens have <Problem>more elongated aspect ratios</Problem> (Tab.1a) and many targets (icons) <Problem>occupy less than 0.1% of the screen</Problem> (§3).

## Contribution (what's new)
The genuine delta is twofold. First, an <Novelty>"any-resolution" (anyres) split: each screen is resized to whichever of two grids (`1x2` or `2x1`) matches its aspect ratio, then divided into exactly 2 sub-images (portrait split horizontally, landscape vertically), each encoded separately and fed to the LLM alongside the global image</Novelty> (§3). The design must take this form because the failure being fixed is detail loss on tiny elements: a single resized global image cannot resolve sub-0.1% icons, so the model needs magnified sub-image features in addition to global context. Second, <Novelty>a UI task taxonomy and instruction dataset — template-generated elementary referring/grounding tasks plus GPT-4-generated advanced reasoning tasks — that teaches the referring + grounding + reasoning capability set from raw pixels alone</Novelty>, with no view hierarchy needed at inference.

## Research questions
Can a single MLLM perform fine-grained UI referring, grounding, and reasoning directly from screen pixels, and does grounding-rich elementary training measurably improve higher-level UI reasoning?

## Methodology
Build on Ferret (CLIP-ViT-L/14 encoder + Vicuna decoder + a spatial-aware visual sampler for region features), add the anyres split, and train on a 250K-sample instruction mixture spanning Spotlight tasks, 7 elementary tasks per platform, and 4 GPT-4-generated advanced tasks. The vision encoder stays frozen; only the decoder and projection layer update. Evaluate on a 14-task dual-platform benchmark.

## Main result
On elementary-task category averages, <Finding src="Tab.2, p.10">Ferret-UI-anyres surpasses GPT-4V on all four categories — referring iPhone 82.4 vs 61.3, referring Android 82.4 vs 37.7, grounding iPhone 81.4 vs 70.3, and grounding Android 83.8 vs 4.7</Finding>. The base Ferret model scores 13.3 / 13.9 / 8.6 / 12.9, so domain training is what closes the gap.

## Key findings
- <Finding src="Tab.4a, §5.2">Elementary tasks measurably lift advanced reasoning</Finding>: advanced-only training scores ~64% on both platforms; adding both elementary sets raises iPhone advanced to 73.4 and Android to 80.5 — direct support for the "elementary as substrate" hypothesis.
- <Finding src="Tab.2, §5.1">Anyres gives a small lift on elementary tasks but a large lift on iPhone advanced tasks</Finding>: ~+2 points on iPhone referring/grounding (80.5→82.4 Ref-i, 79.4→81.4 Grd-i), but iPhone advanced jumps ~20 points (73.4→93.9, Tab.3). Android advanced *drops* (80.5→70.1) because no Android advanced data is in training — a generalization regression.
- <Finding src="Tab.2, §5.1">Ferret-UI beats Spotlight on S2W and WiC despite Spotlight's vastly larger pretraining</Finding> (115.6/113.4 vs 106.7 CIDEr S2W; 140.3/142.0 vs 141.8 WiC) — Spotlight pretrains on 80M web + 2.69M mobile screenshots. But Ferret-UI falls *below* on taperception (TaP F1 72.9/78.4 vs 88.4).
- <Finding src="Appendix C">The TaP shortfall is largely a label-noise artifact, not a model failure</Finding>: human annotation agrees with GPT-4V 76% of the time but with the dataset's taperception label only 8%, so the "ground truth" is unreliable.
- <Finding src="§5.4, p.15">Ferret-UI's grounding is competitive even when handicapped</Finding>: on grounded conversation-interaction box accuracy (manual grading) it scores 91.7% vs GPT-4V's 93.4%, despite emitting raw coordinates while GPT-4V merely *selects* from predefined boxes.
- <Finding src="Fig.6, §5.3">Anyres concretely fixes small-element failure modes</Finding>: the base model predicts neighboring text instead of the target and misclassifies small text-surrounded icons as text; both are alleviated once enlarged sub-images are added.

## Implications
This is the *eyes* of a mobile agent, not the agent itself — the observation layer downstream planners sit on. It directly extends <Related>Ferret</Related>, borrows anyres from <Related>SPHINX, LLaVA-NeXT, Monkey</Related>, follows <Related>LLaVA's GPT-generated instruction recipe</Related>, and sits in the lineage that leads to <Related>SeeClick, ScreenAI, and later GUI-grounding agents</Related>. Worth chasing next: <FollowUp>whether released weights / a Ferret-UI 2 exist</FollowUp>, and <FollowUp>how to escape the UI-detector ceiling so the model can perceive color, design, and status-bar icons the detector misses</FollowUp>.

## Limitations
- <Caveat>Author-stated: the UI detection model is a hard ceiling</Caveat> — Ferret-UI cannot learn anything the detector misses (colors, design, usability, status-bar time/WiFi/battery), since all training signal is bootstrapped from it (§5.4).
- <Caveat>Only tap actions appear in interaction data</Caveat>; scroll, long-press, and text entry are future work (Appendix B).
- <Caveat>The GPT-4V baseline is not apples-to-apples</Caveat>: only 100 sampled instances, and for grounding it is forced into Set-of-Mark candidate *selection*, which the authors show collapses on dense Android screens — so the headline 83.8 vs 4.7 Grd-A gap partly reflects a prompting limitation, not pure model capability.
- <Caveat>Advanced-task evaluation is small and noisy</Caveat>: GPT-4-as-judge over tens of questions (13 unique Android images), and the judge favors verbosity (scores exceed 100); no variance or significance reported anywhere.
- <Caveat>No released artifacts</Caveat>: no code/weights mentioned, training builds on Apple-internal AMP/detector — limiting reproducibility (Not reported for license, exact sampling ratios, GPT-4 scoring prompt).

## Critiques and Questions
1. How much of the dramatic Android-grounding gap survives if GPT-4V is allowed to emit raw coordinates rather than being forced into SoM selection? The authors' own SoM analysis suggests a large fraction of the 4.7 is harness, not model.
2. Anyres regresses Android advanced performance — does the gain on iPhone come partly at the cost of cross-platform generalization, and would balanced Android advanced data remove the trade-off entirely?
3. With a detector-bootstrapped pipeline, is "self-sufficient, raw pixels only" a meaningful claim at training time, or only at inference time?

## Glossary
- <Term def="using a region (point/box/scribble) in the model's INPUT to talk about a specific screen element">referring</Term>
- <Term def="the model producing precise locations (bounding boxes) in its OUTPUT">grounding</Term>
- <Term def="resizing a screen to a 1x2 or 2x1 grid and splitting it into 2 separately-encoded sub-images to magnify small elements">anyres (any-resolution)</Term>
- <Term def="overlay numbered magenta boxes on detected elements so a model can refer to a region by its number; here it forces GPT-4V to SELECT a box rather than emit coordinates">Set-of-Mark (SoM)</Term>
- <Term def="intersection-over-union; a grounding box counts as correct when IoU with the label exceeds 0.5">IoU>0.5 accuracy</Term>
- <Term def="text-similarity metric used to score screen2words (S2W) and widget captions (WiC)">CIDEr</Term>
- <Term def="Apple-internal mobile screenshot dataset spanning many apps; source of iPhone screens">AMP</Term>
