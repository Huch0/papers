PASS 2 Content

Motivation — the problem (NOT the novelty)

Open-source VLMs lag closed models (GPT-4o, GeminiProVision) badly on GUI grounding and on out-of-distribution agentic tasks, so practitioners default to expensive commercial APIs (Abstract; §1, p.1). Two root causes: (1) most VLMs are barely pre-trained on GUI screenshots, and existing grounding corpora are web-/mobile-heavy, often closed, small, and contain no desktop data (§1, p.1–2; Tab.1, p.4) — cross-platform grounding data is scarce and fragmented. (2) Datasets are heterogeneous and suffer action naming conflicts — the same action carries different names across platforms, confusing the model during multitask fine-tuning and degrading generalization (§1, p.2; §3.3, p.5). This is the problem; it ends at "prior data and action spaces fail for cross-platform generalist agents."

Novelty — the genuine delta ★

  • Delta in one sentence (survives deleting "we propose"): a single open action model generalizes across web/desktop/mobile because it is pre-trained on a large, automatically synthesized cross-platform grounding corpus that includes desktop, and then fine-tuned under one unified action vocabulary that removes cross-platform naming conflicts.
  • Mechanistic reason the design must take this form: (a) web-only pre-training fails to transfer — removing mobile+desktop grounding data collapses desktop icon accuracy to 28.57% and mobile icon to 10.04% (Fig.4, p.7), so the desktop/mobile data the new toolkit enables is necessary, not optional. (b) Mixing raw action vocabularies confuses the model — unifying reduces 17 action types to 10 and lifts zero-shot OOD scores (§5.3, p.9–10; Fig.5).
  • Closest prior work & precise difference: SeeClick (grounding model, 3.3M open elements, web+mobile, no desktop) — OS-Atlas scales ~4× elements and adds desktop. CogAgent (70M elements but closed, web-only). UGround (concurrent, strong but web-data-focused) — OS-Atlas is explicitly multi-platform and also ships an action model, not just grounding. Qwen2-VL/InternVL-2 are the backbones, not competitors.
  • Motivation-vs-novelty check: motivation stops at "open VLMs fail at cross-platform grounding + OOD"; novelty is the concrete remedy = toolkit + corpus (with desktop) + unified action space + the trained model. Kept distinct.
  • 30-second test: build the first big open desktop-inclusive cross-platform grounding corpus via an automated toolkit, train a VLM on it, then fine-tune under one merged action space — yielding an open GPT-4o alternative for GUI grounding/agents.

Core idea / method

Two-phase training (§3.1, p.3; Fig.2, p.4). Phase 1 — GUI grounding pre-training: learn from <screenshot, referring expression or instruction, coordinate> triplets (points or boxes) to produce OS-Atlas-Base. Data comes from the toolkit: web pages crawled from FineWeb URLs, rendered full-page and segmented into 1920×1080 tiles with clickable elements parsed from HTML (§3.2, p.4); desktop/mobile collected by automating real/virtual machines (AndroidEnv for Android, OSWorld for Linux, physical machines for Windows/macOS) and reading the A11y tree via pyatspi/pywinauto/ApplicationServices, exploring with DFS and random walk (§3.2, p.5). REG data dominates; IG data is synthesized by prompting GPT-4o with before/after screenshots and Set-of-Mark prompting. Phase 2 — action fine-tuning: train on <screenshot, task instruction, action history> to predict <thoughts, action type, parameters>, using the unified action space = 3 Basic Actions (CLICK, TYPE, SCROLL) shared across all platforms + per-task Custom Actions (e.g. OPEN_APP, LONG_PRESS, DRAG) for OOD extensibility (§3.3, p.5; Tab.6, p.16) — producing OS-Atlas. Operates in three modes: Grounding (coordinates behind a planner), Action (standalone zero-shot OOD), Agent (further SFT) (Fig.1, p.2).

Experimental setup

Grounding: ScreenSpot (single-step grounding across mobile/desktop/web), two settings — Standard (no planner) and with a GPT-4o planner. Baselines explicitly trained on GUI data: Fuyu, CogAgent, SeeClick, Qwen2-VL-7B, InternVL-2-4B, and concurrent UGround-7B (general VLMs like GPT-4V omitted as weak on ScreenSpot). Metrics: grounding accuracy (hit inside ground-truth box) + IoU (§4.1, p.6). Agent: OS-Atlas fine-tuned on only 3 agent datasets (AMEX, AITZ mobile; Mind2Web web) to leave others for OOD; evaluated zero-shot OOD (vs GPT-4o) and SFT (vs InternVL-2, Qwen2-VL, SeeClick). Metrics: Type EM, Grounding, step Success Rate (§5.1, p.8). OS-Atlas-Pro repeats SFT on all 7 agent datasets (§5.4, p.10). Compute: ablations on InternVL-2-4B "due to GPU constraints" (§4.2, p.6); exact GPU count/budget not reported.

Hidden assumptions & failure modes

Assumes A11y trees and HTML attributes give faithful element labels — noisy/incomplete trees would inject label noise (web pass already discards ~57% of raw elements after filtering: 37M→7.7M, §3.2 p.4–5). Assumes screenshots are information-complete; dynamic content, hover/scroll-revealed elements, and tiny icons remain hard (icon/widget accuracy lags text everywhere). Coordinate-distance correctness uses a 14% screen-width threshold for some agent benchmarks (App.E, p.16), a loose criterion that can flatter Grounding/SR. Desktop/web agent fine-tuning data is extremely thin (7k web, 0 desktop), so desktop agent ability rides almost entirely on pre-training transfer. Heavy reliance on GPT-4o both to synthesize IG labels and as the planner baseline risks subtle teacher-leakage and circularity.

Strengths

  • Genuinely useful, reusable artifacts: open desktop-inclusive corpus, toolkit, and a cleaned ScreenSpot-V2 (fixes ~11.32% bad samples).
  • Two contributions each validated by a controlled ablation, not just headline tables.
  • Comprehensive evaluation: 6 benchmarks, 3 platforms, two settings (zero-shot OOD + SFT), plus an interactive OSWorld test.
  • Modular deployment (Grounding/Action/Agent modes) makes it broadly adoptable.

Weaknesses & limitations

  • No variance/error bars/significance anywhere; all results appear single-run.
  • Key ablation magnitudes live only in figures, hindering precise comparison and reproduction.
  • "GPT-4o alternative" overstates the 4B model, which underperforms GPT-4o on several OOD splits; the claim is really 7B-specific.
  • Icon/widget grounding remains weak; OSWorld success (14.63) is far below human (72.36) — usable as a module, not a finished agent.
  • Loose 14%-screen-width and F1>0.5 correctness thresholds may inflate agent metrics.
  • Desktop coverage is thin (Linux 41k, macOS 13k elements), so "cross-platform desktop" leans heavily on Windows.

Relationship to prior work

Genuinely new in scope rather than in a single algorithm: the modeling recipe (grounding pre-training then action fine-tuning on a VLM backbone) follows SeeClick/CogAgent, and the backbones are off-the-shelf. The real delta is the data infrastructure (first automated multi-platform, desktop-inclusive collection at this scale) plus the unified action space that makes multitask fine-tuning across platforms work. Versus concurrent UGround it trades a narrower web edge for genuine cross-platform + an action model. A substantial systems-and-data contribution more than a methodological one.

Glossary / key terms

GUI agent
A system that operates a graphical user interface like a human — reading the screen and issuing mouse/keyboard actions — to complete tasks.
Foundation action model
A large, pre-trained model that maps instructions+screen to executable actions and is meant to be reused (zero-shot, as a module, or fine-tuned) across many GUI tasks. OS-Atlas is positioned as the first such model for GUIs.
GUI grounding
Turning a natural-language instruction into a concrete action in the OS, centrally locating the target element and its screen coordinates. The core ability of a GUI action model.
Action space
The set of action types a model may emit (e.g. CLICK, TYPE, SCROLL) with their formats/parameters.
Unified action space
OS-Atlas's standardized vocabulary: 3 platform-shared Basic Actions (CLICK/TYPE/SCROLL) + per-task Custom Actions, merging duplicates (tap=click) to cut conflicts; reduced 17 action types to 10.
Action naming conflict
When the same logical action is labeled differently across datasets/platforms, confusing multitask training.
Zero-shot OOD
Evaluation on tasks/domains/apps unseen in training, with no task-specific fine-tuning — the realistic generalist-agent setting; OS-Atlas's "Action mode".
REG (referring-expression grounding)
Locating an element explicitly named in the instruction, e.g. "click the Open button"; easy to crawl, dominates the corpus.
IG (instruction grounding)
Mapping a higher-level instruction (which may need reasoning and not name the element) to an action; a superset of REG, scarcer, synthesized here with GPT-4o.
ScreenSpot
A single-step GUI grounding benchmark across mobile/desktop/web (Cheng et al. 2024); accuracy = predicted point inside the ground-truth box.
ScreenSpot-V2
This paper's cleaned ScreenSpot: ~11.32% mislabeled/ambiguous samples fixed (14.4% web, 8.4% desktop, 10.6% mobile), same total count.
IoU (Intersection over Union)
Overlap between predicted and ground-truth bounding boxes; a finer localization metric than hit-in-box accuracy.
Type / Grounding / SR
Agent metrics: Type = exact-match of action type; Grounding = coordinate correctness in the task; SR = step success rate (both action and arguments correct).
A11y tree
Accessibility tree — the OS-exposed hierarchy of UI elements (roles, names, boxes); source of desktop/mobile element labels here (pyatspi/pywinauto/ApplicationServices).
Set-of-Mark prompting
Overlaying numbered marks on image regions so a VLM can refer to elements precisely; used to help GPT-4o synthesize IG labels.
OSWorld
An interactive real-computer benchmark (Xie et al. 2024) where an agent acts step-by-step on a live OS; used in Grounding mode here.
UGround
Concurrent universal visual GUI grounding model (Gou et al. 2024), focused on scaling web data; the closest baseline.
SeeClick / CogAgent
Prior GUI grounding/agent VLMs; SeeClick (3.3M open web+mobile) and CogAgent (70M closed web-only) are the main data/grounding baselines.
VLM
Vision-Language Model — takes images+text and produces text; OS-Atlas backbones are Qwen2-VL-7B and InternVL-2-4B.
