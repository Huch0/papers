## Motivation
<Problem>Open-source VLMs lag closed models (GPT-4o, GeminiProVision) badly on GUI grounding and on <Term def="out-of-distribution: tasks/domains/apps unseen in training — the realistic deployment regime">OOD</Term> agentic tasks</Problem>, pushing practitioners to expensive commercial APIs. Two root causes: (1) <Problem>most VLMs are barely pre-trained on GUI screenshots, and existing grounding corpora are web-/mobile-heavy, often closed, small, and contain no desktop data</Problem> — cross-platform grounding data is scarce and fragmented. (2) <Problem>heterogeneous datasets suffer <Term def="the same logical action labeled differently across platforms, e.g. mobile 'tap' vs desktop 'click'">action naming conflicts</Term> that confuse multitask fine-tuning</Problem>. The problem ends at "prior data and action spaces fail for cross-platform generalist agents."

## Contribution (what's new)
A single open action model that generalizes across web/desktop/mobile because it is <Novelty>pre-trained on a large, automatically synthesized cross-platform corpus that *includes desktop*, then fine-tuned under one unified action vocabulary</Novelty>. Mechanistic why: <Novelty>web-only pre-training does not transfer</Novelty> (dropping mobile+desktop data collapses desktop icon accuracy 45.71→28.57 and mobile icon 58.52→10.04), and <Novelty>mixing raw action vocabularies confuses the model — so unifying 17→10 action types is necessary, not cosmetic</Novelty>. Versus the closest prior work: <Related>SeeClick (3.3M open, web+mobile, no desktop)</Related> is scaled ~4× and gains desktop; <Related>CogAgent (70M)</Related> is closed and web-only; concurrent <Related>UGround</Related> is web-data-focused and grounding-only, whereas OS-Atlas is multi-platform and also ships an action model. Backbones (Qwen2-VL-7B, InternVL-2-4B) are off-the-shelf — the delta is data infrastructure + the unified action space.

## Research questions
Can an open VLM be turned into a generalist GUI action model that (a) matches/beats closed models on cross-platform grounding and (b) generalizes zero-shot to OOD agent tasks — and what data and action-space design make that possible?

## Methodology
Two-phase training on off-the-shelf VLM backbones. Phase 1 (grounding pre-training) learns coordinates from `<screenshot, expression, coordinate>` triplets sourced by a custom multi-platform synthesis toolkit, producing OS-Atlas-Base. Phase 2 (action fine-tuning) trains on `<screenshot, instruction, history>` to emit `<thoughts, action type, parameters>` under the unified action space, producing OS-Atlas. It runs in three modes: Grounding (coordinates behind an external planner), Action (standalone zero-shot OOD), Agent (further SFT).

<Pass title="Method details">
Toolkit/data (Tab.1 p.4; Tab.7 p.17): web pages crawled from FineWeb URLs, rendered full-page and tiled at 1920×1080 with clickable elements parsed from HTML (web pass filters ~37M→7.7M raw elements); desktop/mobile collected by automating real/virtual machines (AndroidEnv for Android, OSWorld for Linux, physical Windows/macOS) and reading the <Term def="accessibility tree — OS-exposed hierarchy of UI elements with roles and bounding boxes">A11y tree</Term> via pyatspi/pywinauto/ApplicationServices, explored with DFS + random walk. <Term def="referring-expression grounding: locate an element explicitly named, e.g. 'click the Open button'">REG</Term> data dominates; <Term def="instruction grounding: map a higher-level instruction (may need reasoning, may not name the element) to an action; superset of REG">IG</Term> data is synthesized via GPT-4o with before/after screenshots and Set-of-Mark prompting.

Unified action space (§3.3, Tab.6 p.16): 3 Basic Actions (CLICK, TYPE, SCROLL) shared across platforms + per-task Custom Actions (OPEN_APP, LONG_PRESS, DRAG…) for OOD extensibility. Coordinates normalized to `[0,1000]`; emitted as Python-like calls, e.g. `CLICK <point>[[x,y]]</point>`.

Backbones: InternVL-2-4B (AnyRes, max 6 tiles of 448×448) and Qwen2-VL-7B (dynamic resolution, max 1024×1024). 30 REG prompt templates, 15 samples grouped per conversation. Ablations run on the 4B model "due to GPU constraints"; exact GPU count/budget Not reported.

Setup: agent training uses only 3 datasets (AMEX, AITZ mobile; Mind2Web web), leaving others for OOD evaluation; OS-Atlas-Pro repeats SFT on all 7. Evaluated on 6 benchmarks across 3 platforms (mobile/web/desktop) plus interactive OSWorld in Grounding mode.
</Pass>

## Main result
The 7B model is the headline. On ScreenSpot (standard, no planner) it reaches <Finding src="Tab.2, p.7">avg 82.47 vs SeeClick 53.40, UGround-7B 73.30, Qwen2-VL-7B 42.89</Finding>, rising to 85.14 with a GPT-4o planner. It <Finding src="Tab.4–5, p.9">beats GPT-4o on all six zero-shot OOD agent splits, e.g. AndroidControl-Low step SR 50.94 vs GPT-4o 28.39</Finding>. As a grounding module behind GPT-4o it lifts <Finding src="Tab.3, p.8">OSWorld success to 14.63 vs +SeeClick 9.21, GPT-4o-alone 5.03 (human 72.36)</Finding>.

## Key findings
- <Finding src="Fig.4, p.7">Web-only pre-training fails to transfer cross-platform</Finding>: dropping mobile+desktop data collapses desktop icon accuracy 45.71→28.57 and mobile icon 58.52→10.04 — the desktop-inclusive corpus is load-bearing, not incremental.
- <Finding src="Fig.5, p.10">Grounding pre-training is what powers OOD generalization</Finding>: removing it ("w/o pre-training") sharply degrades zero-shot SR/grounding, worst on desktop/web where fine-tuning data is thinnest (7k web, 0 desktop) — so OOD gains come from the pre-training corpus, not the agent SFT.
- <Finding src="§5.3, Fig.5, p.10">The unified action space is mechanistically necessary, not cosmetic</Finding>: it cuts 17→10 action types (resolving tap/click, press-home/home, type/input conflicts), and removing it visibly drops OOD SR and grounding across all three platforms.
- <Finding src="Fig.3, p.6">Grounding accuracy scales positively with data, IoU more strongly than top-1 accuracy</Finding>: the authors argue accuracy saturates because it can't capture fine-grained errors, motivating harder benchmarks and IoU-style metrics.
- <Finding src="Tab.4–5, p.9">The "open GPT-4o alternative" claim is 7B-specific</Finding>: in the in-distribution SFT setting the backbone Qwen2-VL-7B already matches/beats OS-Atlas-7B on some splits (GUI-Act-Web SR 82.27 vs 82.70), so the OS-Atlas edge is concentrated in the OOD regime — its intended use.
- <Finding src="Fig.6, §5.4, p.10">Scaling agent SFT to all 7 datasets (OS-Atlas-Pro) lifts every domain</Finding>: e.g. 7B desktop avg SR 94.05→97.04, mobile 72.79→78.13 — head-room remains via more agent data, not just more grounding data.
- <Caveat>The 4B model (ScreenSpot avg 70.13) sits below UGround-7B and sometimes below GPT-4o on OOD splits</Caveat>, and <Caveat>all reported numbers are single runs — no variance, error bars, or significance anywhere, and key ablation magnitudes live only in Figs.3–6, not tables</Caveat>.

## Implications
A genuinely reusable open backbone for computer-use agents: usable as a drop-in grounding module behind any planner, a standalone zero-shot action model, or an SFT base. <Related>The desktop-inclusive corpus, the synthesis toolkit, and cleaned ScreenSpot-V2 are reusable artifacts for future grounding work</Related>. It narrows the open-vs-closed gap for GUI agents and shifts the bottleneck from model architecture to cross-platform data infrastructure. <FollowUp>Whether the toolkit + checkpoints are actually released, and how the 7B grounding module compares head-to-head with UGround on ScreenSpot-V2, are the threads to chase</FollowUp>.

## Limitations
<Caveat>All results appear single-run — no variance, error bars, or significance reported.</Caveat> <Caveat>Key ablation magnitudes live only in figures (Figs.3–6), hindering precise comparison/reproduction.</Caveat> <Caveat>"GPT-4o alternative" overstates the 4B model, which underperforms GPT-4o on several OOD splits.</Caveat> <Caveat>Icon/widget grounding remains weak and OSWorld success (14.63) is far below human (72.36) — a module, not a finished agent.</Caveat> <Caveat>Loose 14%-screen-width coordinate-correctness thresholds may inflate agent metrics; desktop agent fine-tuning data is extremely thin (7k web, 0 desktop) and desktop coverage leans heavily on Windows (Linux 41k, macOS 13k elements).</Caveat> <Caveat>Heavy reliance on GPT-4o to both synthesize IG labels and serve as the planner baseline risks subtle teacher-leakage/circularity.</Caveat> GPU budget, learning rates, and full filtering thresholds are Not reported.

## Critiques and Questions
1. Single-run results with no variance and ablation numbers hidden in figures — I'd want the numeric ablation tables and at least a few seeds before trusting magnitudes.
2. The "GPT-4o alternative" framing is 7B-only; the 4B story is much weaker — worth checking which OOD splits actually flip.
3. GPT-4o appears as both IG-label teacher and planner baseline — how much of the OOD win is teacher leakage vs genuine generalization?
4. Desktop coverage is Windows-heavy with near-zero desktop agent fine-tuning data; how robust is "cross-platform desktop" really on Linux/macOS?
5. What I'd test next: wire the 7B model as a drop-in grounding module behind a planner on OSWorld and compare head-to-head with UGround on ScreenSpot-V2; verify the toolkit + checkpoints are actually released.
