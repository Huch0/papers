PASS 2 Content

Motivation — the problem (NOT the novelty)

Native GUI agents face four open problems (p.3, §1), all about scaling rather than any single new idea. (1) Data scarcity: large-scale long-horizon trajectories that capture reasoning + action + environment state + feedback are far rarer and costlier than text/code corpora. (2) Unstable multi-turn online RL: rewards are sparse/delayed, optimization is unstable, and credit assignment across long action sequences is hard, so training does not scale past short-horizon demos. (3) GUI-only operation is insufficient: realistic workflows (data processing, software dev, sysadmin) are more naturally done through file systems, terminals, and tools than by simulating clicks. (4) Environment instability at scale: rollouts across browsers/VMs/simulators must be reproducible and fault-tolerant for millions of episodes, but such environments are fragile and crash-prone. The motivation ends here — it describes what breaks, not the fix.

Novelty — the genuine delta ★

  • Delta in one sentence (survives deleting "we propose"): a GUI agent improves stably at scale when (a) data and model co-evolve through a routing loop that recycles every generated trajectory to the training stage it best fits, (b) long-horizon online PPO is stabilized with value pretraining + decoupled/length-adaptive GAE + clip-higher and asynchronous stateful rollouts, and (c) GUI, terminal, file system, and tools share one containerized sandbox and one action space.
  • Mechanistic reason the design must take this form: naive scaling fails because (i) human GUI traces are off-policy, so the model never corrects its own mistakes — fixed by on-policy interactive annotation + flywheel recycling; (ii) plain PPO's value estimates decay/are mis-calibrated over long token sequences (the paper reports value estimates were negatively correlated with rewards), so value pretraining and decoupled GAE are needed before policy learning is reliable; (iii) batch rollouts stall on long-tail trajectories, so streaming partially-filled pools keep updates flowing; (iv) a shared file system lets a browser-downloaded file be processed by shell in the same instance, which GUI-only environments cannot express.
  • Closest prior work & precise difference: UI-TARS / UI-TARS-1.5 (same line, but offline correction of pre-collected traces and GUI-only); Kimi-Researcher (streaming partially-filled pools — explicitly credited as conceptually similar); VAPO/VC-PPO/DAPO (source of the GAE/clip tricks, here applied to multi-turn GUI rather than long-CoT reasoning); parameter merging via mode-connectivity (Qin et al. 2022) reused to fuse vertical agents.
  • Motivation-vs-novelty check: passes — the motivation stops at "prior GUI agents are off-policy, GUI-only, and unstable under online RL"; the novelty is the specific flywheel + stabilized-PPO + unified-sandbox combination, plus the mechanistic claim that value mis-calibration is what breaks long-horizon PPO.
  • 30-second test: an engineering-grade recipe + infrastructure for scaling on-policy multi-turn RL on a unified GUI/terminal/file/tool sandbox, not a new algorithm; the deltas are systems-level (flywheel, sandbox, stabilized rollout) rather than a novel objective.

Core idea / method

UI-TARS-2 follows the ReAct loop (thought t, action a, observation o) with a hierarchical memory state (working memory of the last N steps + compressed episodic memory) (p.4–5, §2.1). The action space unifies GUI primitives (click/type/scroll) with pre-defined SDK functions (terminal commands, MCP tool calls). Training runs the CT → SFT → RL flywheel from a Seed-thinking-1.6 checkpoint: each RL model generates trajectories scored by a validator V(s)∈{0,1}; high-quality ones are appended to SFT, low-quality ones routed to CT, so no data is wasted (p.7, §2.3). SFT data is collected on-policy via interactive human-in-the-loop annotation where annotators accept/override the live model's proposed thought+action; CT data uses an in-situ think-aloud protocol that aligns verbalized reasoning with UI interactions (§2.4). Multi-turn RL uses RLVR across GUI-Browsing (obfuscated multi-hop info-seeking, LLM-as-Judge reward), GUI-General (690 websites, scored by UI-TARS-2 as a generative ORM on the last five screenshots), and Gameplay (JS verifiers on HTML5/WebGL mini-games). RL = PPO + value pretraining + Decoupled-GAE + Length-Adaptive GAE (α=0.05) + clip-higher (§2.5.4). Specialized per-domain RL models are then fused by convex parameter interpolation θ(merge)=Σ αₖθ(k) (§2.6). Infrastructure: thousands of cloud VMs (Windows/Ubuntu/Android, PyAutoGUI+ADB) and a hardware-accelerated browser sandbox with fake-clock time control, driven by asynchronous server-based inference and streaming partially-filled rollout pools.

Hidden assumptions & failure modes

Assumes (i) agent success is concretely verifiable, making a VLM/ORM reward safe — plausible for games and reference-answer browsing, riskier for open-ended GUI-General where the model judges itself; (ii) specialized RL models from one SFT init stay linearly mode-connected, so interpolation preserves skills — empirically supported here but domain-dependent. Likely failure modes: very long-horizon planning (Tetris/Sokoban weak), tasks needing precise external search (BrowseComp-en below o3), and reward hacking surfacing as the policy explores (entropy is rising, not collapsing).

Glossary / key terms

Multi-turn RL
RL where the agent takes many sequential actions in a live environment before receiving a (usually sparse) reward; the central challenge is credit assignment and stability over long trajectories. Core to UI-TARS-2's training.
Data flywheel
A self-reinforcing loop: the latest model generates trajectories, a validator V(s)∈{0,1} routes high-quality ones to SFT and low-quality ones to CT, so model and data co-improve each cycle with no data wasted (§2.3).
Sandbox
The containerized execution environment the agent acts in. UI-TARS-2's "all-in-one" sandbox merges cloud VMs (Win/Ubuntu/Android) and a browser game sandbox with a shared file system, so GUI, terminal, and tool actions interoperate.
Rollout
One episode of the agent interacting with the environment to produce a trajectory for training/eval. Here rollouts are asynchronous and stateful (execution state persists across tool calls), pooled in streaming partially-filled batches.
OSWorld
369-task execution-based desktop benchmark (Ubuntu/Windows/macOS) for computer-use agents. UI-TARS-2: 47.5.
WindowsAgentArena (WAA)
>150 Windows-specific tasks adapting the OSWorld framework. UI-TARS-2: 50.6.
AndroidWorld
116 mobile tasks across 20 apps in a live Android emulator with randomized parameters. UI-TARS-2: 73.3.
Online-Mind2Web
300 realistic live-web tasks across 136 sites for browser agents (the "illusion of progress" re-assessment of web agents). UI-TARS-2: 88.2.
BrowseComp (en/zh)
High-difficulty multi-hop browsing benchmarks where the answer is deliberately obfuscated across pages. UI-TARS-2: 29.6 en / 50.5 zh with GUI-SDK.
Terminal Bench
Benchmark of command-line proficiency for AI agents in terminal environments. UI-TARS-2: 45.3 (75/80 tasks).
SWE-Bench (Verified)
Repository-level software-engineering benchmark: resolve real GitHub issues. UI-TARS-2: 68.7 with GUI-SDK.
LMGame-Bench
OOD benchmark of LLM/VLM game play across six classic titles via a Gym-style interface, with optional perception/memory scaffolds (harnessed vs unharnessed).
PPO
Proximal Policy Optimization: actor–critic RL that clips the policy-update ratio to keep updates stable. UI-TARS-2's chosen optimizer, augmented with decoupled/length-adaptive GAE, value pretraining, and clip-higher.
GRPO
Group Relative Policy Optimization: a critic-free RL method that normalizes rewards within a sampled group; effective for reasoning tasks but here found to underperform PPO in reward and stability (Fig.12).
GAE
Generalized Advantage Estimation: a bias–variance-tunable advantage estimator (param λ). UI-TARS-2 decouples λ for policy vs critic and adapts λ to sequence length to stabilize long-horizon training.
ReAct
Reasoning+Acting paradigm interleaving thought, action, and observation each step — the agent's per-step loop.
RLVR
Reinforcement Learning with Verifiable Rewards: rewards from automatic verifiers or reference matching rather than purely learned reward models.
ORM
Outcome Reward Model: scores whether a full trajectory succeeded. UI-TARS-2 is used as a generative ORM (last 5 screenshots + text), F1 83.8 on 300 traces.
GUI-SDK
The extended action space beyond GUI clicks: terminal commands and MCP tool calls. Results marked † use it; baselines generally do not.
MoE
Mixture-of-Experts: only a subset of parameters activate per token. UI-TARS-2 has 23B active / 230B total params + a 532M vision encoder.
Parameter interpolation
Convex blending θ(merge)=Σαₖθ(k) of per-domain RL models from a shared SFT init, exploiting linear mode connectivity to fuse skills without joint RL.
