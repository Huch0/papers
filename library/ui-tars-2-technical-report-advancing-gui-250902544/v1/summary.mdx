## Motivation
Native GUI agents stall on four *scaling* problems, not a missing idea: <Problem>long-horizon trajectories pairing reasoning + action + environment state are far rarer and costlier than text/code corpora</Problem>; <Problem>multi-turn online RL is unstable — rewards are sparse/delayed and credit assignment over long action sequences is hard</Problem>; <Problem>GUI-only operation is unnatural for real workflows (data processing, software dev, sysadmin) better done via files/terminals/tools</Problem>; and <Problem>rollout environments (browsers, VMs, simulators) are fragile and crash-prone at the scale of millions of episodes</Problem>. This is what breaks — not the fix.

## Contribution (what's new)
<Novelty>A GUI agent improves stably at scale when data and model co-evolve, long-horizon PPO is stabilized, and GUI/terminal/file/tools share one sandbox.</Novelty> Concretely: (a) a <Term def="self-reinforcing loop: the latest model generates trajectories that a validator routes back to CT/SFT/RL, so model and data co-improve each cycle">data flywheel</Term> that recycles every generated trajectory to the stage it best fits; (b) a stabilized multi-turn PPO recipe (value pretraining + decoupled & length-adaptive GAE + clip-higher) with asynchronous stateful rollouts; (c) a containerized sandbox with shared file system unifying GUI, terminal, files, and tools under one action space. The mechanistic why: <Novelty>human GUI traces are off-policy so the model never self-corrects</Novelty> (fixed by on-policy annotation + flywheel); <Novelty>plain PPO's value estimates were found negatively correlated with rewards over long sequences</Novelty> (fixed by value pretraining + decoupled GAE); a shared file system lets a browser-downloaded file be processed by shell in the same instance, which GUI-only environments cannot express. The delta is integration and scaling over UI-TARS-1.5, not a new learning principle — RL machinery is imported from <Related>VAPO / VC-PPO / DAPO</Related>.

## Research questions
Can a native GUI agent be scaled by (i) co-evolving training data with the model, (ii) stabilizing long-horizon online RL, and (iii) unifying GUI with system-level actions — and does the resulting RL transfer out-of-distribution?

## Methodology
A single end-to-end <Term def="Reasoning+Acting: interleave thought, action, observation each step">ReAct</Term> policy with hierarchical memory acts in the sandbox over a unified action space (GUI clicks/types + SDK terminal/MCP-tool calls). It is trained through a CT → SFT → RL flywheel from a Seed-thinking-1.6 checkpoint, then per-domain RL models are fused by convex parameter interpolation.

<Pass title="Method details">
- **Model:** 532M vision encoder + MoE LLM, 23B active / 230B total params, init from Seed-thinking-1.6 (§3.1).
- **Flywheel (§2.3):** each RL model generates trajectories scored by a validator `V(s)` returning `{0,1}`; high-quality → appended to SFT, low-quality → routed to CT, so no data is wasted. SFT data is collected *on-policy* via interactive human-in-the-loop annotation (annotators accept/override the live model's proposed thought+action); CT uses an in-situ think-aloud protocol.
- **Multi-turn RL (§2.5):** <Term def="Reinforcement Learning with Verifiable Rewards: rewards from automatic verifiers or reference matching, not purely a learned reward model">RLVR</Term> across GUI-Browsing (LLM-as-Judge reward), GUI-General (690 websites, scored by UI-TARS-2 as a generative <Term def="Outcome Reward Model: scores whether a full trajectory succeeded">ORM</Term> on the last 5 screenshots), and Gameplay (JS verifiers). Optimizer = PPO + value pretraining + Decoupled-GAE + Length-Adaptive GAE (α=0.05) + clip-higher.
- **Merging (§2.6):** `θ_merge = Σ αₖ·θₖ`, convex blend of per-domain RL models from a shared SFT init, exploiting linear mode connectivity.
- **Infra (§2.2):** thousands of cloud VMs (Windows/Ubuntu/Android, PyAutoGUI+ADB) + a hardware-accelerated browser sandbox with fake-clock determinism; asynchronous server-based inference with streaming partially-filled rollout pools and stateful environments; W4A8 quantization for deployment.
</Pass>

## Key findings
The detailed-analysis section (§3.3) is where the reusable lessons live, beyond the headline sweep:
- <Finding src="§3.2, p.14">RL trained mostly on browser tasks transfers out-of-distribution</Finding>: SFT→RL lifts OSWorld 43.0→47.5 and AndroidWorld 64.6→73.3 even though GUI-Browsing/GUI-General data dominated RL — the central evidence for the "scalable RL induces transferable skills" claim.
- <Finding src="Fig.10b, §3.3">plain PPO's value estimates were negatively correlated with rewards, and value pretraining fixes it</Finding>: pretraining the critic to convergence under a fixed policy yields consistently higher reward throughout training — the most interesting mechanistic finding.
- <Finding src="Fig.12, §3.3">PPO beats GRPO by a clear margin here</Finding>: PPO holds higher reward with lower volatility, contradicting the common assumption that critic-free GRPO suffices for agent RL.
- <Finding src="Fig.8, §3.3">entropy rises during GUI/game RL rather than collapsing</Finding>: opposite to reasoning-focused RL's monotonic entropy decay, suggesting the policy keeps expanding its exploration space in visually rich environments.
- <Finding src="Fig.9, §3.3">average step-level think length declines during GUI RL</Finding>: once the correct GUI action is predictable, progress comes from environment interaction, not longer deliberation (in games, think length oscillates with curriculum difficulty).
- <Finding src="§3.3, p.16">an imperfect ORM still trains effectively</Finding>: F1 = 83.8 on 300 human-annotated traces with a high false-positive rate, yet correct intermediate steps in false-positive trajectories still supply useful reward — no substantial reward hacking observed.
- <Finding src="Fig.11, §3.3">inference-time scaling is staircase-like and almost monotonic</Finding>: more allowed steps keep lifting OSWorld/game scores even after RL shortened training trajectories, whereas baseline curves flatten quickly.
- <Finding src="Fig.15, §3.3">hybrid GUI+GUI-SDK training beats the GUI-only baseline on pure GUI tasks despite halved per-interface data</Finding>: capability from the richer SDK interface transfers down to GUI-only, and a shared value model raises explained variance.

## Implications
A concrete, reusable systems blueprint for scaling online multi-turn RL on GUI agents — asynchronous stateful rollouts, streaming pools, value pretraining, decoupled/length-adaptive GAE — plus the practical finding that RL trained mostly on browser tasks transfers OOD to desktop/mobile. Unifying GUI with terminal/file/tool actions in one sandbox is a design pattern beyond pure click-simulation agents, extending the <Related>UI-TARS / native-agent line</Related>. <FollowUp>Whether the "value negatively correlated with reward" pathology is GUI-specific or a general long-horizon PPO failure is worth chasing</FollowUp>, as is <FollowUp>how stable the self-judged ORM + rising-entropy setup is past the reported horizon</FollowUp>. The empirical observations are usable even though weights, code, and data stay closed.

## Limitations
<Caveat>Tech-report rigor: single runs, no ablations isolating flywheel vs sandbox vs RL-recipe contributions.</Caveat>
<Caveat>Cross-model comparisons are not head-to-head — many Claude/OpenAI cells are missing and † numbers use a GUI-SDK action space baselines lack; it trails the best proprietary models on SWE-Bench and BrowseComp-en.</Caveat>
<Caveat>Fully closed: no weights, training code, or data; built on closed Seed-thinking-1.6; figures lack axis scales so quantitative training claims are unverifiable.</Caveat>
<Caveat>Self-judged rewards (ORM / LLM-as-Judge) on open-ended GUI-General tasks risk reward hacking; high ORM false-positive rate noted.</Caveat>
<Caveat>Long-horizon planning ceiling (weak on Tetris/Sokoban, plateaus on Gem-11/Hex-frvr) attributed to the backbone, not addressed.</Caveat>

## Critiques and Questions
1. The headline "outperforms Claude/OpenAI" rests partly on non-comparable cells and a GUI-SDK action space the baselines lack — what do the numbers look like restricted to GUI-only, apples-to-apples?
2. No ablation isolates the three contributions (flywheel vs sandbox vs stabilized RL); which one actually drives the gains?
3. Self-judged ORM rewards with a high false-positive rate plus *rising* entropy are a setup where reward hacking could surface later — how stable is this past the reported horizon?
4. The "value estimates negatively correlated with rewards" finding is the most interesting mechanistic claim; is it GUI-specific or a general long-horizon PPO pathology?

<SelfCheck q="Why did the authors add a value-pretraining stage, and what symptom motivated it?" a="In preliminary PPO runs the critic's value estimates were negatively correlated with the rewards actually obtained over long sequences. ## Glossary
