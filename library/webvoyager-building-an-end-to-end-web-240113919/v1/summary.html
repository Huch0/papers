PASS 2 Content

Motivation — the problem (NOT the novelty)

Prior web agents share two gaps. (1) Single modality on simplified environments: most agents consume only HTML/accessibility-tree text and run inside web simulators or static page snapshots, ignoring that browsers render pages into visual layouts designed for human eyes; verbose HTML is hard to manage and discards layout/visual cues (p.1, §1). (2) Stepwise, offline evaluation: benchmarks like Mind2Web grade each action against a single predefined "golden" trajectory, which cannot credit the many valid ways to finish a task and biases comparison (p.2, §1). Together these limit applicability to the messy live web (floating ads, pop-ups, constant updates) and make fair, scalable evaluation of open-ended, end-to-end task success hard.

Novelty — the genuine delta ★

  • Delta in one sentence (survives deleting "we propose"): a vision LLM can complete tasks on unmodified live websites by acting on a screenshot whose interactive elements are overlaid with numbered boxes, with no object-detection module and no fine-tuned element selector — and such open-ended runs can be graded automatically by a second VLM that watches the screenshot trajectory.
  • Mechanistic reason the design must take this form: HTML/accessibility-tree text is verbose and discards layout, while rendered pages encode information visually; marking elements with numeric labels lets the model refer to a target by index (sidestepping pixel-coordinate grounding) so a generic VLM can both see and act (p.1,§1; p.3–4,§3.3). The numbers also let the judge re-derive what happened from screenshots, enabling trajectory-level scoring without a golden path.
  • Closest prior work & precise difference:
    • SoM (Yang 2023a): WebVoyager reuses the marking idea but drops the object-detection model — it extracts interactable elements rule-based via the GPT-4V-Act JS tool (p.4, §3.3).
    • WebArena (Zhou 2023): WebVoyager does not host sites locally — it runs the open web live; the text-only baseline adopts WebArena's accessibility-tree representation.
    • SeeAct (Zheng 2024, concurrent): also LMM-based, but its best agent relies on a fine-tuned cross-encoder for element selection; WebVoyager "does not require any additional modules" (p.3, §2).
  • Motivation-vs-novelty check: passes. The problem (text-only + offline-golden eval) is stated independently of the solution; the novelty is the specific mechanism (live browser + module-free SoM screenshots + VLM trajectory judge), not merely "prior X fails."
  • 30-second test: drive real websites from labeled screenshots with an off-the-shelf VLM, no extra detection/selection modules, and grade the open-ended results with another VLM.

Core idea / method

Given an instruction, WebVoyager opens a real browser (Selenium) and loops: observe → think → act. The observation at step t is the current screenshot plus auxiliary text (each interactive element's type, embedded text, and any aria-label), with numbered Set-of-Mark boxes drawn on the elements (single black color chosen empirically for higher success). The model emits a ReAct-style Thought then an Action from a fixed set: Click [n], Type [n];[text] (clears box + auto-ENTER), Scroll [n/WINDOW];[up/down], Wait, GoBack, Google (jump to a search engine to escape dead ends), and ANSWER;[content] (terminate). Context is clipped to the three most recent observations but keeps the full thought/action history (p.3, §3.2). Max 15 steps; execution errors are fed back and consume a step (p.4, §3.3). The evaluator is a separate GPT-4V given the task, WebVoyager's response, and the last k trajectory screenshots, returning a binary SUCCESS / NOT SUCCESS at temperature 0 (p.6, §5.1; App. B).

Harness relevance

Environment / workspaceLive, unmodified public websites driven through Selenium; fixed 1024×768 window; single tab (multi-tab disabled); CAPTCHA/login sites avoided (p.3–4, §3.1, §4.1).
Observation interfaceRendered screenshot with numbered Set-of-Mark boxes + per-element auxiliary text (type, inner text, aria-label). Three most-recent observations retained (p.3–4, §3.2–3.3).
Action interfaceGUI-level browser actions referenced by numeric label: Click, Type, Scroll, Wait, Back, Google (jump-to-search), Answer (p.4, §3.4; App. C).
Planner/executor/verifier/searchSingle ReAct loop (Thought + Action) acts as planner+executor; "Google" action is an escape/restart heuristic. No separate planner/verifier module; no tree search.
Evaluation harnessHuman binary judgment over full trajectories (3 annotators on 300 tasks) plus a GPT-4V auto-judge over the last k screenshots; metric = Task Success Rate (p.5–6, §5).
Training harnessNone — fully prompting-based; backbones are GPT-4 Turbo with vision (gpt-4-vision-preview), Claude 3 Opus, GPT-4o (p.5, §5).
Repro / logging / safetyCode + data promised at GitHub (MinorJerry/WebVoyager). All screenshots logged for evaluation. Safety: non-login tasks only, monitored runs, manual query inspection; CAPTCHAs respected (p.3 fn.4; Ethics/Limitations, p.9–10).

Experimental setup

Datasets: own benchmark of 643 tasks across 15 sites, 40–45 tasks each (p.4, §4.2; Tab.1) built by self-instruct + human verification; 22.3% have "Golden" answers, the rest "Possible" (open-ended / real-time / multiple-valid). Plus 90 web tasks (Level 1+2) from GAIA and 50 live tasks from SeeAct's online set. Baselines: GPT-4 (All Tools) (OpenAI's integrated browsing/tools agent) and WebVoyagertext-only (accessibility tree à la WebArena). Backbones: GPT-4V (primary), Claude 3 Opus, GPT-4o. Metric: Task Success Rate (binary completion, optimality ignored). Eval: human (primary) + GPT-4V auto-judge (3 runs → mean ± std). Temperature 1 for the agent, 0 for the judge.

Hidden assumptions & failure modes

Assumes: target sites are reachable without login/CAPTCHA (15 sites chosen accordingly, p.4 §4.1 — selection bias toward agent-friendly sites); element extraction by GPT-4V-Act is reliable; tasks finish within 15 steps and ~7000 tokens; the VLM can read on-screen text (fails on dense/small text). Documented failure taxonomy on 300 sampled fails (Tab.4, p.9): Navigation Stuck 44.4% (loops, scroll confusion, runs out of steps), Visual Grounding 24.8% (picks adjacent/wrong element, misreads calendar numbers as labels), Hallucination 21.8% (partially-correct answers, wrong text box), Prompt Misalignment 9.0% (unparseable output, premature ANSWER). Context clipping to 3 observations can cause repeated mistakes (p.8, §5.4).

Strengths

  • Tackles the real open web rather than a simulator/snapshot — high external validity.
  • Module-free recipe (off-the-shelf VLM + rule-based marking + ReAct) is simple and broadly reusable; became a de-facto baseline.
  • The GPT-4V auto-judge is properly validated against 3-annotator human labels with Cohen's κ, and ablated over k screenshots.
  • Honest, quantified error taxonomy and per-site analysis; multilingual demos (Chinese, Spanish, App. D).

Weaknesses & limitations

  • Live-web evaluation is inherently non-reproducible; numbers are a snapshot in time.
  • Headline 59.1% (human-judged) appears single-run with no variance; std-dev is only reported for the GPT-4V-judged numbers, and abstract (59.1%) vs Table 1 auto-eval (57.1%) can be conflated.
  • Self-evaluation bias: each backbone's judge favors its own outputs (Tab.3), confounding the cross-backbone comparison.
  • Site selection excludes login/CAPTCHA sites — biases toward agent-friendly tasks and limits real-world coverage.
  • Reads poorly on text-heavy pages; no Drag, limited file formats (no video); only 15-step / single-tab budget.
  • Real deployment risks: could download malicious content, leak private input, or generate fake server traffic (Limitations, p.10).

Relationship to prior work

Genuinely new as an integration on the live web: it is the first to combine module-free Set-of-Mark screenshots, a ReAct loop, real online browsing, and a validated VLM-as-judge into one end-to-end system+benchmark. The individual ingredients (SoM, ReAct, GPT-4V, accessibility trees) are prior; the live-web setting, the no-extra-module claim vs SeeAct, and the open-ended trajectory-judge protocol are the deltas. Incremental on representation theory, foundational on agent practice.

Glossary / key terms

Web agent
An autonomous program that reads a web page and issues actions (click/type/scroll) to complete a natural-language instruction without human help.
End-to-end
The agent runs the whole task from instruction to final answer with no intermediate human intervention.
LMM / LMM-powered
Large Multimodal Model — an LLM that also takes images. Here GPT-4V, Claude 3 Opus, GPT-4o serve as the agent's "brain."
GPT-4V (gpt-4-vision-preview)
OpenAI's GPT-4 Turbo with vision; the primary backbone. "GPT-4o" is the later "Omni" model; "Claude 3 Opus" is Anthropic's model — both used as alternative backbones and judges.
Set-of-Mark (SoM) prompting
Overlaying numbered bounding boxes on an image so a vision model can reference targets by index ("Click [17]") instead of pixel coordinates. WebVoyager extracts the boxes rule-based (GPT-4V-Act), with no object-detector.
GPT-4V-Act
A JavaScript tool that finds interactive elements by HTML type and draws numbered boxes on the screenshot; rule-based, no learned model.
ReAct prompting
A prompting style where the model writes a free-text "Thought" before each "Action," interleaving reasoning and acting.
Accessibility tree
The structured textual representation of a page (roles, labels, hierarchy) browsers expose for screen readers; used as input by the text-only baseline (and by WebArena).
Task Success Rate
Primary metric: fraction of tasks judged successfully completed (binary), regardless of whether the steps taken were optimal.
GPT-4V-as-judge (auto-evaluator)
A separate GPT-4V call that reads the task, the agent's final response, and the last k trajectory screenshots and returns SUCCESS / NOT SUCCESS — a scalable stand-in for human grading.
Agreement & Cohen's κ (kappa)
Agreement = fraction of labels where judge and human match (85.3% at full trajectory). Cohen's κ corrects for chance agreement; κ=0.70 here is "substantial," matching the human inter-annotator κ.
Self-instruct
A data-generation method where an LLM creates new tasks from a few in-context seed examples; used (with human filtering) to build the 643-task set.
The 15 sites
Allrecipes, Amazon, Apple, ArXiv, BBC News, Booking, Cambridge Dictionary, Coursera, ESPN, GitHub, Google Flights, Google Map, Google Search, Huggingface, Wolfram Alpha — login/CAPTCHA sites omitted.
"Golden" vs "Possible" answers
Golden = stable, exhaustively listed correct answers (22.3% of tasks); Possible = partial/changeable answers for open-ended or real-time tasks.
GAIA
A general-assistant benchmark; here only its 90 web-browsing Level-1/Level-2 tasks are used (agent starts from Google Search).
WebArena / Mind2Web / SeeAct
Prior web-agent benchmarks/agents: WebArena = self-hosted realistic env (accessibility tree); Mind2Web = offline golden-trajectory eval; SeeAct = concurrent GPT-4V agent whose best variant uses a fine-tuned element selector.
GPT-4 (All Tools)
OpenAI's Oct-2023 integrated agent (vision + Bing browsing + code + plugins) used as a baseline; limited because it browses via Bing-fetched pages and can't directly drive many sites.
