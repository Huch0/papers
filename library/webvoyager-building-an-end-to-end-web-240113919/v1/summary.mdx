<TLDR>A vision-first web agent that drives live, real-world websites end-to-end by acting on marked-up screenshots with an off-the-shelf multimodal LLM, paired with a GPT-4V "judge" that scores open-ended trajectories automatically and a 643-task / 15-site benchmark.</TLDR>

<WhyItMatters>Foundational for my computer-use / web-agent track: it popularized the "screenshot + Set-of-Mark + ReAct on a real browser" recipe and the "VLM-as-judge for open-ended trajectories" evaluation pattern that later agent papers reuse.</WhyItMatters>

## Motivation
Prior web agents share two gaps. (1) They consume only <Term def="the textual tree (roles, labels, hierarchy) a browser exposes for a page">HTML / accessibility-tree</Term> text and run inside simulators or static page snapshots, ignoring that browsers *render* pages into visual layouts; verbose HTML is hard to manage and discards layout cues. (2) Benchmarks like Mind2Web grade each action against one predefined "golden" trajectory, which cannot credit the many valid ways to finish a task. Together these limit applicability to the messy live web (ads, pop-ups, constant updates) and make fair, scalable evaluation of open-ended end-to-end success hard.

## Contribution (what's new)
<Claim>A generic VLM can complete tasks on *unmodified live websites* by acting on a screenshot whose interactive elements are overlaid with numbered boxes — with no object-detection module and no fine-tuned element selector — and such open-ended runs can be graded automatically by a second VLM watching the screenshot trajectory.</Claim> Mechanistic why: rendered pages encode information visually, and numbering elements lets the model *refer* to a target by index (`Click [17]`) instead of solving pixel-coordinate grounding, so one model can both see and act; the same numbers let the judge re-derive what happened without a golden path. Closest prior work: <Term def="Set-of-Mark: overlay numbered boxes on an image for visual grounding">SoM</Term> (reused, but drops the object detector — elements extracted rule-based), WebArena (not self-hosted — runs the open web), and concurrent <Term def="a concurrent GPT-4V web agent whose best variant uses a fine-tuned cross-encoder to rank elements">SeeAct</Term> (WebVoyager needs no extra modules).

## Research questions
Can an off-the-shelf multimodal LLM complete real-world tasks end-to-end on live websites using only marked-up screenshots plus minimal element text? And can a VLM reliably stand in for a human when scoring such open-ended trajectories?

## Methodology
Given an instruction, WebVoyager drives a real browser (Selenium) in an observe→think→act loop. Each observation is the current screenshot with numbered Set-of-Mark boxes plus per-element text (type, inner text, `aria-label`); the model emits a <Term def="prompting style where the model writes a free-text Thought before each Action">ReAct</Term> Thought then one Action from a fixed set, terminating with `ANSWER`. The evaluator is a separate GPT-4V given the task, the agent's response, and the last k trajectory screenshots, returning binary SUCCESS / NOT SUCCESS.

## Implications
A simple, module-free recipe (off-the-shelf VLM + rule-based marking + ReAct on a live browser) became a de-facto baseline architecture for multimodal computer-use agents, and the validated VLM-as-judge protocol gave the field a scalable way to score non-deterministic, open-ended agents without golden trajectories.

## Limitations
<Limit>Live-web evaluation is inherently non-reproducible — numbers are a snapshot in time as sites, ads, and real-time data change.</Limit>
<Limit>The headline 59.1% (human-judged) appears single-run with no variance; abstract 59.1% vs Table 1 auto-eval 57.1% are easy to conflate.</Limit>
<Limit>Self-evaluation bias: each backbone's judge favors its own outputs (Tab.3), confounding the cross-backbone comparison (GPT-4o lenient, GPT-4V strict, Claude prefers itself).</Limit>
<Limit>Site selection excludes login/CAPTCHA sites, biasing toward agent-friendly tasks; reads poorly on text-heavy pages; no Drag, single tab, 15-step budget. Documented failures: Navigation Stuck 44.4%, Visual Grounding 24.8%, Hallucination 21.8%, Prompt Misalignment 9.0% (Tab.4).</Limit>

## Critiques and Questions
1. The "both modalities needed" claim is argued from per-site splits, not from an ablation that feeds HTML text into the vision agent — the cleaner test would add accessibility text to WebVoyager and measure the lift.
2. Cross-backbone scores (Tab.3) are confounded by self-evaluation bias; a single fixed neutral judge across all backbones would make the GPT-4V / Claude / GPT-4o comparison fair.
3. Headline 59.1% is single-run human-judged — I'd want repeated human passes or at least variance before treating the GPT-4V-vs-baseline gap as robust.
4. Site selection avoids login/CAPTCHA; how much does success drop on the harder, gated web that real assistants must handle?

## Glossary
- <Term def="an autonomous program that reads a web page and issues actions to complete a natural-language instruction without human help">Web agent</Term> — the system class.
- <Term def="Large Multimodal Model: an LLM that also takes images; here GPT-4V, Claude 3 Opus, GPT-4o serve as the agent's brain">LMM</Term> — the backbone family.
- <Term def="overlay numbered bounding boxes on interactive elements so a vision model can reference targets by index instead of pixel coordinates">Set-of-Mark (SoM)</Term> — the marking technique.
- <Term def="a JavaScript tool that finds interactive elements by HTML type and draws numbered boxes; rule-based, no learned model">GPT-4V-Act</Term> — how the boxes are produced.
- <Term def="fraction of tasks judged successfully completed (binary), regardless of whether the steps were optimal">Task Success Rate</Term> — the primary metric.
- <Term def="agreement corrected for chance; 0.70 is 'substantial', matching the human inter-annotator level">Cohen's kappa</Term> — judge-reliability statistic.
- <Term def="an LLM creates new tasks from a few seed examples; used with human filtering to build the 643-task set">Self-instruct</Term> — benchmark construction.
- <Term def="a general-assistant benchmark; here only its 90 web-browsing Level-1/2 tasks are used">GAIA</Term> — secondary benchmark.
