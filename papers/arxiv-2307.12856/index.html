WebAgent splits real-website automation into a HTML-T5 planner+summarizer and a frozen 540B program synthesizer (Flan-U-PaLM) that emits Python/Selenium actions, and shows this modular, self-experience-finetuned recipe lifts real-website success by over 50% while HTML-T5 alone sets SoTA on MiniWoB++ and Mind2Web.

An early, frequently-cited push of web agents off simulators onto live real sites — three of its choices (HTML-as-observation bottleneck, code-as-action, planner/summarizer/actor split) are load-bearing for today’s computer-use harnesses.

Motivation

Web automation on real websites suffers three problems prior LLM agents sidestepped by staying in simulators. There is no pre-defined action space — real sites are open-domain, unlike MiniWoB++/WebShop with their fixed categorical actions. Real HTML is far longer than simulated observations: ~0.5K tokens in simulators vs 7K-14K on real sites (Fig.2, ~15x larger, p.3), overflowing LLM context. And general LLMs lack HTML-specific inductive bias, while prior HTML transformers (MarkupLM, TIE) use text-XPath alignment too costly for long real documents.

Contribution (what’s new)

The genuine delta is two-fold. HTML-T5: an encoder-decoder pre-trained specifically for long HTML — local + transient-global attention to match HTML’s tree structure, plus a mixture of long-span denoising (mean span mu in {8,64}). The design must take this form because short NLP-standard spans (mu=3) mask low-signal fragments like </ or id=, and dense attention cannot afford 16K-token HTML. Second, a planner/summarizer/actor decomposition across specialized models aligned to real sites via “self-experience supervision” (scripted plans + program execution, with environment feedback filtering failures), avoiding manual step-level annotation.

Research questions

Can a real-website agent overcome open-domainness, long HTML, and missing HTML inductive bias? Specifically: does an HTML-specialized LLM beat general LLMs on HTML understanding, and does modular planning + summarization + program-synthesis beat a single prompted LLM on live sites?

Methodology

Three sub-problems, three specialist roles. HTML-T5 does closed-loop planning (predict the next NL sub-instruction from instruction + history + current HTML) and extractive summarization (predict data-ref IDs of task-relevant snippets). Flan-U-PaLM does grounded program synthesis, turning each sub-instruction + snippet into executable Python/Selenium. Actions are open-ended code (e.g. driver.find_element(...).click()), not a categorical action space.

Method details
  1. HTML-T5 architecture (§3.1, Fig.4) — Encoder swaps dense attention for local attention (radius r=127) + transient-global attention (block size k=16), following LongT5/ETC; decoder keeps dense attention. Initialized from PEGASUS weights (Appendix H.1 shows this is critical).
  2. Pre-training — Span-denoising with Gaussian span lengths, mixture mu in `{8,64}`, 15% masking, 4096 input / 910 output. Corpus: 100 CommonCrawl WARC files (Apr 2019), subtrees around <label for=...> elements -> 3.41M examples (avg 1020 tokens). T5X+SeqIO, TPU-v3 128 cores, 100K iters, batch 128 (Appendix C).
  3. Self-experience supervision (§3.2) — Sample instruction templates; rule-based parser decomposes into sub-instructions, regex retrieves reference IDs; Flan-U-PaLM generates programs; environment feedback filters failures; HTML-T5 finetuned on survivors (260 real-estate / 230 social-media / 410 map episodes).
  4. Grounded program synthesis (§3.3) — Flan-U-PaLM few-shot prompted with generic action examples + next sub-instruction + extracted snippet, decoding Selenium-executable Python; sub-instruction carried as a code comment.

Environments: three anonymized live sites (real-estate ~20 steps, social-media ~10 steps with many page transitions, map = easiest), driven via Selenium WebDriver under human supervision, rate-limited. Also MiniWoB++ (56 tasks), offline Mind2Web (137 sites, 3 generalization splits), static WebSRC. No public code/model release stated; anonymized sites are not reproducible.

Main result

On real sites WebAgent reaches 65% / 70% / 80% success (real-estate / social-media / map) vs ~10% for a single Flan-U-PaLM (Tab.1, p.7) — the “over 50%” headline is this absolute success-rate gap to a deliberately weak open-loop baseline, on only 20 instructions per site with no variance reported. Key findings

  • HTML-T5-XL gains +18.7% absolute on MiniWoB++ over WebN-T5-XL (Tab.3, p.8) (67.1 vs 48.4 with 12K demos) and beats LongT5/Flan-LongT5 at every size; a 3B HTML-T5 beats an 11B Flan-T5-XXL (85.6 vs 79.0) — but Synapse’s 99.2% SoTA stays far ahead.
  • Local+global attention drives the gain, not just length (Tab.2 left, p.7): 53.6% (L=4096) vs 35.3% dense Flan-T5-Base (+18%), with local-only at 44.0% (~+9%) — isolating the inductive-bias benefit.
  • The HTML-tuned denoising objective matters (Tab.2 right, p.7): span mu in {8,64} beats the NLP-standard {3,8,64}+Prefix-LM (MiniWoB++ 57.0 vs 55.2; real-estate 82.46 vs 80.56) and beats no-HTML-denoising (53.8 / 78.07).
  • HTML-T5-XL sets Mind2Web SoTA across all three splits, surpassing MindAct (GPT-4) (Tab.4, p.8) on element acc / op F1 / step SR — yet overall full-task SR stays 5-10%, so the “SoTA on Mind2Web” claim holds at the per-step level, not full-task.
  • Most residual real-world failures are planning errors (§4.1) (e.g. 70% of real-estate failures), per the authors’ error analysis — pointing at the planner, not the actor, as the bottleneck.
  • The generalist actor is swappable (Appendix K): gpt-3.5-turbo matches Flan-U-PaLM-540B on map (80%/93.8%), Flan-PaLM-62B gets 60%, but Flan-PaLM-8B = 0% (cannot synthesize valid programs) — a capability floor for code-as-action.

Implications

Treating long messy HTML as the core observation bottleneck — attacked with architecture + a domain-specific objective rather than just bigger windows — and using program synthesis as the action interface are both reusable beyond this system. Closest lineage: MindAct / Mind2Web (Deng et al. 2023) (also summarize-then-act, but offline with off-the-shelf models); builds on LongT5/ETC, UL2 mixture-of-denoisers, PEGASUS, and Code-as-Policies / PAL / Voyager for action-as-program. Whether Google ever released HTML-T5 weights or code, and how later web/computer-use agents tighten real-site evaluation rigor beyond 3 anonymized sites, are worth chasing.

Limitations

  • Tiny, non-reproducible real-world test: 3 anonymized sites x 20 instructions, no variance/CI, strict all-attributes-covered binary success.
  • The “over 50%” headline is a gap to a weak open-loop single-LLM baseline, not to any prior real-world agent (none comparable exists, which the authors acknowledge).
  • Strongest configuration depends on a proprietary 540B model; modular multi-model design adds compute/latency, and the program synthesizer is hard to give execution feedback to (left as future work).
  • Mixed WebSRC results: EM 75.50 / F1 85.75 beats single Flan-U-PaLM but trails TIE-Large (81.66 EM) and is weaker on KV/Table tasks (Tab.6, Fig.7).

Critiques and Questions

  1. The real-world claim rests on so few instructions and such a weak baseline that the magnitude, while plausible, is essentially uncheckable — anonymized sites make independent replication impossible.
  2. “SoTA on Mind2Web” is true per-step but full-task SR (5-10%) shows real-task success is still mostly unsolved; the framing risks overstating maturity.
  3. No statistical significance testing anywhere, and the headline conflates the HTML-T5 architecture win (well-evidenced) with the modular-agent win (thinly evidenced).
Why does WebAgent use a mixture of long denoising spans (mu in `{8,64}`) instead of the NLP-standard short spans?
Short spans (mu=3) tend to mask low-signal HTML fragments like `</` or `id=`; longer spans force the model to reconstruct meaningful HTML structure, which Tab.2 (right) shows yields higher downstream success (57.0 vs 55.2 on MiniWoB++).

Personal notes

Watch the “over 50%” claim: absolute success-rate gap vs a weak open-loop single-LLM baseline on 3 sites, not vs any prior real-world agent. Mind2Web “SoTA” is per-step; full-task SR is still 5-10%. Most reusable artifact is HTML-T5 (local-global + long-span denoising, PEGASUS init) — relevant whenever long structured-document context is the bottleneck.
