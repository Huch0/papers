AgentBench is the first systematic multi-environment benchmark that scores 29 LLMs as autonomous agents across 8 interactive, executable text environments (code, game, web), exposing a large gap between top commercial models and the best sub-70B open-source models.

Foundational reference for the LLM-as-agent evaluation lineage and a reusable harness design (HTTP server-client, Docker-isolated workers, max-flow scheduling) that predates and motivates SWE-bench, WebArena, OSWorld, and tau-bench.

Motivation

There was no systematic, standard benchmark for evaluating LLMs as agents in interactive environments. Prior text-game environments (TextWorld, Jericho, LIGHT) had closed/discrete action spaces and narrow commonsense focus; embodied simulators were complex but excluded text-only LLMs; and most agent benchmarks targeted a single environment, giving no cross-domain view (§1, §6).

Contribution (what’s new)

A multi-environment, text-only, executable benchmark that scores LLMs on real task success across diverse “groundings” under one toolkit. Agent evaluation is cast as a POMDP (S, A, T, R, U, O) and instantiated as 8 environments. The design must use real executable backends (bash, MySQL, Freebase, simulated web) graded by outcome metrics rather than text overlap, because the central question — can an LLM actually act — cannot be answered by static QA or text-similarity scoring.

Research questions

Can current LLMs operate as autonomous agents in realistic interactive environments; how large is the commercial-vs-open-source gap; and what are the dominant failure modes that block usable agents?

Methodology

8 environments in three groundings: Code (Operating System, Database, Knowledge Graph), Game (Digital Card Game, Lateral Thinking Puzzles, House-Holding), Web (Web Shopping, Web Browsing). Each exposes a textual observation stream and a per-environment action interface; the agent interacts over multiple rounds using only primitive Chain-of-Thought (Thought + Action per round, adapted from ReAct) with greedy decoding (temperature 0) and no reflection, search, or ensemble.

Method details
  • Environments (Tab.2): OS = Ubuntu bash in Docker (SR); DB = real MySQL (SR); KG = Freebase, >45M entities / 3B facts, partially observable (F1); DCG = Aquawar turn-based card game (Reward/win-rate); LTP = yes/no/irrelevant riddle host (Game Progress); HH = ALFWorld household tasks (SR); WS = WebShop simulated e-commerce (Reward); WB = Mind2Web web actions adapted to prompting, no fine-tuning (Step SR). Five of the eight are newly built for this benchmark.
  • Protocol: two-role dialogue (user = instruction + environment feedback; agent), 1-shot format demonstration, history truncated to ~3500 tokens with an omission marker (§2). Finish reasons categorized as Context Limit Exceeded (CLE), Invalid Format (IF), Invalid Action (IA), Task Limit Exceeded (TLE), and Complete.
  • Overall score (OA): each task’s raw scores are rescaled so the mean across all evaluated models is 1, then averaged with fixed reciprocal-mean weights so easy high-scoring tasks (e.g., Web Shopping) do not dominate (§3). Splits: Dev 269 / Test 1,014; ~3k + ~11k inference calls.
  • Harness: decoupled Task Server / Agent Server / Evaluation Client communicating over HTTP; isolated Docker workers; an agent-task bipartite graph scheduled via the Edmonds-Karp max-flow algorithm; resumable evaluation. Code/data/toolkit released at github.com/THUDM/AgentBench (Appendix A).

Main result

gpt-4 leads with OA 4.01; the best open-source model ≤70B, codellama-34b, reaches only 0.96 — below even gpt-3.5-turbo’s 2.32. (Tab.3, p.7) Averaged, API-based models score 2.32 vs 0.51 for OSS (p.8), and all API-based models exceed 1.00.

Key findings

  • Task Limit Exceeded (TLE) is the dominant failure mode (Tab.4, §4.3, p.8): the agent fails to finish within the round budget (e.g., LTP 82.5, KG 67.9), indicating weak long-term reasoning and decision-making rather than knowledge gaps.
  • Repeating previously generated content is the single biggest cause of TLE (Fig.9, §4.3): >90% of TLE trajectories contain a repeated response (high Rouge-L overlap), with models looping through a few states.
  • Different environments fail for different reasons (Tab.4, §4.3): Invalid Format dominates DB (53.3) and DCG (38.5) due to strict output formats, while Invalid Action dominates House-Holding (64.1) where models emit out-of-space actions.
  • Code training is “a double-edged sword” (§4.3, p.8): codellama edges llama-2 on procedural tasks (Web Shopping) but underperforms on Digital Card Game and OS — code pretraining does not uniformly help agent ability.
  • High-quality alignment data matters more than scale (§4.3, p.8): vicuna-13b (ShareGPT/gpt-4-distilled) beats llama-2-13b and rivals the 3x-larger codellama-34b; llama-2-70b (0.78) barely beats llama-2-13b (0.77), suggesting under-alignment of the 70B.
  • gpt-4 is best on 6 of 8 environments (Tab.3, §4.2) but is still not a practically usable agent (e.g., OS SR 42.4, DB SR 32.0, LTP 16.6).

Implications

Establishes the template — prompted LLMs inside real executable environments, scored by outcome — that the SWE-bench, WebArena, OSWorld, and tau-bench harnesses later extend; the reusable server-client/Docker/max-flow toolkit is a direct design precedent. Whether training on high-quality multi-round alignment data closes the OSS gap and how later scaffolds (reflection, tree search, tool augmentation) reduce TLE-via-repetition are the open threads.

Limitations

  • Author-stated: even gpt-4 is not a usable agent; OSS capped at ≤70B for compute; only primitive CoT (no reflection/search/ensemble).
  • Text-only — GUI/vision ignored (Web Browsing screenshots are optional and unused for scoring).
  • The OA metric uses a model-pool-specific normalization, so absolute OA values are not portable across model sets.
  • No statistical significance or variance — greedy single-run means (n=1 per item), so small differences (e.g., llama-2-13b vs -70b) may be noise; mechanistic claims are uncontrolled pairwise comparisons.

Critiques and Questions

  1. claude-3 and glm-4 were scored after the task weights were fixed (marked with *), so their OA values are slightly inconsistent with the rest of the pool — how much does this distort their ranking?
  2. With no variance reporting and greedy decoding, are the “code is double-edged” and “70B ≈ 13B” conclusions robust, or artifacts of single runs?
  3. Adapted datasets (WebShop, Mind2Web, Freebase) are static 2023 snapshots prone to drift and possible leakage; the interactive setting mitigates but does not eliminate this.
Why can't AgentBench's Overall (OA) scores be compared directly against a benchmark run on a different set of models?
OA normalizes each task so the mean score across the evaluated model pool equals 1, then applies fixed reciprocal-mean weights derived from that same pool — so the scale is defined by the 2023 model set and is ordinal within the paper, not an absolute cross-set scale.
