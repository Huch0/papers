SWE-bench turns 2,294 real merged GitHub issue→pull-request pairs into an execution-based benchmark where a model must edit a whole repository so the hidden, issue-resolving tests pass — and frontier models of the time resolved almost none.

This is the artifact that moved coding-agent evaluation from function-level synthesis to repository-level, test-validated issue resolution. Nearly everything in my SWE-agent track reports on a SWE-bench variant, so it is necessary context, not optional.

Motivation

Prior code benchmarks (HumanEval, MBPP) test self-contained functions with short context, and had largely saturated. They do not test what real software engineering demands: long-context code understanding, edits spanning multiple files, and interaction with an execution environment. The gap: a realistic, scalable, hard-to-game benchmark whose ground truth comes from real developer behavior.

Contribution (what’s new)

The novelty is the mining pipeline + test-gated metric, not a model. The insight that survives deleting “we propose”: a real merged PR that both resolves an issue and edits tests already contains a verifiable task — the gold patch is the reference, and the touched tests become an objective, execution-based grader. Because instances can be drawn continuously from new issues, the benchmark can refresh to resist contamination — a property a static, hand-written benchmark cannot offer.

Research questions

Can language models resolve real GitHub issues end-to-end — and how far below human/expert performance are they when judged by actual tests rather than string similarity?

Methodology

Mine merged PRs that (a) close an issue and (b) modify the repo’s test files; the gold patch is the solution and the touched tests are the grader. Give the model the issue text + the repo at the pre-PR commit; it must emit a patch. Success = applying the patch makes the relevant fail-to-pass tests pass while pass-to-pass tests stay green.

Method details (pipeline, retrieval, SWE-Llama)
  1. Scrape — Pull requests across 12 popular Python repos that reference/close an issue.
  2. Filter — Keep PRs that edit tests AND where the repo installs and tests go fail→pass with the gold patch.
  3. Formulate — Input = issue + codebase at base commit; output = a unified-diff patch applied with git/patch.
  4. Grade — Execution-based: fail-to-pass + pass-to-pass test gates; metric = % of instances resolved.

Long repos exceed context windows, so the main experiments feed code via BM25 retrieval (an “oracle” setting supplies the gold files). SWE-Llama fine-tunes CodeLlama on ~19k held-out-repo instances to study whether open models adapt to the patch-generation format.

Main result

The headline, verified in the text: the best configuration — Claude 2 with a BM25 retriever — resolves only 1.96% of issues (§1, §5); frontier models of late 2023 essentially fail the benchmark. Low scores partly reflect the BM25 retrieval bottleneck, not only reasoning/editing ability — the oracle-retrieval setting isolates this only partially.

Implications

Established the now-standard execution-based, repository-level evaluation and the contamination-resistant “issues after training cutoff” idea. Directly enabled the ACI/SWE-agent line and the SWE-bench-Lite / Verified / Multimodal derivatives.

Limitations

  • Test-derived ground truth can be under- or over-specified: a correct patch that differs from the gold one may fail, and the gated tests may not fully characterize the issue.
  • Python-only, 12 repos → limited external validity across languages/ecosystems.
  • The released static split is contamination-susceptible over time despite the refreshable design.

Critiques and Questions

  1. How much of the near-zero score is retrieval vs. reasoning? The oracle setting helps but doesn’t fully decompose it.
  2. Could a model “resolve” tests without truly fixing the issue (reward hacking the gated tests)? Worth probing on the harder instances.
Why is SWE-bench harder to game than a similarity-based code benchmark?
Success is decided by executing the repo's real fail-to-pass and pass-to-pass tests on the model's patch — not by string/semantic similarity to a reference — so superficially-plausible but non-working edits fail.
