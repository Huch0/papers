SWE-bench turns 2,294 real merged GitHub issue→pull-request pairs into an execution-based benchmark where a model must edit a whole repository so the hidden, issue-resolving tests pass — and frontier models of the time resolved almost none.

The artifact that moved coding-agent evaluation from function-level synthesis to repository-level, test-validated issue resolution. Nearly every SWE-agent paper reports on a SWE-bench variant.

Motivation

Prior code benchmarks (HumanEval, MBPP) test self-contained functions with short context and had largely saturated — they don’t test long-context code understanding, edits spanning multiple files, or interaction with an execution environment. The gap: a realistic, scalable, hard-to-game benchmark whose ground truth comes from real developer behavior.

Contribution (what’s new)

The novelty is a mining pipeline + test-gated metric, not a model. The insight that survives deleting “we propose”: a merged PR that both closes an issue and edits tests already contains a verifiable task — the gold patch is the reference and the touched tests are an objective grader. Because instances can be drawn continuously from new issues, the benchmark refreshes to resist contamination — impossible for a static, hand-written benchmark.

Research questions

Can language models resolve real GitHub issues end-to-end, and how far below human/expert level are they when judged by actual tests rather than string similarity?

Methodology

Give the model the issue + the repo at the pre-PR commit; it emits a patch. Success = applying it makes the fail-to-pass tests pass while pass-to-pass tests stay green.

Method details (pipeline, retrieval, SWE-Llama)
  1. Scrape — PRs across 12 Python repos that reference/close an issue.
  2. Filter — Keep PRs that edit tests AND where the repo installs and tests go fail→pass with the gold patch.
  3. Formulate — Input = issue + codebase; output = unified-diff patch; metric = % resolved.
  4. Grade — Execution-based: fail-to-pass + pass-to-pass gates.

Repos average ~438K lines — far beyond context windows — so the main runs use BM25 retrieval; an “oracle” setting supplies the gold files. SWE-Llama tests whether open models adapt to patch generation.

Key findings

  • Difficulty correlates with context length (§5, Fig.5): as total context grows, Claude 2’s resolve rate drops sharply — models are distracted by extra code and fail to localize the lines to edit. Even raising BM25’s max context (higher recall of the gold files) lowers performance.
  • Localization, not knowledge, is the bottleneck (Tab.6): “oracle”-collapsed retrieval (keep only edited lines ±15) lifts Claude 2 4.8%→5.9% and GPT-4 1.3%→3.4%.
  • Finetuned models are sensitive to context distribution shift (§5): SWE-Llama, trained on oracle context (edit-every-file), performs poorly with BM25 context where most provided files shouldn’t change.
  • Robust to contamination by date (Tab.7): before/after-2023 splits barely differ, so models aren’t “cheating” via memorized newer code.
  • Models generate shorter, simpler edits (§5, Tab.8) than gold patches; regenerating whole files is even worse than patches (Claude 2 2.2% vs 4.8%).

Implications

Established execution-based, repository-level evaluation and the refreshable, contamination-resistant design. Directly enabled the SWE-agent / agent-computer-interface line and the SWE-bench-Lite / Verified / Multimodal derivatives. Worth following up: how later agent scaffolds attack the localization bottleneck this paper exposes.

Limitations

  • Test-derived ground truth can be under- or over-specified: a correct patch differing from the gold one may fail, and the gated tests may not fully characterize the issue.
  • Python-only, 12 repos → limited external validity across languages.
  • The released static split is contamination-susceptible over time despite the refreshable design.

Critiques and Questions

  1. How much of the near-zero score is retrieval vs. reasoning, given the oracle-collapsed jump? A stronger retriever might change the story more than a stronger generator.
  2. Could a model pass the gated tests without truly fixing the issue (reward-hacking the tests)? Worth probing on harder instances.
Why does giving the model MORE context (higher BM25 limit) hurt resolve rate?
More context raises recall of the right files but also floods the model with irrelevant code; models get distracted and fail to localize the few lines to edit, so net performance drops — localization, not knowledge access, is the bottleneck (§5, Tab.2).
