Motivation

Building an agent that operates any smartphone app like a human, using only on-screen GUI gestures (tap/swipe), without back-end API access in the style of Siri (§1). Prior work falls short three ways: text-only LLM agents cannot perceive GUIs; fine-tuning embodied/GUI models needs large app-demonstration datasets that are costly to collect and may not generalize to unseen apps (§1); and GPT-4V can read typical UIs but struggles on new/atypical ones (§2.1).

Contribution (what’s new)

Two ideas survive deleting “we propose.” First, a coordinate-free action interface: interactive elements get integer IDs overlaid on the screenshot, so the model references tap(5) instead of predicting xy pixels (§3.1) — this form is forced by the observation that LLMs are bad at producing accurate coordinates (§4.2). Second, a learned per-app “document”: during an exploration phase the agent acts on each element and diffs before/after screenshots to write a natural-language description of what it does, later consulted at deployment (§3.2). Together these give cross-app generalization with no parameter updates — the document, not weights, holds the app knowledge.

Research questions

Can a multimodal LLM operate arbitrary smartphone apps purely through GUI gestures without training, and does learned per-app documentation (from autonomous exploration or human demos) meaningfully improve over plain prompting?

Methodology

A two-phase, training-free framework (Fig.2) running on Android via a command-line interface. Each step the agent sees a screenshot plus an XML list of interactive elements, each tagged with a unique integer ID rendered as a semi-transparent number on the screenshot (§3.1). The agent acts, then summarizes the step into a running memory carried into the next prompt.

Method details
  1. Observation interface — Per step: a real-time screenshot + an XML file of interactive elements. Each element gets an integer ID from its XML resource-id, or synthesized from class name + size + content, overlaid as semi-transparent numbers — a Set-of-Mark-style labeling scheme (§3.1).
  2. Action space — Four core functions over numbered elements: `Tap(element:int)`, `Long_press(element:int)`, `Swipe(element:int, direction, dist)`, `Text(text:str)` (writes directly into the focused field, bypassing the virtual keyboard), plus system-level `Back()` and `Exit()` (§3.1).
  3. Exploration phase — Goal-oriented (task-driven) trial-and-error: act on an element, diff before/after screenshots, write/refine its document entry; back out of irrelevant pages (e.g. ads) via `Back()` rather than blind DFS/BFS. A demo-watching variant records only the elements/actions a human used, shrinking the search space (§3.2).
  4. Deployment phase — For a new task, each step ingests the screenshot + dynamically retrieved document for visible elements, then emits Observation → Thought → Action (ReAct-style) and summarizes into carried-forward memory; terminates on self-judged completion via `Exit()` (§3.3).

A single GPT-4V plays observer + planner + executor in one loop; there is no separate verifier, and the document is the only learned component. Exploration capped at 40 steps, deployment at 10 steps (§4.1).

Main result

On the headline benchmark (45 tasks / 9 apps; Lightroom excluded for ambiguous completion), the full system reaches 73.3% success with auto-explored docs and 84.4% with demo-watched docs, approaching the 95.6% manual-doc oracle (Tab.1, p.7), versus a 48.9% no-doc baseline using the same action space.

Key findings

  • The simplified action space alone lifts success from 2.2% to 48.9% (Tab.1, p.7): coordinate-free element-ID actions are the single biggest driver — the model “struggles with producing accurate xy coordinates” (§4.2), and removing that requirement is most of the win.
  • A learned document adds a further large gain (48.9% → 73.3% auto, → 84.4% demos) (Tab.1, p.7): per-app memory matters on top of the action space, with watching demos beating autonomous exploration.
  • The “auto docs comparable to manual” claim is only loosely supported: auto (73.3%) and even demos (84.4%) still trail the manual oracle (95.6%) by a wide margin despite the paper’s framing.
  • In the Lightroom image-editing study, doc-equipped agents rank higher than the GPT-4 baseline (Tab.2, p.7): Avg. Rank 1.75 (manual) and 1.95 (demos) vs 2.30 baseline (lower is better), on 5 prepared images.
  • With a document the agent uses more editing tools (Tab.2, p.8): 5.8 (demos) and 4.0 (manual) tools vs 2.4 for the baseline — documents push it toward richer, multi-step manipulation.

Implications

This crystallized a recipe — ReAct-style loop (Yao et al., 2023) + Set-of-Mark-style numbered-element prompting — into a concrete mobile-agent design, and motivated later mobile/computer-use systems and benchmarks. Its distinctive delta over contemporaneous zero-shot GUI navigation (“GPT-4V in Wonderland”, Yan et al., 2023) is the persistent, exploration-learned per-app document; versus fine-tuned embodied models (RT-1/RT-2, Gato) it is training-free. How later reproducible, frozen-environment harnesses (AndroidWorld, Mobile-Agent, CogAgent) handle the live-app non-reproducibility this paper accepts. The project-page repo for the exact automation backend (adb/UIAutomator) and the document format.

Limitations

  • Author-stated: the simplified action space excludes multi-touch and irregular gestures (§5, Limitation).
  • Small, single-seed benchmark (45 tasks / 9 apps for the headline number) with no variance, CI, or significance reporting, and a single proprietary model (GPT-4V).
  • The Reward metric uses a custom reward model whose construction and validation are essentially unclear (§4.2); SR (binary, 10-step cutoff) is the more trustworthy figure.
  • Lightroom is excluded from the main table for ambiguity yet carries the vision claims via a tiny, subjective 5-image user study.
  • Task completion is self-declared via Exit() with author-defined success; no independent verifier or safety/permission layer is described.
  • Tasks run on live consumer apps whose UIs drift, so results are not reproducible against a frozen environment.

Critiques and Questions

  1. The “auto-docs comparable to manual” narrative overstates a 22-point gap (73.3% vs 95.6%) — how much of the manual oracle’s lead is doc quality vs exploration coverage?
  2. With self-judged Exit() and an under-specified reward model, how reliable are the absolute SR numbers, and would they survive a frozen-environment, multi-seed re-run?
  3. What are the failure modes of stale or incorrect documents as a UI evolves — the mechanism that is supposed to give “long-term applicability”?
Why does AppAgent reference integer element IDs instead of predicting screen coordinates, and what evidence shows it matters?
LLMs are poor at producing accurate xy coordinates (§4.2). Replacing raw coordinate actions with taps/swipes over numbered elements raises success from 2.2% to 48.9% (Tab.1) — the largest single jump in the ablation.
