SeeClick is a screenshot-only GUI agent built by continually pre-training Qwen-VL on ~1M auto-curated GUI grounding samples, showing that grounding is the bottleneck capability whose improvement directly lifts downstream agent performance.

One of the founding papers of the screenshot-in / coordinate-out computer-use paradigm, and the source of the widely reused ScreenSpot benchmark — a clean minimal baseline against richer accessibility-tree harnesses.

Motivation

Prior GUI agents act through structured text (HTML, DOM, Android view hierarchy), which has three concrete failure modes (Intro, p.1): it is not always accessible (iOS, desktop apps); it is verbose as LLM context and drops layout, images, and icons; and its variety forces task-specific observation/action spaces. The preliminary study (§3, p.2) surfaces the real blocker for a vision-only alternative: existing LVLMs ground poorly on GUIs because screenshots differ from natural images (dense text, many icons/widgets), and prior UI-grounding work was largely confined to a single 2017 Android dataset with no realistic cross-platform benchmark.

Contribution (what’s new)

Treat GUI grounding as plain language generation: given a screenshot and an instruction, the LVLM emits the location as natural-language numbers, e.g. click (0.49, 0.40), optimized by ordinary cross-entropy — no special coordinate-token vocabulary (§3.1, p.4). This must take this form because the baseline fails at grounding, not at planning: rather than 1000-bin coordinate tokens (Pix2Act/Pix2Seq), normalized text coordinates reuse the LVLM’s existing generation head, so all capacity goes into learning UI layouts. The recipe is continual pre-training of a grounding-capable backbone on ~1M auto-curated GUI samples, plus ScreenSpot, the first realistic cross-platform grounding benchmark.

Research questions

Can a GUI agent run on screenshots alone, with no metadata? Is grounding the capability that gates downstream agent success — i.e., does improving grounding measurably raise task performance?

Methodology

Reformulate grounding (p(y|s,x)) and OCR/captioning (p(x|s,y)) as text generation over normalized two-decimal [0,1] coordinates, then continually pre-train Qwen-VL (LoRA on both vision encoder and LLM) on a ~1M mix before per-task fine-tuning.

Method details: data, backbone, ScreenSpot construction
  1. Grounding as generation — Query prompt like 'In the UI, where should I click if I want to <instruction>?'; ground-truth target 'click (0.49, 0.40)'; cross-entropy loss. Coordinates are plain text, not a 1000-token vocabulary (§3.1, p.4).
  2. Web data (~300k pages) — From Common Crawl; elements = (a) visible-text elements and (b) elements with a 'title' attribute. Tasks: text-to-point, text-to-bbox, point-to-text, bbox-to-text (web OCR) (§3.2, p.4).
  3. Mobile data — Widget-captioning data (Li et al. 2020b: ~20k screenshots / 40k widgets / 100k descriptions) reversed into grounding; plus RICO-derived elements; plus UI summarization (Screen2Words) (§3.2, p.4).
  4. General data — LLaVA instruction-following data to preserve natural-image capability. Total ~1M with 30 prompt templates per task (§3.2, p.4).
  5. Training — Qwen-VL backbone (448x448 resolution, existing grounding ability), LoRA on vision encoder + LLM, ~10k steps (~1 epoch) (§3.3, p.4).
  6. ScreenSpot benchmark — >600 screenshots, 1200+ instructions across iOS, Android, macOS, Windows, web; explicit Text vs Icon/Widget split; human-annotated and curated to avoid training overlap; web pages from WebArena categories (§4, p.5). Metric: predicted click point inside the ground-truth bounding box.

Main result

On ScreenSpot grounding, SeeClick (9.6B) reaches 53.4% average click accuracy, beating GUI-specific CogAgent (18B) at 47.4% despite fewer parameters, and crushing generalist LVLMs: GPT-4V 16.2%, Qwen-VL 5.2%, MiniGPT-v2 5.7% (Tab.1, p.5). The grounding gain transfers to agent tasks: on MiniWob, SeeClick scores 67.0% on the 35-task vision overlap, beating Pix2Act (64.6%) while using less than 0.3% of its 1.3M training data (Tab.2, p.6).

Key findings

  • GUI grounding pre-training, not the backbone, drives the gain (Tab.2, p.6): at an equal 2.8K episodes SeeClick (Image, 73.6%) beats WebGUM (HTML+Image, 65.5%), and it beats its own Qwen-VL backbone by ~19 points (67.0 vs 48.4) — same model, only the grounding pre-training differs.
  • On AITW (instruction-wise split), SeeClick leads at 59.3 overall vs Qwen-VL 54.3 and GPT-4V 50.5, with click accuracy 66.4 vs 57.4 — the stated “9% increase in click accuracy” (Tab.3, p.7): precise clicking is the lever, consistent with the grounding thesis.
  • On Mind2Web (vision-only), SeeClick nearly doubles the Qwen-VL baseline — Cross-Task Step SR 25.5 vs 13.3, Ele.Acc 28.3 vs 15.9 (Tab.4, p.8): but it still trails HTML-based MindAct (Cross-Task Step SR 52.0), so vision-only real-web navigation remains hard.
  • Across several SeeClick checkpoints, higher ScreenSpot grounding consistently tracks higher MiniWob/AITW/Mind2Web scores (Fig.6, §5.2.4, p.8): the headline grounding↔downstream correlation — though shown qualitatively, with no correlation coefficient reported in-text.
  • All models, including SeeClick, struggle on Icon/Widget elements (SeeClick Desktop-Icon 30.0%, Web-Icon 32.5%) (Tab.1, p.5): non-text grounding is the benchmark’s hard axis, and resolution is the suspected bottleneck (SeeClick even loses to CogAgent on Desktop-Text 72.2 vs 74.2 and Web-Text 55.7 vs 70.4).
  • A single unified agent jointly trained on all three downstream tasks is feasible but not free (Tab.5, p.8): e.g. MiniWob drops 67.0 to 64.1 vs per-task training.

Implications

Establishes the vision-only GUI agent contract — screenshot in, normalized coordinate out, with no metadata — and gives the field ScreenSpot, now a standard grounding benchmark plus the grounding-first recipe that CogAgent, UGround, OS-Atlas and later grounding models build on. Whether the grounding↔agent link is causal or just co-driven by general UI exposure, and how much later gains come purely from higher-resolution / tiled screenshots, are the live threads.

Limitations

  • Author-stated: action space simplified to mainly click/type, excluding drag and double-click; agent-specific fine-tuning is still required per task (not zero-shot) (Limitations, p.9).
  • The central grounding→agent correlation is qualitative — no correlation statistic, and the “grounding” effect is not isolated from general UI pre-training exposure.
  • Low 448x448 backbone resolution caps fine-grained desktop/web text grounding and forces cropping hacks on Mind2Web’s long pages.
  • ScreenSpot is small (>600 screens / 1200+ instructions) and baselines use their own recommended prompts, adding prompt-sensitivity confounds (especially GPT-4V).

Critiques and Questions

  1. Figure 6’s underlying numbers are not in the text, so “grounding correlates with agent performance” rests on a checkpoint trend plus cross-baseline deltas — suggestive, not a controlled causal test.
  2. Honest evaluation is a genuine strength: matched MiniWob task subsets and the AITW instruction-wise re-split (which lowers the headline AITW number vs the overfitting-prone episode-wise split, Tab.7) make the comparisons fairer than usual.
  3. Given the Icon/Widget weakness tracks resolution, how much of SeeClick’s deficit vs CogAgent is method vs the 448x448 cap alone?
Why does SeeClick emit coordinates as plain text rather than as special coordinate tokens, and what does the Qwen-VL comparison show this buys?
Plain-text normalized coordinates reuse the LVLM's existing language-generation head with no new vocabulary, so all learning capacity goes into UI layouts rather than a new tokenization. The Qwen-VL comparison (same backbone, no grounding pre-training) trailing SeeClick by ~19 points on MiniWob (48.4 vs 67.0) shows the gain comes from grounding pre-training, not the backbone.

Personal notes

The cleanest takeaway for harness design: the entire agent contract is “screenshot in, normalized click/type out, k=4 action history” with zero metadata — a useful minimal baseline to compare richer accessibility-tree harnesses against. ScreenSpot’s Icon/Widget vs Text split is the most reusable diagnostic here. Watch the resolution bottleneck: many later gains come simply from feeding higher-res / tiled screenshots.
