A vision-first web agent that drives live, real-world websites end-to-end by acting on marked-up screenshots with an off-the-shelf multimodal LLM, paired with a GPT-4V “judge” that scores open-ended trajectories automatically and a 643-task / 15-site benchmark.

Foundational for my computer-use / web-agent track: it popularized the “screenshot + Set-of-Mark + ReAct on a real browser” recipe and the “VLM-as-judge for open-ended trajectories” evaluation pattern that later agent papers reuse.

Motivation

Prior web agents share two gaps. (1) They consume only HTML / accessibility-tree text and run inside simulators or static page snapshots, ignoring that browsers render pages into visual layouts; verbose HTML is hard to manage and discards layout cues. (2) Benchmarks like Mind2Web grade each action against one predefined “golden” trajectory, which cannot credit the many valid ways to finish a task. Together these limit applicability to the messy live web (ads, pop-ups, constant updates) and make fair, scalable evaluation of open-ended end-to-end success hard.

Contribution (what’s new)

A generic VLM can complete tasks on unmodified live websites by acting on a screenshot whose interactive elements are overlaid with numbered boxes — with no object-detection module and no fine-tuned element selector — and such open-ended runs can be graded automatically by a second VLM watching the screenshot trajectory. Mechanistic why: rendered pages encode information visually, and numbering elements lets the model refer to a target by index (Click [17]) instead of solving pixel-coordinate grounding, so one model can both see and act; the same numbers let the judge re-derive what happened without a golden path. Closest prior work: SoM (reused, but drops the object detector — elements extracted rule-based), WebArena (not self-hosted — runs the open web), and concurrent SeeAct (WebVoyager needs no extra modules).

Research questions

Can an off-the-shelf multimodal LLM complete real-world tasks end-to-end on live websites using only marked-up screenshots plus minimal element text? And can a VLM reliably stand in for a human when scoring such open-ended trajectories?

Methodology

Given an instruction, WebVoyager drives a real browser (Selenium) in an observe→think→act loop. Each observation is the current screenshot with numbered Set-of-Mark boxes plus per-element text (type, inner text, aria-label); the model emits a ReAct Thought then one Action from a fixed set, terminating with ANSWER. The evaluator is a separate GPT-4V given the task, the agent’s response, and the last k trajectory screenshots, returning binary SUCCESS / NOT SUCCESS.

Method details
  1. Observation — Screenshot at 1024x768 with numbered boxes drawn on interactive elements via the rule-based GPT-4V-Act JS tool (no object detector; single black box color chosen empirically). Auxiliary text per element: type, embedded text, aria-label. Context clipped to the 3 most recent observations but full Thought/Action history kept.
  2. Action space — Click [n]; Type [n];[text] (clears box + auto-ENTER); Scroll [n/WINDOW];[up/down]; Wait; GoBack; Google (jump to a search engine to escape dead ends); ANSWER;[content] (terminate).
  3. Loop budget — Max 15 steps, ~7000 tokens; execution errors fed back and consume a step. Single tab; agent temperature 1.
  4. Benchmark — 643 tasks across 15 popular sites (40-45 each), built by self-instruct from Mind2Web seeds + human verification; 22.3% have stable 'Golden' answers, rest 'Possible' (open-ended / real-time). Plus 90 GAIA web tasks (Level 1+2) and 50 SeeAct online tasks. Login/CAPTCHA sites excluded.
  5. Evaluation — Human binary judgment (3 annotators on 300 tasks) is primary; GPT-4V auto-judge over last k screenshots at temperature 0 reported as mean +/- std over 3 runs. Backbones: GPT-4V (gpt-4-vision-preview, primary), Claude 3 Opus, GPT-4o.

Main result

WebVoyager reaches 59.1% human-judged task success (GPT-4V backbone) vs 40.1% text-only and 30.8% GPT-4 (All Tools) (Abstract; Tab.1, p.7), and its GPT-4V auto-judge hits 85.3% agreement with humans, Cohen’s kappa = 0.70 — matching the human inter-annotator kappa of 0.70 (Tab.2, p.7). The headline 59.1% is human-judged and appears single-run; only the GPT-4V-judged numbers carry variance (the same agent auto-judges to 57.1% +/- 0.2%), so abstract 59.1% vs Table 1 57.1% reflect different judges, not different agents.

Implications

A simple, module-free recipe (off-the-shelf VLM + rule-based marking + ReAct on a live browser) became a de-facto baseline architecture for multimodal computer-use agents, and the validated VLM-as-judge protocol gave the field a scalable way to score non-deterministic, open-ended agents without golden trajectories.

Limitations

Live-web evaluation is inherently non-reproducible — numbers are a snapshot in time as sites, ads, and real-time data change. The headline 59.1% (human-judged) appears single-run with no variance; abstract 59.1% vs Table 1 auto-eval 57.1% are easy to conflate. Self-evaluation bias: each backbone’s judge favors its own outputs (Tab.3), confounding the cross-backbone comparison (GPT-4o lenient, GPT-4V strict, Claude prefers itself). Site selection excludes login/CAPTCHA sites, biasing toward agent-friendly tasks; reads poorly on text-heavy pages; no Drag, single tab, 15-step budget. Documented failures: Navigation Stuck 44.4%, Visual Grounding 24.8%, Hallucination 21.8%, Prompt Misalignment 9.0% (Tab.4).

Critiques and Questions

  1. The “both modalities needed” claim is argued from per-site splits, not from an ablation that feeds HTML text into the vision agent — the cleaner test would add accessibility text to WebVoyager and measure the lift.
  2. Cross-backbone scores (Tab.3) are confounded by self-evaluation bias; a single fixed neutral judge across all backbones would make the GPT-4V / Claude / GPT-4o comparison fair.
  3. Headline 59.1% is single-run human-judged — I’d want repeated human passes or at least variance before treating the GPT-4V-vs-baseline gap as robust.
  4. Site selection avoids login/CAPTCHA; how much does success drop on the harder, gated web that real assistants must handle?
