Mobile-Agent is a purely vision-based Android GUI agent that drives apps from screenshots alone — using OCR + icon-detection + CLIP to ground GPT-4V’s text/icon click intents into pixel coordinates — evaluated on a new 10-app, 3-difficulty benchmark (Mobile-Eval).

An early (Jan 2024) landmark in the mobile/computer-use agent lineage that stakes out the “no view-hierarchy, screenshot-only” design point — directly relevant to harness design — and introduces a reusable real-app benchmark. A design reference more than a results anchor.

Motivation

GPT-4V can decide what to do on a screen but cannot reliably say where to do it — it “struggles to accurately locate the positions of these operations on the screen” (p.2, citing Zheng et al. 2024). Prior fixes inject UI layout files to recover clickable coordinates: AppAgent reads Android XML view-hierarchies and SeeAct reads web HTML — but these require file-access permissions that “may not be available” in many scenarios (p.2), making them OS- and app-specific.

Contribution (what’s new)

Decouple decision from grounding: GPT-4V plans and emits a symbolic action, and a separate pipeline of off-the-shelf perception tools (OCR, Grounding DINO, CLIP) translates that target into pixel coordinates from the screenshot alone (p.2–4). The design must take this form because the baseline failure is specifically localization, not planning — so the fix is an external grounding module rather than a better prompt, and dropping the view-hierarchy is what buys OS-agnostic portability. Second contribution: Mobile-Eval, a real-app benchmark with a difficulty gradient (concrete → abstract → multi-app) and four complementary metrics (p.2, p.5).

Research questions

Can a mobile agent localize and execute operations from screenshots only — with no XML/view-hierarchy/system metadata — and still complete realistic, multi-step, multi-app instructions on live mainstream apps?

Methodology

Two coupled components plus a benchmark. A vision-only perception module locates targets, and a ReAct-style loop lets GPT-4V plan, act, and self-reflect over a fixed 8-operation vocabulary.

Method details
  1. Text grounding (OCR) — An OCR tool finds the target text box; the agent clicks its center. Three cases (p.3): (a) no match → reselect or pick another op (attributed to GPT-4V hallucinations); (b) one match → click center; (c) multiple → if many, reselect; if few, crop each region, expand the box outward, draw detection boxes, and feed the crops back to the agent to disambiguate.
  2. Icon grounding (DINO + CLIP) — The agent describes the wanted icon's color/shape; Grounding DINO with the prompt `icon` detects all icons; CLIP scores each detected crop against the description and the highest-similarity box is clicked (p.4).
  3. 8-operation vocabulary — Open App, Click text, Click icon (with a `Position` hint in `{top,bottom,left,right,center}`), Type, Page up/down (scroll), Back, Exit, Stop (p.4). Concrete taps/types/scrolls execute at coordinates from the perception pipeline — not from the LLM directly.
  4. Self-Planning loop — Each iteration captures a screenshot and gives the agent the system prompt (built from the instruction), operation history, and current screenshot; it outputs the next op until it emits Stop (p.4). ReAct-style Observation / Thought / Action format.
  5. Self-Reflection — Two triggers (p.4): (1) screenshot unchanged or wrong page after an op → retry / alter parameters; (2) after self-planning declares done → re-check screenshot + history + instruction for overlooked sub-requirements, else resume.

Training: none — Mobile-Agent is training-free, prompt-engineered around frozen GPT-4V and frozen perception tools.

Benchmark (Mobile-Eval, p.5): 10 mainstream apps (Alibaba.com, Amazon Music, Chrome, Gmail, Google Maps, Google Play, Notes, Settings, TikTok, YouTube) plus a Multi-App tier requiring information transfer between two apps. Three instructions per app of rising difficulty: basic op; basic + extra requirement; abstract instruction that does not name the app. Four metrics: Success (binary), Process Score (correct steps / total steps), Relative Efficiency (agent steps vs human-optimal steps), Completion Rate (human steps completed / total human steps; =1 if fully done).

Main result

On Mobile-Eval (no baseline), per-instruction-tier averages were Success 0.91 / 0.82 / 0.82, Process Score 0.89 / 0.77 / 0.84 (~80%), and Completion Rate 98.2% / 90.9% / 91.3% (Tab.2, p.6) across tiers 1/2/3. Relative Efficiency was 4.9/4.2, 7.9/6.3, 7.5/6.2 agent-vs-human steps — about 17–25% more steps than the human optimum, summarized as “80% capability of reaching human-optimal operations” (Tab.2, p.6) (p.6).

Key findings

  • Self-reflection rescues low-accuracy trajectories (Tab.2, p.6): several cells have Process Score < 1 yet still succeed (e.g. Notes Instruction 1: PS 0.57, success), supporting the claim that mid-trajectory errors get corrected (p.6).
  • Difficulty hurts step accuracy more than final success (Tab.2, p.6): PS dips to 0.77 on tier 2 while Success holds near 0.82 — harder instructions cause more wrong steps that reflection partially recovers from.
  • Cross-app information transfer works (Fig.6, Fig.7, p.6): the agent carries data from one app into another (e.g. read a calendar date → write it into Notes), relying on in-context operation history as memory.
  • Vision-only perception handles a Chinese-language system (Fig.8, p.6) despite GPT-4V’s weaker Chinese text generation — perception is decoupled from language fluency.
  • Reporting inconsistency: the prose calls 91% / 82% / 82% “completion rates,” but those are the Success averages — the actual CR averages in Tab.2 are 98.2% / 90.9% / 91.3%. The table is authoritative; the prose conflates Success with Completion Rate.

Implications

Establishes the screenshot-only / no-view-hierarchy point as a viable design and provides a named benchmark for it. Differentiates from AppAgent (Yang et al. 2023d), which tags UI via Android XML, and from SeeAct (Zheng et al. 2024), which grounds web agents via HTML; borrows Grounding DINO and CLIP wholesale and ReAct for the prompt format. Compare vision-only accuracy/latency against view-hierarchy-based agents and later set-of-mark / native-grounding approaches; track Mobile-Agent-v2 for added baselines and larger eval.

Limitations

  • No baselines and no ablations — the central “vision-only is enough / replaces XML” claim is never tested head-to-head, and the contributions of OCR vs Grounding DINO vs CLIP vs self-reflection are not isolated (the single largest evidentiary gap).
  • Tiny, single-run, author-judged evaluation: ~30 instructions total, no repeated runs, no seeds, no variance or CI; success judged by the authors and not blinded (p.5).
  • Latency and cost unaddressed — a per-step pipeline of GPT-4V + Grounding DINO + CLIP + OCR is plausibly slow/expensive, but no numbers are reported.
  • Grounding ceiling inherited from the tools (OCR/detector failures, CLIP mismatches) with no error taxonomy; the OCR engine identity and GPT-4V snapshot are Not reported.
  • Android-only by the authors’ own admission; other OSes left to future work, and no safety/confirmation gate despite executing real irreversible actions (sending emails, posting comments, adding to cart) (p.4).

Critiques and Questions

  1. Without an XML-based control (AppAgent) on the same tasks, the “vision is enough” thesis is asserted, not measured — how much of the success is the perception pipeline vs GPT-4V’s planning alone?
  2. The SU-vs-CR conflation in the prose signals loose reporting; when citing, quote Tab.2 (SU 0.91/0.82/0.82; CR 98.2%/90.9%/91.3%), not the p.6 sentence.
  3. Live-app non-determinism (network, changing content) makes the benchmark hard to reproduce exactly — how stable are these single-run numbers?
Why does Mobile-Agent need OCR, Grounding DINO, and CLIP at all if GPT-4V already decides the action?
Because GPT-4V can choose the correct operation but cannot reliably output where on the screen to perform it. The perception tools ground a symbolic target (a text string or an icon description) into pixel coordinates: OCR locates text boxes, Grounding DINO detects all icons, and CLIP matches the detected icons to the agent's description — replacing the XML view-hierarchy that prior agents relied on.

Personal notes

Watch the SU-vs-CR conflation when citing this paper — quote the table (SU 0.91/0.82/0.82; CR 98.2%/90.9%/91.3%), not the p.6 sentence. The icon-grounding recipe (describe attributes → Grounding DINO → CLIP) is the most reusable nugget. Compare its accuracy/latency against view-hierarchy-based agents (AppAgent) and against later set-of-mark / native-grounding approaches.
