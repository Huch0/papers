OS-Copilot gives an LLM agent a single universal code-execution interface to the whole OS, and the FRIDAY agent built on it autonomously writes, scores, and accumulates reusable Python tools — letting it generalize to unseen apps and self-improve from 0% to 60% on a spreadsheet benchmark.

An early (Feb 2024) move from single-sandbox agents (one browser, one terminal) toward the full OS as the environment. It is a reusable blueprint — planner + memory + actor-with-critic, plus self-directed tool acquisition — that maps cleanly onto today’s computer-use agents.

Motivation

Prior language agents are confined to a narrow domain — one website, one terminal, one game, one database — which limits applicability to general computer tasks (p.2). Two stated obstacles to OS-level agents (p.2): there is no unified interface across a heterogeneous OS (code, keyboard/mouse, APIs), and the sheer number of distinct applications makes hand-crafting per-app tools and prompts impractical, hurting generalization. Even multi-scenario frameworks (AutoGPT, OpenAgents) are claimed to still “operate within the confines of either a terminal or a web browser” (p.8).

Contribution (what’s new)

Two coupled pieces. (1) Code execution as the universal action interface: rather than engineering a tool per app, the runtime consolidates Python interpreter, bash, API POST, and mouse/keyboard behind one interface (p.2, Tab.4 p.13). This must take this form because per-app tooling cannot scale to “every application” — code is the lowest common denominator that already drives most apps. (2) Self-directed tool accumulation: given an objective (e.g. “master spreadsheets”), FRIDAY self-instructs a curriculum, solves it by trial and error, and keeps tools the critic scores above 8/10 into a persistent repository (p.5-6). The individual components (ReAct, Reflexion, Voyager-style skill libraries) are explicitly not claimed as new (p.7); the genuine delta is unifying them into one working OS-level system.

Research questions

Can a single code-execution interface plus a swappable planner/memory/actor scaffold drive a generalist agent across the whole OS, and does autonomous tool accumulation measurably improve performance on unseen applications?

Methodology

Per request the loop is planner -> configurator (memory) -> actor (executor + critic + refiner), each component swappable (p.3, Fig.2). Memory follows a biological metaphor: declarative (user profile + semantic knowledge), procedural (the tool repository), and working memory linking the components (p.3-4). FRIDAY is prompting-based — no weight updates (p.13).

Method details
  1. Plan — Planner decomposes the request into subtasks, retrieving in-house tools and OS info. A proposed DAG planner encodes subtask dependencies so independent subtasks run in parallel via subagents (p.3, App. A).
  2. Configure — For each subtask the Configuration Tracker uses dense retrieval to pull relevant tools/profiles/OS state into the prompt. If no tool clears a similarity threshold, the Tool Generator writes a new Python class subclassing BaseAction with a __call__ method, e.g. AppleScript to set dark mode (Fig.3, Tab.13).
  3. Act — Executor emits an executable action — a bash command or a generated-tool invocation — run in the OS runtime. Critic judges completion (boolean) from pre/post system state (OS version, cwd, files, stdout/stderr), returns error analysis, and scores tool generality 1-10 (Tab.9). Refiner self-corrects; the loop runs up to 3 attempts per subtask (p.5).
  4. Learn — For self-directed learning FRIDAY self-instructs ~10 tasks (e.g. Excel via openpyxl), solves them, and accumulates tools scoring >8 into the persistent repository — 8 tools for spreadsheets (p.6, §4.2).

Observation is text-only: system state before/after execution plus critic feedback. Web access is retrieval-only (no clicks/forms) and there is no screenshot/GUI grounding in this version (p.13).

Main result

On GAIA, FRIDAY reaches 40.86% on Level 1 vs the prior best 30.30% (GPT-4 Plugins) — a 35% relative gain (Tab.1, p.6), and is the only system non-zero at Level 3 (6.12% vs 0% for all baselines).

Key findings

  • Self-directed learning lifts spreadsheet Pass@1 from 0% to 60% (Tab.2, p.7): FRIDAY without learning fails every SheetCopilot-20 task (defaults to pandas/matplotlib); after self-generating and solving just 10 tasks and keeping 8 tools, it surpasses SheetCopilot/GPT-4 (55%), /Claude (45%), and /GPT-3.5 (40%) — and all FRIDAY tools are auto-generated vs SheetCopilot’s hand-crafted operations.
  • On GAIA the learning gain is real but modest (Tab.1, p.6): +4.30 pts at L1 (36.56→40.86) and +2.52 pts at L2 (17.61→20.13), and exactly zero at L3 — most of FRIDAY’s headline lead over baselines comes from the scaffold itself, not from self-learning.
  • FRIDAY is time-efficient (Tab.3, p.6): ~118s vs AutoGPT-4’s 456s on L1 (~4x faster) and ~3x faster than humans; it is slower than GPT-4 Plugins, but that excludes human plugin-selection time.
  • Capability is lopsided (Fig.5b): strong on coding and file I/O, weak on web browsing (retrieval-only, no clicks) and multimodality — the gaps trace directly to the text-only, click-less interface.
  • Generality comes from code, not a fixed app list (p.6): seeded with only 4 basic tools (+9 auto-generated on the GAIA dev set), the framework never enumerates “apps controlled” — new capability is written on demand.

Implications

A clean, reusable architectural blueprint (planner / memory-configurator / actor-with-critic+refiner) for OS-level agents, and concrete evidence that autonomous tool accumulation pays off. Closest analogue is Voyager (Wang et al. 2023a) — lifelong skill library, here ported from Minecraft to the OS; the primary contrast is AutoGPT (shares memory + tool repo + task decomposition; FRIDAY’s edge is self-criticism/refinement and self-directed tool generation), with component lineage from ReAct, Plan-and-Solve, Reflexion, Toolformer, and Self-Instruct. Watch later versions / later computer-use agents for the web-action and visual-grounding (screenshot→action) gaps this paper flags.

Limitations

  • Author-stated (p.13-14): heavy reliance on prompt engineering (brittle to LLM changes), no support for closed-source GUI-only apps, no multimodal/screenshot-to-action, subtask-completion judged by an LLM critic without ground truth (called potentially inaccurate and unscalable), and safety/interpretability/alignment left as open work.
  • Inferred: small evaluation sets with single runs and no variance/CIs — the 0%→60% headline is 12 of 20 tasks, and GAIA L3 (~6%) is near-floor where small-count noise dominates. Per-capability numbers use the authors’ own annotations, which they admit differ from the server.
  • The agent runs generated code directly in the live OS with no described sandbox or permission gating — an operational safety risk.

Critiques and Questions

  1. The “surpasses a system designed for spreadsheets” claim rests on a +5-pt margin (60 vs 55) over a different scaffold with different assumptions — suggestive, not decisive.
  2. Most of the GAIA lead is the scaffold, not self-learning (which adds ~0-4 pts); how much of the framework’s value is just GPT-4-turbo plus a good harness?
  3. No fine-tuning baseline and everything is prompt-dependent — would the architecture survive a model swap, given the authors’ own warning?
