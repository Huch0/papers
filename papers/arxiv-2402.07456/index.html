One-sentence takeaway

OS-Copilot is a framework that gives an LLM agent a single universal interface (Python, bash, API calls, mouse/keyboard) to the whole OS, and the FRIDAY agent built on it adds self-directed learning that autonomously writes and accumulates reusable Python tools, letting it generalize to unseen applications and self-improve from 0% to 60% on a spreadsheet benchmark.

Why this paper matters

This is an early (Feb 2024) attempt to move agents beyond a single sandbox (a browser, a terminal) toward the full OS as the environment, with the web, files, multimedia, and third-party apps all as targets. Two ideas remain central to current computer-use agent research: (a) treating code execution as the universal action interface rather than per-app tool engineering, and (b) self-directed tool acquisition (Voyager-style skill libraries) applied to general computer tasks rather than Minecraft. It is a useful reference point for the design pattern “planner + memory + actor with self-criticism” and for what self-improvement buys you empirically. CLAIMED contributions are framework-level and demonstration-level rather than a new training method, so its value is more architectural-blueprint than benchmark SOTA that would survive today.

Problem and gap

CLAIMED problem: prior language agents are confined to a narrow domain (one website, one terminal, one game, a database), which limits applicability to general computer tasks (p.2). Two stated obstacles to OS-level agents (p.2):

  1. No unified interface to interact with a heterogeneous OS (code, keyboard/mouse, APIs).
  2. The huge number of distinct applications makes hand-crafting per-app tools/prompts impractical, hurting generalization and scalability.

The gap relative to prior multi-scenario frameworks (AutoGPT, OpenAgents) is CLAIMED to be that those still “operate within the confines of either a terminal or a web browser and can hardly interact with other applications” (p.8).

Core idea

Two coupled contributions:

  • OS-Copilot framework: a universal runtime that consolidates four OS interaction methods — Python interpreter, bash terminal, API calls (POST), and mouse/keyboard control — behind one interface (p.2, Table 4 p.13). The agent loop is planner -> configurator (memory) -> actor (executor + critic), each component swappable (p.3, Fig. 2).
  • FRIDAY agent: instantiates OS-Copilot and adds (1) self-refinement (critic/refiner loop, up to 3 attempts per subtask) and (2) self-directed learning — given a learning objective (e.g., “master spreadsheets”), FRIDAY self-instructs a curriculum of tasks, solves them by trial and error, and accumulates the resulting Python tools into a persistent repository (p.5-6). Tools scoring above 8/10 from the critic are kept for reuse (p.5).

The configurator is organized around a biological-memory metaphor: declarative memory (user profile + semantic knowledge), procedural memory (tool repository = skill set), and working memory (short-term hub connecting planner/configurator/actor) (p.3-4).

Harness relevance

  • Environment / workspace: an entire operating system (Linux and MacOS), spanning the web, code terminals, files, multimedia, and third-party applications (p.1-2). Figure 1 shows FRIDAY running on MacOS.
  • Observation interface: system state gathered before/after execution and fed to the critic/configurator — OS version, current working directory, files/folders in the working directory, execution stdout/stderr, and critic feedback (p.4-5, prompt Tables 5-9). Note: web is read-only (retrieval) and there is no general screenshot/GUI observation in this version (p.13).
  • Action interface: tool/API invocation plus code execution. The executor emits an executable command or function call (bash command, or Class()(args) invocation of a generated Python tool) run through the universal runtime (Python/bash/API/mouse-keyboard) (p.4, Tables 6/8/13).
  • Tool/API/shell/GUI layer: four interaction methods (Python interpreter, bash, API POST services, mouse/keyboard). Tools live either as deployed API services or as Python files in the tool repository (p.4). GUI control via mouse/keyboard is listed as supported but not the focus; the paper notes web actions (clicks) are not supported (p.13).
  • Planner/executor/verifier/search structure: planner decomposes a request into subtasks; supports Plan-and-Solve and a proposed directed-acyclic-graph planner enabling parallel independent subtasks via per-subtask subagents (p.3, Appendix A p.11). Actor = executor + critic; critic judges subtask completion (boolean), gives error analysis, suggests subtask restructuring, and scores tool generality 1-10 (Table 9). Refiner does self-correction; loop iterates up to 3 attempts (p.5).
  • Evaluation harness: GAIA benchmark, results submitted to the official HuggingFace evaluation server for the private test set (p.6). For per-capability breakdown the authors manually annotated the 93 level-1 tasks, noting their annotations “slightly differ” from the server (p.12-13).
  • Training harness: none — FRIDAY is prompting-based (GPT-4-turbo-1106 for GAIA; GPT-4 for the spreadsheet study). Authors explicitly note no fine-tuning and that they followed an OpenAI-Gymnasium-style interface to enable future RL/fine-tuning (p.13). “Self-directed learning” here means autonomous tool accumulation, not weight updates.
  • Logging/trace/reproducibility: trajectories and acquired knowledge are stored in semantic memory; tool repository is persistent. Exact logging/trace format Not reported in extraction.
  • Safety/permission mechanism: None implemented. Safety, interpretability, and alignment during learning are flagged as open future work (p.13-14). The agent executes generated code directly in the live OS with no described sandboxing or permission gating — an UNCLEAR/under-specified risk.

Method

Pipeline per request (p.3, Fig. 2): planner -> configurator -> actor.

  • Planner reasons over the request, retrieves the agent’s capabilities (in-house tools, OS info), and emits subtasks; the DAG planner encodes subtask dependencies so independent subtasks can run in parallel via subagents (p.3, Appendix A).
  • Configurator (Fig. 3): for each subtask, the Configuration Tracker uses dense retrieval to pull relevant tools/profiles/OS info into a prompt. If no existing tool clears a similarity threshold, the Tool Generator writes a new application-tailored Python tool (a class subclassing BaseAction with a __call__ method, e.g., AppleScript to set dark mode — Table 13).
  • Actor: executor produces an executable action and runs it in the OS runtime; critic evaluates completion using pre/post system state, returns boolean judge + error analysis + a 1-10 generality score; refiner self-corrects. Tools scoring >8 are persisted (p.5). Loop runs up to 3 attempts per subtask.
  • Self-directed learning (p.6, section 4.2): FRIDAY self-instructs a curriculum (e.g., 10 Excel tasks using openpyxl), solves them, and accumulates the resulting tools (8 for spreadsheets; a separate set for PowerPoint via python-pptx).

Experimental setup

  • Benchmarks: GAIA (Mialon et al. 2023b), 466 general-assistant QA tasks across 3 difficulty levels; SheetCopilot-20 (Li et al. 2023), 20 representative spreadsheet-control tasks drawn from a 221-task set; plus a single qualitative PowerPoint-slide task (Appendix G).
  • Baselines (GAIA): GPT-4, GPT-4-Turbo, AutoGPT-4 (GPT-4 backend), GPT-4 Plugins (human-selected plugins), and human performance; ablation FRIDAY (w/o learning) disables dev-set learning. SheetCopilot baselines: SheetCopilot with GPT-3.5-Turbo / Claude / GPT-4.
  • Models: GPT-4-turbo-1106 for FRIDAY on GAIA (chosen due to budget); GPT-4 for the spreadsheet experiments.
  • Metrics: GAIA success rate per level; SheetCopilot Pass@1; average time-to-answer in seconds (Table 3); per-capability score breakdown (Fig. 5b).
  • Compute/cost: not quantified beyond “limited budget” motivating model choice. Time efficiency reported: FRIDAY ~118s on level-1 vs AutoGPT-4 456s (Table 3).
  • Implementation/artifacts: project page https://os-copilot.github.io/; code namespace friday.action. Tools seeded = 4; +9 generated on GAIA dev set; +8 for spreadsheets. License Not reported in extraction.

Key results

GAIA private test set (Table 1, p.6), success rate %:

  • Level 1: FRIDAY 40.86 vs prior best GPT-4 Plugins 30.30 — VERIFIED 35% relative improvement (40.86/30.30 ~ 1.348). AutoGPT-4 = 15.05 in Table 1.
    • DISCREPANCY (flag): the abstract/intro (p.2) cites AutoGPT-4 as 14.4%, while Table 1 lists 15.05%. The intro’s “14.4%” is not reproduced in the table; treat the table value 15.05% as the reported number and the 14.4% as UNCLEAR.
  • Level 2: FRIDAY 20.13 vs AutoGPT-4 0.63, GPT-4 Plugins 9.70.
  • Level 3: FRIDAY 6.12 vs all baselines 0 (CLAIMED “previously unsolvable by any other system”). FRIDAY w/o learning also 6.12 at level 3.
  • Ablation: FRIDAY w/o learning = 36.56 / 17.61 / 6.12. Self-directed learning gain on GAIA = +4.30 pts at L1 (36.56 -> 40.86) and +2.52 pts at L2 (17.61 -> 20.13); zero change at L3.
  • Human ceiling: 93.90 / 91.80 / 87.30 (averaged over dev+test).

Self-directed learning on SheetCopilot-20 (Table 2, p.7), Pass@1:

  • FRIDAY w/o learning = 0% (VERIFIED: fails every task, defaults to pandas/matplotlib).
  • FRIDAY (after learning) = 60% (VERIFIED), surpassing SheetCopilot/GPT-4 = 55%, SheetCopilot/Claude = 45%, SheetCopilot/GPT-3.5-Turbo = 40%.
  • Achieved by self-generating and solving just 10 tasks and accumulating 8 tools; all FRIDAY tools auto-generated vs SheetCopilot’s hand-crafted/verified operations.

Apps/interface counts (VERIFIED): 4 OS interaction methods (Python, bash, API, mouse/keyboard); seeded with 4 basic tools; +9 tools auto-generated on GAIA dev; +8 tools for spreadsheets. The framework does not enumerate a fixed number of “apps controlled” — generality comes from code execution, not a per-app list.

Time (Table 3): FRIDAY ~4x faster than AutoGPT-4 on L1 (118 vs 456s) and ~3x faster than humans; slower than GPT-4 Plugins (which excludes human plugin-selection time).

Per-capability (Fig. 5b): strong on coding and file I/O; weak on web browsing (retrieval-only, no clicks) and multimodality.

Evidence quality

Mixed. Strengths: results are on the GAIA private test server (not self-reported on a leaked test set), the ablation cleanly isolates self-directed learning, and the SheetCopilot 0%->60% jump is a striking, well-controlled demonstration of tool accumulation. Caveats:

  • No statistical reporting: single runs, no variance/CIs; GAIA level-3 is only ~6%, i.e., a handful of tasks where small-count noise dominates.
  • Small benchmarks for the headline self-improvement claim: SheetCopilot-20 is 20 tasks; 60% = 12 tasks. The “surpasses a system specifically designed for spreadsheets” claim rests on a +5-point margin (60 vs 55) over a different scaffold, with different underlying assumptions — suggestive, not decisive.
  • The abstract’s “35%” relative improvement is real for level-1 but the abstract’s “AutoGPT-4 14.4%” conflicts with Table 1’s 15.05% (flagged above).
  • Critic uses an LLM to judge subtask completion without ground truth; authors themselves call this potentially inaccurate and unscalable (p.13). Per-capability numbers rely on the authors’ own annotations, which they admit differ from the server.
  • Qualitative PowerPoint result is a single task — illustrative only.
  • No fine-tuning baseline; everything is prompt-dependent and the authors warn performance may swing with the underlying LLM (p.13).

Strengths

  • Clear, reusable architectural blueprint (planner / memory-configurator / actor with critic+refiner) that maps cleanly onto later computer-use agents.
  • Code execution as the universal action interface is an elegant answer to per-app tool explosion.
  • Self-directed tool accumulation demonstrated to move spreadsheet Pass@1 from 0% to 60%, a concrete payoff for the self-improvement story.
  • Evaluated on a held-out private test server (GAIA) and reports time efficiency, not just accuracy.
  • Full prompts and a generated-tool example included, aiding reproducibility of the scaffold.

Weaknesses and limitations

  • Author-stated: reliance on prompt engineering (brittle to LLM changes); cannot handle closed-source GUI-only apps; no multimodal/screenshot-to-action support; evaluation of subtask completion lacks ground truth; safety/interpretability/alignment unaddressed (p.13-14).
  • Inferred: web is retrieval-only (no clicks/forms), capping GAIA web tasks; small evaluation sets and single runs with no variance; level-3 success is near-floor; abstract vs Table 1 numeric inconsistency for AutoGPT-4; LLM-as-critic scoring may be unreliable; executes generated code in the live OS with no described sandbox/permission layer (operational safety risk).

Relationship to prior work

  • Closest skill-accumulation analogue: Voyager (Wang et al. 2023a) — self-directed learning + lifelong tool library, here ported from Minecraft to the OS.
  • General-purpose agent frameworks: AutoGPT (the primary contrast — shares memory + tool repository + task decomposition; FRIDAY’s edge attributed to self-criticism/refinement and self-directed tool generation) and OpenAgents (Xie et al. 2023).
  • Component lineage: ReAct, Plan-and-Solve, Tree/Graph-of-Thoughts (planning); Reflexion (reflection); Toolformer / LLMs-as-tool-makers (tool use); Self-Instruct (curriculum generation).
  • Domain-specific contrasts: SheetCopilot (spreadsheets), Mind2Web/WebArena (web), SWE/InterCode (terminal). Genuinely new = the unified OS-level universal interface + general-task self-directed tool accumulation combined into one working system; the individual components are explicitly acknowledged as not first-introduced here (p.7).
