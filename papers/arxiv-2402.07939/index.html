UFO is a GPT-Vision dual-agent framework that completes natural-language requests on Windows by reading screenshots plus the UI Automation control tree and grounding actions through pywinauto, reaching 86% success on a 50-request, 9-application benchmark.

An early, concrete blueprint for a desktop-OS computer-use agent: two-tier agent split, hybrid pixel + accessibility-tree observation, and real action grounding — design choices that recur in later Windows/desktop agents.

Motivation

Visual LLMs can interact with software UIs, but desktop OS automation was largely unexplored: at the time most GUI agents targeted smartphones or web, leaving Windows — high market share, versatile apps, and tasks needing long-horizon, cross-application planning — without a general agent. The unmet need is a system that interprets a request, perceives a Windows GUI, grounds actions without human intervention, and spans multiple apps.

Contribution (what’s new)

The genuine delta is not a model but a Windows-native harness: a two-tier agent split (cross-app orchestration vs. in-app execution) on top of control-tree grounding. Observation combines screenshots with the Windows UI Automation control tree rather than pure pixels — so the agent picks an addressable, enumerated control instead of guessing coordinates. There is no training; the novelty is the orchestration loop, the Windows-specific grounding, and the cross-app switching mechanism.

Research questions

Can an off-the-shelf VLM, given the right observation and grounding scaffolding, autonomously fulfill everyday natural-language requests across native Windows applications — including tasks that span multiple apps?

Methodology

Two GPT-Vision agents over a deterministic grounding layer. A HostAgent selects the app; an AppAgent executes per-step actions; a Control Interaction module translates choices into real pywinauto operations. Multi-app tasks work via a shared Memory module that carries extracted text/image descriptions between apps.

Method details
  1. HostAgent selects an app — Observes the desktop screenshot + active-app list, picks an app, emits a global plan and CONTINUE/FINISH status (Sec 3.2, p.5-6).
  2. AppAgent acts in-app — Each step receives three screenshots (previous-with-highlighted-control, clean, Set-of-Mark-annotated) + a filtered control list, then picks one control + one function. Status set: CONTINUE / FINISH / PENDING / SCREENSHOT / APP_SELECTION (Sec 3.3, p.6-7).
  3. Control Interaction grounds the action — pywinauto over the UI Automation backend enumerates actionable controls, annotates them with Set-of-Mark, and executes Click / SetText / GetText / Scroll plus custom Annotate / Summary (Sec 3.4, p.7-8).
  4. Cross-app switching — On APP_SELECTION, control returns to HostAgent to switch apps (Phase 2…), passing data through shared Memory (Sec 3.3, p.7).

Special designs (Sec 3.5): Interactive Mode (user follow-ups / password entry); Action Customization (register custom ops into the prompt); dual-level Control Filtering (hard filter to 10 relevant control types {Button, Edit, TabItem, Document, ListItem, MenuItem, TreeItem, ComboBox, Hyperlink, ScrollBar}; soft filter re-annotates a smaller set via the SCREENSHOT status); Plan Reflection (re-plan every step); and a Safeguard that requests confirmation before sensitive actions.

Main result

On WindowsBench, UFO reaches 86% success at 5.48 avg steps, 89.6% completion rate, 85.7% safeguard rate, more than doubling the best baseline (GPT-4 surrogate at 42%) (Tab.3, p.12). Baselines are weak by construction — GPT-3.5/GPT-4 emit step-by-step instructions executed by a human surrogate, since no other Windows agent existed to compare against.

Key findings

  • Per-app success spans 100% (Outlook, Word, File Explorer, WeChat) down to 60% on Adobe Acrobat (Tab.4, p.12): the Acrobat dip (78.7% completion) is attributed to control types unsupported by Windows UI Automation — grounding coverage, not reasoning, is the bottleneck.
  • Cross-application tasks hold up: 80% success, 83% completion, 100% safeguard, at the highest step count (avg 9.8) (Tab.4, p.13): shared Memory as a cross-app data bus genuinely enables multi-app workflows (e.g. Word + Photos → Outlook email).
  • UFO uses the fewest steps (5.48 vs 7.86–8.44 for baselines) while being most accurate (Tab.3, p.12): hybrid observation + grounding is more efficient, not just more capable.
  • UFO discovers a hidden PowerPoint shortcut (‘Remove All Presentation Notes’ via File → Info → Check for Issues → Inspect) (§4.3.1, p.13): the agent navigates non-obvious menu paths rather than only surface controls.
  • 85.7% safeguard rate shows the agent can self-classify sensitive actions and pause for confirmation (§4.2, p.12): safety is driven by the model’s own judgment over a non-exhaustive sensitive-action list (Tab.2).

Implications

A foundational template for the Windows-native branch of GUI agents: hybrid screenshot + accessibility-tree observation, dual-agent orchestration, pywinauto grounding, and an explicit safeguard. Lineage: builds on Set-of-Mark prompting (Yang et al., 2023a) and the multi-agent line (AutoGen, MetaGPT), with reflection from Reflexion / TaskWeaver; distinct from mobile agents (AppAgent, MobileAgent) and the trained-VLM approach of CogAgent. Watch how later work fixes grounding coverage (Win32 backend or dedicated GUI vision models) and adds external-knowledge retrieval for niche apps; UFO2 and OSWorld-style benchmarks are the natural next read.

Limitations

  • Capabilities are bounded by what pywinauto / Windows UI Automation expose; non-standard controls (much of Adobe Acrobat) are unsupported (author-stated, Sec 5, p.14).
  • Poor handling of unfamiliar/niche app UIs (author-stated); proposed fix is web-search-derived external knowledge.
  • Weak evaluation rigor: a small self-made 50-task benchmark, human-surrogate baselines, no ablations, no variance reporting, and “best of 3 runs” selection — the 86% should be read as an upper-ish estimate, not a robustly measured rate.
  • No training/learning; performance is entirely tied to GPT-Vision’s zero/few-shot ability, and safety relies on a non-exhaustive list plus the model’s own judgment.

Critiques and Questions

  1. With no ablations, which components actually drive the 86% — Set-of-Mark, control filtering, plan reflection, the three-screenshot input, or memory — is Unclear.
  2. Completion Rate is a self-defined “correct steps / total steps” metric whose grading is presumably manual and not described as blind; potential for optimistic scoring.
  3. “Best of 3” plus N=5 requests/app makes per-app percentages coarse (each task = 20 points) and inflates headline numbers vs. single-shot.
  4. Compute, latency, token cost, exact GPT-V version, and decoding params are all Not reported.
Why does the dual-agent design split HostAgent from AppAgent, and what makes multi-application tasks possible?
HostAgent handles cross-app orchestration (pick/switch app + global plan) while AppAgent handles in-app per-step execution — separating coarse planning from fine control. Multi-app tasks work because a shared Memory module carries extracted text/image descriptions between apps, and the AppAgent's APP_SELECTION status hands control back to HostAgent to switch (Sec 3.3, p.7).
