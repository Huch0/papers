One-sentence takeaway

OmniACT is a 9,802-example dataset/benchmark that pairs a single screenshot (macOS/Linux/Windows/web) plus a natural-language task with a gold PyAutoGUI script, and shows that even the best 2024 model (GPT-4V) reaches only ~21% of human action score, making it a hard test of static-screen visual grounding for computer-use agents.

Why this paper matters

For a harness focused on computer-use agents, OmniACT is one of the earliest benchmarks that (a) outputs an executable low-level action script (PyAutoGUI coordinate clicks/keystrokes) rather than DOM-element selection, and (b) explicitly spans native desktop apps across three OSes plus web, where no DOM is available (p2-p3, p6). That makes it a probe for pure visual grounding — the agent must predict pixel coordinates from an image, which is the core capability modern computer-use models still struggle with. CLAIMED novelty: “first-of-a-kind dataset and benchmark for assessing an agent’s capability to generate executable programs to accomplish computer tasks” beyond web automation (p1-p2). WHY-read: the action-space design and the Sequence/Action Score metric are directly reusable as evaluation primitives, and the large human-vs-model gap is a useful calibration point. Caveat for current relevance: it is a single-screen, static benchmark (one screenshot in, one script out) with no live environment, so it tests grounding + short-horizon planning but NOT interactive multi-step execution, error recovery, or environment feedback — see Evidence quality.

Problem and gap

Goal: build generalist agents that take a natural-language instruction and a screen and perform the task via mouse/keyboard (p2). CLAIMED gap (p2-p4): prior agent benchmarks are almost entirely Web (Mind2Web, WebArena, VisualWebArena, MiniWoB++, WebShop) or mobile/Android (AITW, MetaGUI, MoTIF, PixelHelp), and the web agents operate on the HTML DOM. DOM-based, text-only agents (1) cannot touch native desktop apps that have no DOM, (2) struggle with long/complex HTML, and (3) underperform on visually-grounded actions like “click the blue button in the top-right corner” (p2-p3). EVIDENCE: Table 1 (p4) compares 11 benchmarks; OmniACT is the only one marked “Desktop and Web,” supporting desktop apps, with “Continuous Scale Adaptive Evaluation” and task type “Code Generation” rather than navigation. The desktop emphasis (≈3:1 desktop:web split, p6) is the deliberately harder, DOM-free, vision-dependent regime.

Core idea

Reframe a computer task as a transition function f: T × S → A, where T is the NL task, S is a single screenshot from {Linux, Windows, MacOS, Webpage}, and A is a sequence of PyAutoGUI actions (p6, “Task Formulation”). The dataset provides parallel triples (task, screenshot, gold PyAutoGUI script). Two contributions wrap this:

  1. The dataset itself (9,802 gold, human-annotated, execution-verified scripts) plus two custom metrics (Sequence Score, Action Score) that reward correct action order and penalize coordinate/key/text errors in a spatially-aware way (Section 4, p9-p10).
  2. DetACT, a perception module that converts the screenshot into a text list of UI elements (via OCR + SAM segmentation + icon-template matching + color bucketing), then GPT-4-filters that list down to task-relevant elements to feed a downstream LLM (Section 5, p10-p12). This is the bridge that lets text-only LLMs participate at all.

Harness relevance

This is a DATASET/BENCHMARK, not a deployed agent, so the “harness” is the evaluation rig and the I/O contract it imposes:

  • Environment / workspace: NO live environment. Each example is a single static screenshot S ∈ {Linux, Windows, MacOS, Webpage} (p6). “OmniACT can be accomplished as a standalone task as it is not under a mock environment” (p5) — i.e., evaluation is offline against gold scripts, not by executing in a real OS. Coverage: 22 macOS + 8 Linux + 8 Windows native apps and 27 web apps, ≈3-4 screens each, >200 screens total (p7); 6 web task categories (Shopping, Entertainment, Service, Government, Travel, Health) (p7).
  • Observation interface: raw screenshot pixels. For text-only LLM baselines, the screen is mediated through DetACT into a textual UI-element list; for multimodal baselines the image is additionally passed to the vision encoder (p10-p12, Figure 4).
  • Action interface: a PyAutoGUI Python script. Action space A ⊇ {click, doubleClick, rightClick, moveTo, dragTo, scroll, hscroll, press, hotkey, write} (Table 2, p7). Mouse actions require pixel (x,y) coordinate arguments. CLAIMED: action space is “much larger than other benchmarks that resort to two or three interaction options” (p6). Action distribution is heavily click-dominated: Click 63.73%, Press 16.28%, Write 11.65%, Hotkey 3.00%, Move/Hover 1.85%, Scroll 1.68%, Right Click 0.77%, Double Click 0.58%, Drag 0.29%, Horizontal Scroll 0.17% (Table 2, p7).
  • Tool/API/GUI layer: PyAutoGUI as the universal action encoding; the prompt includes a typed API reference (def click(x,y), def write(text), …) (Figure 4, p11).
  • Planner/executor/verifier/search: baselines are single-shot LLM/VLM generation (DetACT perception → prompt with role + API + 5 retrieved in-context examples + filtered UI elements + task → script). No iterative planner/verifier loop, no search, no environment feedback (p12, Figure 4).
  • Evaluation harness — the load-bearing piece:
    • Sequence Score (Eq. 1, p9): if the predicted action-type sequence exactly matches gold, score = β1 + β2·(s−1) with β1=0.1, β2=1 and s = sequence length; else 0. So it rewards getting the ordered list of action types right and gives more credit for longer correct sequences. It does NOT check coordinates/text.
    • Action Score (Eq. 2-6, p9-p10): starts from SeqScore and subtracts three penalties only when the sequence already matched: Click penalty M (coordinate error, M = α·(1 − μ/(μ+L2)) where L2 is Euclidean distance from predicted point to the target bounding box, 0 inside the box; μ = inverse box-diagonal length, so larger boxes penalized more for equal L2); Key penalty K (1 if predicted key set ≠ gold key set); Write penalty Wp (1 − BLEU between gold and predicted typed text). α = SeqScore / sequence length normalizes Action Score into [0,1] per example; Table 4 reports both scores on a 0-100 scale.
    • LIMITS of the metric (analysis, not author-stated as a flaw): (1) Action Score is gated on an exact action-type sequence match — a single wrong/missing/extra action type zeroes the whole example regardless of how good the coordinates were. (2) It scores a static script against gold, so a different but valid path to the goal scores 0; there is no execution-based success check in the metric (despite “Executional Correctness” being claimed in Table 1, p4, the per-example metric is similarity/penalty-based, not run-and-check). (3) Coordinate correctness is judged against a single gold bounding box, so alternative valid targets are not credited. (4) Write penalty uses BLEU, which is weak for short strings.
  • Training harness: only light fine-tuning baselines — LLaMA-13B and Vicuna-13B fine-tuned with QLoRA (rank 64, scaling 16, 300 steps) on DetACT-description→code (p12). No large-scale training pipeline is contributed.
  • Logging/trace/reproducibility: dataset is on HuggingFace (Writer/omniact, p1); authors plan to release bounding boxes + labels as evaluation metadata (p8) and “all collected data” (p9). No per-run trace/logging harness for the agents themselves.
  • Safety/permission: none at the agent level (offline benchmark). Ethics statement (p14) covers data collection — dummy accounts, no PII, MTurk/student workers paid $25/hr.
  • Contamination / aging risk: the dataset (released early 2024) is public on HuggingFace, so post-2024 models may have seen tasks, screenshots, or gold scripts — Action Score on closed frontier models should be read with contamination in mind. Aging: it captures specific 2023-2024 app UIs at fixed coordinates; UI redesigns silently break gold coordinates, and the single-screen static format does not reflect modern interactive computer-use agents. UNCLEAR: the paper does not discuss contamination or temporal staleness beyond noting “temporal biases” and English-only data as limitations (p14).

Method

Data pipeline (Figure 2, p5; Section 3.2, p6-p9):

  1. Select 60+ applications/websites across desktop (3 OSes) and web for diversity.
  2. UI screen segmentation: desktop screens hand-annotated with a custom PyQt5 bounding-box tool; web screens parsed via JavaScript to extract visible, interactable HTML regions (buttons, dropdowns, banners, etc.).
  3. Functionality tagging: Amazon MTurk workers label each bounding box with a functional description (e.g., “find-product-search-bar”).
  4. Task creation: college students with basic Python write tasks per screen using the labeled boxes, each task given 3 NL reformulations; rephrasings kept in the same split to avoid train/test leakage.
  5. Reverse mapping + filtering: textual labels mapped back to numeric coordinates, scripts executed/syntax-checked, broken ones removed, then manual review → 9,802 gold points across >200 screens, split 7:1:2 (train/val/test).

DetACT (Section 5, p10-p12): text module (EasyOCR + SAM region classification), icon module (1,600 labeled icon templates matched to SAM regions via grayscale-resized SSIM, threshold 0.95), color module (average RGB → one of 11 named color buckets). The combined element list is filtered by prompting GPT-4 with the task to keep only relevant elements, which is concatenated into the baseline prompt.

Baselines (Section 6, p12): prompt-only LLMs with a fixed prompt template (role + PyAutoGUI API reference + 5 MiniLM-cosine-retrieved in-context examples + DetACT UI elements + rules + task); multimodal models additionally receive image pixels. Set-of-Mark prompting was deliberately NOT used because interactive elements are hard to extract from desktop images (p12).

Key results

From Table 4 (p13), AS = Action Score, SS = Sequence Score (0-100 scale):

  • GPT-4V (multimodal): best overall, AS = 17.02, SS = 38.72.
  • GPT-4 (text + DetACT): best text-only, AS = 11.60, SS = 32.75.
  • Gemini-Pro: AS = 11.46, SS = 30.98.
  • GPT-3.5-turbo-0613: AS = 7.89, SS = 22.85.
  • CodeLLaMA-34B: AS = 3.72 (strong SS = 10.09 for its size, attributed to code training).
  • Smaller open LLMs (LLaMA/Vicuna 7B-13B): AS ≈ 0.5-1.8.
  • Fine-tuning helps: LLaMA-13B SS 4.80→8.92 and AS 1.62→2.14; Vicuna-13B FT AS = 2.72 (p13).
  • Human Performance: AS = 80.14, SS = 82.23, with tiny penalties (Mp 0.12, Kp 0.36, Wp 1.61).

VERIFIED headline claim: abstract/conclusion say GPT-4 “reaches only 15% of human proficiency” (p2) — GPT-4 AS 11.60 / Human AS 80.14 = 14.5%, consistent with ~15% (the comparison is text-only GPT-4, not GPT-4V). Even the best model, GPT-4V (AS 17.02), is only ≈21% of human (17.02/80.14). EVIDENCE: text-only LLMs suffer large click/coordinate penalties because they lack pixel grounding; passing the full image (multimodal) markedly improves coordinate prediction (p13). The big human-model gap supports the CLAIM that OmniACT is hard for 2024-era models.

Evidence quality

  • Broad, sensible baseline sweep (open + closed, text + multimodal, prompt + fine-tuned) and a human ceiling — strong for a benchmark paper.
  • WEAK SPOT: the central evaluation is offline similarity-against-gold, not live execution. Table 1 (p4) claims “Executional Correctness” for OmniACT, but the per-example Action Score (Eq. 2-6) is a penalty/similarity score against one gold script, and the benchmark “is not under a mock environment” (p5) — so “executional correctness” appears to refer to the data-collection filter (scripts were verified to execute, p8) rather than to evaluation-time execution of model outputs. This is a meaningful gap between claim and metric.
  • The exact-sequence gate makes Action Score brittle: a valid alternative action ordering scores 0, and single-gold-box coordinate scoring under-credits valid alternative targets.
  • No statistical reporting (no variance/CI over examples or over the 10 human evaluators).
  • Human evaluation uses only 10 evaluators over the test set with no inter-annotator agreement reported (p13).
  • Train/test web-split table inconsistency (Table 3) is unexplained.
  • Single screenshot per example means long-horizon, multi-screen tasks (cross-app workflows highlighted as motivation, p2) are not actually evaluated end-to-end; tasks are confined “within a single screen” (p7-p8). None of these invalidate the core finding (large human-model grounding gap), but they bound how much “Action Score” should be trusted as a success proxy.

Strengths

  • First benchmark to combine native desktop (macOS/Linux/Windows) and web in one DOM-free, vision-grounded setting (Table 1, p4).
  • Executable PyAutoGUI-script output target with a rich (~10-action) action space, closer to real computer use than element-selection benchmarks.
  • Thoughtful spatially-aware metrics (Sequence Score + bounding-box-distance Click penalty with size-aware smoothing).
  • Rigorous, multi-stage human annotation with execution/syntax filtering and explicit leakage control (rephrasings kept in-split).
  • Comprehensive baseline sweep with a human ceiling, yielding a clean, large, well-motivated human-model gap.

Weaknesses and limitations

  • Author-stated (p14): GPT-4/GPT-4V hallucination and bias; reliance on costly closed models; English-only dataset; possible temporal biases.
  • Inferred: offline metric ≠ true task success; exact-sequence gating is brittle; single-screenshot scope excludes the long-horizon cross-app tasks used as motivation; single gold bounding box under-credits valid alternatives; no statistical reporting; only 10 human evaluators; no per-domain result breakdown; public dataset → contamination/aging risk for frontier models; fixed coordinates age as UIs change.

Relationship to prior work

  • Closest web benchmarks: Mind2Web [9], WebArena [62], VisualWebArena [20], MiniWoB++ [17], WebShop [56] — all DOM/web, navigation-typed (Table 1, p4). OmniACT differs by adding desktop, no DOM, and code-generation output.
  • Mobile: AITW/Android-in-the-Wild [35], MetaGUI [42], MoTIF [5], PixelHelp [23]; AutoUI [59] builds an Android agent. OmniACT targets desktop+web with PyAutoGUI.
  • Pixel-to-action lineage: Pix2Act / “From pixels to UI actions” [39] and WebAgent [13] (grounded program synthesis) are conceptual cousins.
  • Genuinely new: desktop-multi-OS coverage, PyAutoGUI-script formulation, and the Sequence/Action Score metric. Incremental: DetACT is an engineering composition of off-the-shelf parts (EasyOCR, SAM, SSIM icon matching, color bucketing, GPT-4 filtering) rather than a learned model.
