An Android GUI agent should not emit raw click (x,y) from a screenshot alone — Chain-of-Action-Thought (CoAT) routes the decision through an explicit semantic chain (screen description → action think → action description → action result), and the paired AitZ dataset makes that chain trainable for small models.

A concrete, reusable observation/reasoning interface for AitW-style mobile agents, plus a labeled dataset and an actionable data-quality critique of AitW. Foundational for the computer-use-agents track.

Motivation

Existing LLM/LMM GUI agents predict an action sequence while ignoring the semantics of intermediate screenshots and operations. Prior methods (AUTO-UI / Chain-of-Action, SeeClick) concentrate on action coordinates like click on (0.17, 0.89) and discard the logic behind the action (§1). A second gap: complex context modeling is claimed to “emerge at a large model scale,” so small models cannot acquire it by fine-tuning without high-quality CoAT-style data — and no such data existed (§1; the “emerges at scale” claim is cited to Zhang et al. 2023, not demonstrated here). Third, AitW itself has heavy label noise: for “check the settings for the spotify app,” 13 of 15 episodes never open Spotify (Fig.4).

Contribution (what’s new)

Factor the GUI policy through an interpretable intermediate chain instead of mapping screenshot → coordinate directly. A standard agent learns pi(a_t | o_t, h_{t-1}, u) (§2.1); the relation between history, observation, and action is implicit and hard to learn. CoAT inserts four textual states — Screen Description (SD), Action Think (AT), Next Action Description (AD), Action Result (AR) — so the decision factors through states the model (and a human) can read. The components are composable: route context-providers (SD, previous-AR) to model input and reasoning (AT, AD) to model output. Second contribution: AitZ, the first Android-nav dataset combining screen description + action coordinate + action description + action thinking + episode-feasibility verification (Tab.1).

Research questions

Does an explicit semantic action chain improve action prediction over coordinate-only / free-form-CoT prompting, and can a CoAT dataset let a small fine-tuned model rival a much larger GUI agent?

Methodology

Two artifacts: the CoAT schema and the AitZ dataset built to supply it. Evaluation is offline action-matching against gold AitW trajectories (not live device execution), scored by atomic action-match and episodic goal progress.

Method details
  1. Instruction sampling from AitW — AitW has 715k episodes / 30k instructions with template redundancy and train/test leakage. Subset-specific sampling: uniform per-instruction for GENERAL/GOOGLEAPPS/INSTALL (x=3/5/3), balanced sampling over site×object for WEBSHOPPING, verb-grouping + tf-idf clustering + balanced sampling for SINGLE. Yields 3,461 instructions / 7,180 episodes (§3.1).
  2. Human completion check — 10 annotators verify task completion; 5,147 successful episodes remain; one episode kept per unique instruction. (The 3,461 → final 2,504 instruction count is not explicitly reconciled — Unclear.)
  3. Semantic annotation — GPT-4V (Azure API) generates SD/AT/AD/AR; an icon-detection model marks clicked elements; SCROLL/PRESS use templates. SD is query-independent; AD gets query + gold coordinate (prompts in Fig.8).
  4. Expert verification — 3 UI-expert annotators check AD against gold actions; on mismatch they revise AD and force GPT-4V to regenerate AT/AR. Inter-annotator agreement: Not reported.
  5. Split — 1,998 train / 506 test episodes (~70/30); 13,919 / 4,724 screens (Tab.4). SINGLE (single-step) is all placed in train. Authors stress no instruction overlap → no leakage.

Action space (simplified, App.A): CLICK(y,x) in relative [0,1], SCROLL(direction), TYPE(text), PRESS(back/home/enter), STOP(task_state); AitW’s DUAL_POINT is split into CLICK and SCROLL. UI elements are exposed either as set-of-mark tags drawn on the image or as a textual element list. Agents: zero-shot prompting on Qwen-VL-Max, Gemini-1.0-Pro-Vision, GPT-4V, and CogAgent-Chat-18B (training data not public → ZS only); AUTO-UI-base (~1.2B; BLIP2 encoder, FLAN-Alpaca decoder, 224×224) fine-tuned from scratch on AitZ (≤10 epochs, lr 1e-4).

Main result

Zero-shot, CoAT beats the best of coordinate-only (CoA) / free-form (CoT) prompting on all three LMMs, with the largest gain on the strongest model — GPT-4V rises 64.1 → 73.5 acc, +9.4 over CoT (+10.7 over CoA) (Tab.2, p.3), while format hit rates stay 94.5–99.8. Fine-tuned, AUTO-UI+CoAT (~1.2B) reaches 47.69 total match / 14.51 goal progress — more than doubling its 6.59 GP baseline and rivaling CogAgent-Chat-18B (17.13 GP, 53.28 match) (Tab.5, p.6); “on-par” is most defensible on goal progress and overall match, less so per-action-type.

Key findings

  • CoAT’s benefit scales with model capability (Tab.2, p.3): zero-shot acc gain is +3.0 (Qwen 49.4→52.4) and +2.5 (Gemini 52.0→54.5) but +9.4 on GPT-4V — consistent with the “emerges at scale” framing.
  • Previous-Action-Result is the single highest-leverage component (Tab.6, p.7): adding AR-style coherence drives the STOP action-match from 60.12 to 79.17, because explicit textual outcomes link consecutive steps so the agent can judge task progress.
  • Learning to think (AT as output) without supporting input is hard (Tab.6, rows 5–7): AT-only configs underperform; performance jumps only once SD and/or previous-AR are added to the input.
  • Adding Screen Description to AUTO-UI’s input slightly hurts (Tab.6, rows 9 vs 10) (a reported negative result), attributed to its low 224×224 visual encoder being unable to ground the textual description.
  • CoAT does not help uniformly per action type (Tab.5, p.6): for CogAgent, +CoAT lifts STOP (4.76→24.60) and SCROLL-type (56.41→70.22) but CLICK-match drops (67.40→45.80).
  • AitW has substantial label noise (Fig.4, p.4): 13/15 episodes for one instruction never open the target app, and AitW’s train/val/test instruction distributions are near-identical (leakage) — motivating AitZ’s resampling.

Implications

The portable idea for harness design is the SD→AT→AD→AR factorization with context-pieces in input and reasoning in output. It generalizes Chain-of-Thought (Wei et al. 2022) with screen/action-specific structure, is positioned against the coordinate-only AUTO-UI / Chain-of-Action (Zhan & Zhang 2023), builds episodes from Android in the Wild (Rawles et al. 2023), and uses CogAgent-18B (Hong et al. 2023) as the strong reference. Whether CoAT survives live device execution rather than offline action-matching; how much of the small-model gain is the AR coherence signal vs. the dataset cleaning; whether a higher-resolution encoder would flip the SD-hurts result.

Limitations

  • Offline action-matching only — no live execution / true task-success metric.
  • The headline cross-model comparison is asymmetric: fine-tuned 1.2B AUTO-UI vs zero-shot 18B CogAgent, and among the +CoAT rows CogAgent still leads on both total match and goal progress — so “on-par” is generous framing.
  • The zero-shot prompting study uses only 50 AitW episodes; no variance, seeds, or significance testing anywhere.
  • Annotation relies on GPT-4V plus a 3-expert pass; inter-annotator agreement Not reported.
  • Author-stated: differing architecture/training data make agent comparison “less intuitive”; impact of image resolution, text-recognition, and GUI-grounding pretraining is left unmeasured.

Critiques and Questions

  1. The “on-par with 18B” claim mixes metrics: among the +CoAT rows CogAgent leads on both total match (53.28 vs 47.69) and goal progress (17.13 vs 14.51) — so the cleaner statement is “a fine-tuned 1.2B model gets within striking distance of a zero-shot 18B model,” not parity.
  2. CogAgent is evaluated zero-shot while AUTO-UI is fine-tuned on AitZ — does CoAT add value, or is the gain mostly the cleaned in-distribution training data? The ablation isolates components but not the data-cleaning vs CoAT-schema split.
  3. With only 50 episodes and no seeds, how stable are the zero-shot prompting deltas, especially the small Gemini gain where CoAT’s hit rate (96.4) is below CoT’s (97.5)?
Why does the paper put Screen Description and previous-Action-Result in the model input but Action Think and Action Description in the output?
SD and previous-AR are context that does not directly decide the current action, so they are given as input; AT and AD are the reasoning/decision the agent must learn to produce, so they are supervised as output. The ablation (Tab.6) shows learning AT as output is hard without SD/AR in the input.

Personal notes

The most portable idea for harness design: factor a GUI policy through SD → AT → AD → AR and route context-providing pieces (SD, previous AR) to input and reasoning pieces (AT, AD) to output. The negative SD-on-low-res result is a useful caution: textual screen descriptions help only if the visual encoder can ground them; on a 224×224 encoder they add noise.
