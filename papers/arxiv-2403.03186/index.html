CRADLE defines General Computer Control (GCC) and gives a training-free six-module GPT-4o framework that, as a preliminary demonstration, plays four commercial games and five desktop apps plus OSWorld purely from pixels and OS-level keyboard/mouse.

GCC is the strictest possible observation/action interface for computer-use agents — directly on-target for harness design — and the six-module decomposition is a concrete, reusable agent architecture.

Motivation

Existing foundation agents do not generalize across virtual environments because each environment ships its own hand-designed observation and action space — raw HTML/DOM, internal APIs, predefined semantic actions, accessibility element IDs — so an agent built for one cannot transfer (p.1, p.4-5). Prior screenshot-aware web/GUI/game agents still depend on those built-in hooks: web/GUI agents use Set-of-Mark or DOM element IDs; game agents (JARVIS-1, CivRealm) use internal textual state and predefined semantic actions; VPT outputs raw keyboard/mouse but needs costly action-labeled video; SIMA needs expensive human gameplay and targets ~10-second tasks (p.4-5). None operate from pure screen pixels with raw keyboard/mouse across both complex games and office software.

Contribution (what’s new)

Two genuine deltas. First, the GCC setting itself — vision-only observation, OS-level keyboard+mouse action, no internal APIs/DOM/accessibility tree — as a single standardized interface for any software. Second, the mechanism that makes it operable without APIs: the LMM writes Python “skill” functions that wrap an io_env class (e.g. key_hold('tab'), place_residential_zone(x1,y1,x2,y2)), bridging semantic intent to low-level control (p.5-6, Fig.5). Because there is no API to call, semantic actions must be materialized as executable code over a raw input device — and that action space models press duration and mouse-move speed, which most prior GUI agents ignore but which matter in real-time games.

Research questions

Can a single training-free LMM framework operate arbitrary commercial software — both AAA/2D/simulation games and office/creative apps — end to end, using only screenshots and raw keyboard/mouse, with no environment-specific APIs?

Methodology

CRADLE is a training-free GPT-4o orchestration loop that turns a short screen-video clip into low-level keyboard/mouse code, following a “reflect on the past, summarize the present, plan for the future” cycle backed by episodic and procedural memory.

Method details

Backbone: GPT-4o (gpt-4o-2024-05-13) for all LMM calls; text-embedding-ada-002 for skill embeddings; skills retrieved by similarity (p.8). Perception is augmented with external vision tools — template matching, Grounding DINO, and SAM — for grounding/localization (p.6). Per step the six modules run as:

  1. Information Gathering (perceive) — LMM + optional Grounding DINO/SAM/template matching extract visual state and OCR on-screen text from a screen-video clip of the last action's execution, sampled at 2 fps (p.5-6).
  2. Self-Reflection (verify) — LMM judges whether the last action and current task succeeded and diagnoses failures from sequential key screenshots + prior context (p.6-7).
  3. Task Inference (plan) — LMM selects the highest-priority next subtask and decides when to stop or switch tasks (p.7).
  4. Skill Curation — Retrieve relevant code skills from procedural memory; update or generate new ones (from in-game tutorials/manuals or self-exploration) (p.7-8).
  5. Action Planning + Executor — Fill skill parameters (duration / position / speed) to form the executable action; the Executor runs it as OS keyboard/mouse commands (p.8).

Memory: episodic (multimodal experience + periodic summaries) and procedural (code skills). Action primitives: key_press, key_hold, key_release, mouse_move, wheel_scroll plus mouse buttons, with press duration and mouse speed, simulated via Python in io_env (p.5). Note: the RDR2 and Stardew Valley environments are paused while waiting for the LMM, so “real-time” should be read loosely (p.8).

Main result

On OSWorld — the only auto-graded benchmark here — CRADLE reaches 7.81% overall vs GPT-4o 5.03% and GPT-4o+SoM 4.59% (All = 369), without any internal APIs (Tab.4, p.10), driven largely by the Professional split (20.41% vs GPT-4o 4.08% / +SoM 2.04% (Tab.4)). Absolute success is very low for every method, and GPT-4o+SoM still beats CRADLE on the OS split (20.83% vs 16.67%).

Key findings

  • The ablation cleanly isolates what each module buys: ReAct-like, Reflexion-like, and Voyager-like variants all fail every run (N/A) on the harder RDR2/Stardew tasks (Hitch Horse, Protect Dutch, Search for Supplies, Cultivation) — only the full CRADLE completes them (Tab.6, p.11-12): task inference + procedural memory are load-bearing, not decorative.
  • Removing vision collapses the agent (Tab.6): the Voyager-like text-only variant (image described to text) is the worst, and Claude 3 Opus fails from unreliable OCR of in-game guidance.
  • The abstract’s “maximal weekly total profit of 87%” is a single-run peak (+87.4%), not the mean (Tab.2, p.9): the mean Total Profit Rate in Dealer’s Life 2 is only 39.6 ± 27.3% — a marquee number presented in a way that overstates typical autonomous behavior.
  • The “city of a thousand people” in Cities: Skylines is reached only with human assistance (Tab.3, p.10): autonomous max population is 450 ± 224, rising to 850 ± 142 with ≤3 human unit operations (Fig.10b shows 1100+ post-fix).
  • RDR2 succeeds on navigation/follow tasks but fails real-time combat and indoor search (Fig.7, p.8-9): poor enemy/object localization drops nighttime combat (Protect Dutch) to 20% even with Grounding DINO.
  • Even familiar office apps trip the LMM on UI grounding (Fig.11, p.10-11): it forgets to click Save and confuses enabled vs disabled buttons in Chrome/Outlook, and degrades further on non-standard layouts (CapCut, Meitu, Feishu).

Implications

GCC stakes out the hardest harness setting and shows it is operable end-to-end, even if absolute competence is weak — the value is in the setting and architecture, not a leaderboard number. The code-as-skill io_env executor extends the Voyager skill-curation / skill-embedding lineage to pure-pixel control, and the framing sits directly against API/DOM-based GUI agents (UFO, OS-Copilot, Mind2Web, WebVoyager) and the contemporaneous OSWorld benchmark; its closest relative is SIMA (screenshot-in/keyboard-mouse-out, but trained via behavior cloning on short ~10s tasks). Whether later versions add audio, training (RL/imitation), or fully-autonomous Cities: Skylines results; how subsequent scaffolds fix the localization/UI-grounding failures that cap performance here.

Limitations

  • Author-stated (p.11): poor on out-of-distribution icons/styles (e.g. Stardew’s 2D pixel art) due to LMM limits; no audio modality despite GCC including it; heavy per-step LMM calls → high cost/latency; no training (RL/imitation) yet; only a limited task set explored.
  • Inferred: low absolute success (OSWorld 7.81%; many 1/5 results); flagship results lean on human assistance or single-run peaks, risking over-claiming; human evaluation with tiny task counts and no inter-rater reliability; baselines are author-adapted “-like” approximations, not canonical implementations; per-app numbers live only in figures; two game environments are paused for latency, weakening the “precise timing” motivation.

Critiques and Questions

  1. The abstract foregrounds the most flattering numbers (87% profit = single run; thousand-person city = human-assisted; 40-min RDR2 = human-judged trajectory). The most defensible quantitative claim is the OSWorld comparison — but at 7.81% absolute, what does “operable” actually translate to in deployable terms?
  2. With no automatic success signal for games/apps, how trustworthy is the human evaluation given no reliability reporting and 5-run sample sizes?
  3. Does pausing two environments during inference undercut the GCC motivation that raw keyboard/mouse with duration/speed is needed for real-time control?
How does CRADLE issue actions to software without any built-in API, and why must the action space include press duration and mouse-move speed?
The GPT-4o backbone writes Python skill functions that wrap an io_env class, which simulates OS-level keyboard/mouse events (key_press/key_hold/key_release, mouse_move, wheel_scroll). Because there is no API to call, semantic intent must be materialized as executable low-level code; duration and mouse speed are modeled because real-time games require timed holds and paced movement that discrete clicks cannot express.

Glossary

  • General Computer Control (GCC) — the paper’s proposed unified interface.
  • io_env — the executor abstraction.
  • SoM — the GPT-4o+SoM OSWorld baseline.
  • skill — CRADLE’s unit of executable behavior.

Personal notes

The most defensible quantitative claim is the OSWorld comparison (CRADLE 7.81% vs GPT-4o 5.03% vs +SoM 4.59%, no internal APIs), driven largely by the Professional split (20.41%).
