A 6B open model (ChatGLM3-6B) turned into a browser agent via an HTML-simplification interface plus a three-stage pipeline (curriculum SFT → self-sampling DPO → rejection-sampling finetuning), reaching or beating GPT-4 across English/Chinese web-navigation benchmarks.

Foundational for the computer-use / browser-agent track: it shows a small open model can be a practically deployable web agent by co-designing the observation interface and a multi-stage training recipe, rather than scaling model size.

Motivation

Most LLM web agents perform far below usable levels in real navigation because of three coupled difficulties: HTML is verbose and complex, so models struggle to read pages accurately; there is no universal action space covering the operations needed across diverse websites; and the open-domain web makes tasks hard, and agents lack inference and self-checking, so once stuck in an erroneous loop they cannot recover. Prior approaches were either deployment-impractical (e.g. WebAgent’s 540B-scale Flan-U-PaLM) or query-inefficient (Mind2Web’s MindAct uses 10+ model calls per single web operation), and state-of-the-art models cannot reliably auto-annotate training traces.

Contribution (what’s new)

The genuine delta is a co-designed interface + training recipe that makes a 6B model a competent agent. An HTML-simplification algorithm (the “HTML Pruner”) compresses pages into a concise operable representation while preserving structure and content, paired with a self-consistent, fixed action space of 10 operations so the model emits one executable command per step instead of multi-round element selection. Training then bootstraps the model: self-sampling DPO that builds contrastive pairs from the model’s own wrong vs golden operations (teaching it from its own mistakes), with an added SFT-loss term to stabilize DPO, followed by rejection-sampling finetuning (RFT) that self-plays in sandbox environments and keeps only environment-verified successful traces. The recipe must take this form because pure SFT mimics reasoning but ignores page state and prior actions (causing hallucinated operations), and complex tasks exceed what any LLM can auto-label, forcing the hybrid human-AI data construction.

Research questions

Can a small (6B) open LLM be made into a web-navigation agent that is both capable (rivaling GPT-4-class agents) and practically deployable in real browser environments — and what interface design and training stages are required to get there?

Methodology

Web browsing is framed as sequential decision-making over a state (HTML, URL, window position) and an action set (click, type, scroll, etc.). An interaction framework parses raw pages (HTML parser + OCR) into simplified HTML, marks operable elements, and executes the model’s predicted command; the loop runs until finish or a max length.

Method details

Observation space (§3.2.1). Four indicators per step: task description, simplified HTML, current location (scroll position + page size), and past operation records. Past actions are passed explicitly to stop the agent looping on failed operations.

Action space (Tab.1). Ten operations: click(id), hover(id), select(id, option), type_string(id, text, enter), scroll_page(direction), go(direction), jump_to(url, newtab), switch_tab(id), user_input(message), finish(answer).

Data construction (§4.1, hybrid human-AI, ~10k traces).

  1. Web Recognition — Collect Chinese/English top sites (Similarweb); parse operable components; GPT-3.5-Turbo generates diverse questions and bounded answers about page structure and component functions.
  2. Simple Task Operation — Build single-step operations by rules (not model generation): assemble actionable elements into operations, then GPT-3.5-Turbo writes the matching task + intent; templates for fixed actions (scroll, jump_to), LLM-assisted for flexible ones (click, type).
  3. Complex Task Operation — ~50 Evol-Instruct-style tasks per site → ~20 manually selected; humans annotate full action traces via a browser plugin; GPT-4 acts as intent predictor using a 'global thought chain' prompt (all steps + key HTML at once) for coherent CoT. Merged with Mind2Web + MiniWoB++ training sets.

Three-stage training (§4.2, Fig.6).

  1. Step 1 — Curriculum SFT — Easy→hard: first read/operate pages (4.1.1 data), then plan/reason and decompose tasks (4.1.2 data), yielding the base agent π_SFT.
  2. Step 2 — Reinforcement Learning (self-sampling DPO) — K=20 samples per complex task; keep cases solved 1..K-1 times (skip all-right and all-wrong); pair model errors vs golden ops; train with DPO + SFT loss (β=0.15, SFT weight 0.8) → π_DPO. Contrastive set ≈13k pairs.
  3. Step 3 — Rejection-Sampling Finetuning (RFT) — Self-play in MiniWoB++ and WebArena sandboxes; keep only env-verified successful traces (MiniWoB++ ≈15k traces / 66k steps; WebArena 240 traces / 2k steps, 64 samples each) and finetune per-environment.

The SFT objective is L_SFT = -E[(x,y)~D] log π_θ(y|x); RL combines DPO and SFT losses as L_total = α·L_DPO + L_SFT.

Benchmark — AutoWebBench (§4.3). Bilingual (EN/ZH), split into in-domain and out-of-domain; 50 verified traces per split; metric is Step Success Rate (SSR), following Mind2Web’s per-step evaluation.

Main result

On simulated environments the 6B model clears GPT-4 by wide margins: AutoWebGLM scores 89.3% on MiniWoB++ and 18.2% on WebArena, vs GPT-4’s 32.1% and 14.4% (Tab.4). On offline Mind2Web it leads GPT-4 and trails only a 543B model: 59.5 average SSR vs GPT-4† 30.9, behind Html-T5-XL (543B) at 66.9 (Tab.3). On its own benchmark the gap over closed models is largest: on AutoWebBench, 64.8/58.6 (EN cross-task/cross-domain) and 65.4/61.8 (ZH) vs GPT-4’s ~36–40 (Tab.2).

Key findings

  • Complex-task data is the dominant lever; simple-task data alone barely helps (Tab.6): on MiniWoB++ adding Stage 1 (simple) over the base train set moves 44.3→48.3, while adding Stage 2 (complex) jumps to 78.9; Stage 1+2 reaches 81.7. Training on complex data alone causes basic operational errors that the simple data fixes.
  • Each training stage compounds, and RFT is decisive in-domain (Tab.6): SFT 81.7 → +DPO 80.8 → +RFT 89.3 on MiniWoB++, and 8.3 → 8.5 → 18.2 on WebArena — RFT roughly doubles WebArena success via environment self-play.
  • DPO gives a modest but consistent lift on offline tasks (Tab.6): Mind2Web 56.7→59.5 and AutoWebBench 61.8→62.7, supporting the claim that learning from its own mistakes reduces hallucinated operations.
  • Hallucination is the leading failure mode at 44% (Tab.7), followed by poor graphical recognition (28%), misinterpretation of task context (20%), and pop-up interruption (8%) — i.e. errors are dominated by reasoning/perception, motivating the proposed multimodal and self-check future work.
  • Prediction (model inference) dominates per-step latency (Tab.5): it is the largest component of execution time (~47.9% of average step time), with page loading second (~41.6%), so the agent’s bottleneck is the model, not the browser plumbing.
  • Naive DPO was unstable; adding an SFT-loss term stabilized RL training (§4.2.2) without losing the model’s natural-language and agent abilities — a practical recipe detail rather than just an architecture claim.

Implications

Co-designing a compact observation/action interface with a self-bootstrapping training pipeline lets a 6B model match GPT-4-class browser agents and run as a deployable Chrome extension, shifting the cost frontier for web agents away from raw scale. It extends the WebGPT / WebGLM lineage from QA toward operational web automation and contributes AutoWebBench, the first bilingual (EN/ZH) step-level web-navigation benchmark. Worth chasing next: multimodal (HTML + screenshot) inputs for maps, animations, and video where HTML-only falters; reasoning/self-check strategies beyond Chain-of-Thought to verify state and the effect of an intended operation; and porting the recipe to mobile, where smaller viewports and gesture actions change the action space.

Limitations

  • HTML-only observation fails on advanced/visual web apps (maps, animations, video); the authors explicitly propose multimodal input as the fix.
  • Absolute real-world performance is still low — 18.2% on WebArena, far from reliable autonomy on realistic open-web tasks.
  • RFT gains are domain-specific: improvements come from per-environment self-play in MiniWoB++/WebArena sandboxes, so generalization to arbitrary live sites is unverified, partly due to network-policy constraints on real environments.
  • Evaluation leans on step-level SSR and small test splits (50 traces per AutoWebBench split), which may overstate end-to-end task completion.
  • Robustness drops on unfamiliar sites with unusual logic, and real environments are unstable (network, pop-ups), with no self-check mechanism yet in place.

Critiques and Questions

  1. Step Success Rate rewards matching individual golden steps; how much does high SSR translate to full-task completion on Mind2Web, where a single mis-step can derail a trajectory? The headline benchmark choice may flatter the model relative to WebArena’s end-to-end success.
  2. Several baselines in Tab.2/Tab.3 are zero/few-shot while AutoWebGLM is finetuned on in-distribution data (including merged Mind2Web/MiniWoB++ train sets) — is the comparison apples-to-apples, and how much of the lead is contamination/distribution alignment vs genuine capability?
  3. MiniWoB++/WebArena numbers come from RFT finetuned per environment; a single general agent’s cross-environment number isn’t reported, leaving open how much is one model vs several specialists.
  4. Many results appear to be single runs without variance; the WebArena 8.3→8.5→18.2 progression hinges on small absolute counts, so confidence intervals would matter.
