Ferret-UI adapts the Ferret referring-and-grounding MLLM to mobile screens with an “any-resolution” 2-way sub-image split plus curated UI instruction data over 11 tasks, producing a raw-pixel model that beats GPT-4V on every elementary UI task category.

An early, foundational perception backbone for mobile/GUI agents: it fixes the low-level capabilities (OCR, icon/widget recognition, find-X, grounding) that any downstream agent needs before it can plan or act, and its anyres trick and GPT-generated data recipe are widely reused.

Motivation

General-domain MLLMs cannot reliably comprehend or act on UI screens: a UI system must both read the whole screen and focus on specific elements, mapping language to actions and locations. Prior approaches each fall short: natural-image grounding MLLMs (Ferret, Shikra, Kosmos2) are restricted to natural images and lose detail when a UI screen is resized to one low-resolution global image, while UI-targeted works either process whole screens as a single input (Pix2Struct, ILuvUI, CogAgent), support only one input box (Spotlight), or lean on GPT-4V navigation — and none covers all dimensions of UI understanding. Two measured properties make this hard: UI screens have more elongated aspect ratios (Tab.1a) and many targets (icons) occupy less than 0.1% of the screen (§3).

Contribution (what’s new)

The genuine delta is twofold. First, an “any-resolution” (anyres) split: each screen is resized to whichever of two grids (1x2 or 2x1) matches its aspect ratio, then divided into exactly 2 sub-images (portrait split horizontally, landscape vertically), each encoded separately and fed to the LLM alongside the global image (§3). The design must take this form because the failure being fixed is detail loss on tiny elements: a single resized global image cannot resolve sub-0.1% icons, so the model needs magnified sub-image features in addition to global context. Second, a UI task taxonomy and instruction dataset — template-generated elementary referring/grounding tasks plus GPT-4-generated advanced reasoning tasks — that teaches the referring + grounding + reasoning capability set from raw pixels alone, with no view hierarchy needed at inference.

Research questions

Can a single MLLM perform fine-grained UI referring, grounding, and reasoning directly from screen pixels, and does grounding-rich elementary training measurably improve higher-level UI reasoning?

Methodology

Build on Ferret (CLIP-ViT-L/14 encoder + Vicuna decoder + a spatial-aware visual sampler for region features), add the anyres split, and train on a 250K-sample instruction mixture spanning Spotlight tasks, 7 elementary tasks per platform, and 4 GPT-4-generated advanced tasks. The vision encoder stays frozen; only the decoder and projection layer update. Evaluate on a 14-task dual-platform benchmark.

Method details
  1. Collect screens — Android: a subset of RICO via Spotlight tasks (26,527 train / 3,080 test images). iPhone: a subset of Apple's AMP dataset (84,685 train / 9,410 test). Resolution breakdown in Tab.1a.
  2. Annotate elements — A pre-trained pixel-based UI detector (Screen Recognition / AMP, ref [58]) emits each element's UI type + bounding box, with on-element text from the Apple Vision Framework OCR; heuristics group detections (merge text lines, pair image+caption).
  3. Build elementary tasks — Template-based, with GPT-3.5 Turbo expanding base prompts for variants. 7 tasks per platform — referring (boxes in input): OCR, icon recognition, widget classification; grounding (boxes in output): widget listing, find text, find icon, find widget. ~40k samples per task (Tab.1b); widget listing treated as auxiliary.
  4. Build advanced tasks — Follow LLaVA: send normalized detections + prompts to GPT-4 (images NOT sent) to generate detailed description, conversation perception, conversation interaction, and function inference — iPhone-only, ~10k each, 40K valid conversations total (Tab.1b, §4.2).
  5. Architecture — Global image features + per-sub-image features + regional features (visual sampler) + text embeddings are all consumed by the LLM (Fig.2). Anyres uses only 1x2 and 2x1 grids — the split is into exactly 2 sub-images, not a larger N×M grid.
  6. Train — Same objective as Ferret; 250K-sample mixture; 8 A100 GPUs — ~1 day for Ferret-UI-base, ~3 days for Ferret-UI-anyres (§5).

Benchmark: 14 tasks = 3 Spotlight tasks + iPhone/Android versions of the 11 elementary+advanced tasks. Grounding scored by IoU>0.5 accuracy; referring by exact-match/accuracy; Spotlight by CIDEr (S2W, WiC) and F1 (TaP); advanced tasks scored by GPT-4-as-judge as prediction-score over label-score (§5.1). Advanced test sets are tiny (56 unique iPhone images, 13 Android).

Main result

On elementary-task category averages, Ferret-UI-anyres surpasses GPT-4V on all four categories — referring iPhone 82.4 vs 61.3, referring Android 82.4 vs 37.7, grounding iPhone 81.4 vs 70.3, and grounding Android 83.8 vs 4.7 (Tab.2, p.10). The base Ferret model scores 13.3 / 13.9 / 8.6 / 12.9, so domain training is what closes the gap.

Key findings

  • Elementary tasks measurably lift advanced reasoning (Tab.4a, §5.2): advanced-only training scores ~64% on both platforms; adding both elementary sets raises iPhone advanced to 73.4 and Android to 80.5 — direct support for the “elementary as substrate” hypothesis.
  • Anyres gives a small lift on elementary tasks but a large lift on iPhone advanced tasks (Tab.2, §5.1): ~+2 points on iPhone referring/grounding (80.5→82.4 Ref-i, 79.4→81.4 Grd-i), but iPhone advanced jumps ~20 points (73.4→93.9, Tab.3). Android advanced drops (80.5→70.1) because no Android advanced data is in training — a generalization regression.
  • Ferret-UI beats Spotlight on S2W and WiC despite Spotlight’s vastly larger pretraining (Tab.2, §5.1) (115.6/113.4 vs 106.7 CIDEr S2W; 140.3/142.0 vs 141.8 WiC) — Spotlight pretrains on 80M web + 2.69M mobile screenshots. But Ferret-UI falls below on taperception (TaP F1 72.9/78.4 vs 88.4).
  • The TaP shortfall is largely a label-noise artifact, not a model failure (Appendix C): human annotation agrees with GPT-4V 76% of the time but with the dataset’s taperception label only 8%, so the “ground truth” is unreliable.
  • Ferret-UI’s grounding is competitive even when handicapped (§5.4, p.15): on grounded conversation-interaction box accuracy (manual grading) it scores 91.7% vs GPT-4V’s 93.4%, despite emitting raw coordinates while GPT-4V merely selects from predefined boxes.
  • Anyres concretely fixes small-element failure modes (Fig.6, §5.3): the base model predicts neighboring text instead of the target and misclassifies small text-surrounded icons as text; both are alleviated once enlarged sub-images are added.

Implications

This is the eyes of a mobile agent, not the agent itself — the observation layer downstream planners sit on. It directly extends Ferret, borrows anyres from SPHINX, LLaVA-NeXT, Monkey, follows LLaVA’s GPT-generated instruction recipe, and sits in the lineage that leads to SeeClick, ScreenAI, and later GUI-grounding agents. Worth chasing next: whether released weights / a Ferret-UI 2 exist, and how to escape the UI-detector ceiling so the model can perceive color, design, and status-bar icons the detector misses.

Limitations

  • Author-stated: the UI detection model is a hard ceiling — Ferret-UI cannot learn anything the detector misses (colors, design, usability, status-bar time/WiFi/battery), since all training signal is bootstrapped from it (§5.4).
  • Only tap actions appear in interaction data; scroll, long-press, and text entry are future work (Appendix B).
  • The GPT-4V baseline is not apples-to-apples: only 100 sampled instances, and for grounding it is forced into Set-of-Mark candidate selection, which the authors show collapses on dense Android screens — so the headline 83.8 vs 4.7 Grd-A gap partly reflects a prompting limitation, not pure model capability.
  • Advanced-task evaluation is small and noisy: GPT-4-as-judge over tens of questions (13 unique Android images), and the judge favors verbosity (scores exceed 100); no variance or significance reported anywhere.
  • No released artifacts: no code/weights mentioned, training builds on Apple-internal AMP/detector — limiting reproducibility (Not reported for license, exact sampling ratios, GPT-4 scoring prompt).

Critiques and Questions

  1. How much of the dramatic Android-grounding gap survives if GPT-4V is allowed to emit raw coordinates rather than being forced into SoM selection? The authors’ own SoM analysis suggests a large fraction of the 4.7 is harness, not model.
  2. Anyres regresses Android advanced performance — does the gain on iPhone come partly at the cost of cross-platform generalization, and would balanced Android advanced data remove the trade-off entirely?
  3. With a detector-bootstrapped pipeline, is “self-sufficient, raw pixels only” a meaningful claim at training time, or only at inference time?
Why does Ferret-UI split each screen into exactly two sub-images, and how is the split orientation chosen?
UI screens have elongated aspect ratios and tiny target elements (icons under 0.1% of the screen), so a single resized global image loses the detail needed to resolve them. Anyres adds magnified sub-image features: the screen is matched to a 1x2 or 2x1 grid by aspect ratio, then portrait screens are split horizontally and landscape screens vertically — each sub-image encoded separately and combined with the global image features (§3, Fig.2).
