A live, runnable Android environment giving programmatic reward signals for 116 parameterized tasks across 20 real apps, so agents are scored on actual task completion across endlessly varying goals — not on matching one frozen human demonstration.

Foundational evaluation substrate for mobile / computer-use agents. Its core moves — read durable reward from OS state, and randomize task parameters so one fixed seed can’t flatter an agent — generalize well beyond Android.

Motivation

Most agent evaluation compares an agent’s per-step action to a previously collected human demonstration, which is misleading online: goals are reachable by many paths, environments are non-deterministic, and agents recover from mistakes — none of which a fixed-trajectory match rewards. The gold standard is outcome-based online evaluation, but real apps expose no built-in reward, and human / LLM judges scale poorly or are unreliable. Execution-based environments that do give ground-truth reward existed, but only on desktop (ignoring mobile, used by billions) and only for static task specs — when parameters change, the reward check breaks. No reproducible, outcome-rewarding, parameterizable environment existed for Android.

Contribution (what’s new)

The first live Android environment where every one of 116 tasks across 20 real apps carries programmatic, parameterized setup + success-check + teardown, so agents are rewarded on real task completion across a practically unbounded space of randomized goals/states. The mechanistic why: real apps emit no reward and naive parameter randomization breaks brittle UI-matching checks, so AndroidWorld instead reads durable reward from system state via adb — a check that stays valid when parameters change. This lets it measure what priors could not: outcome success on real apps, and robustness across task parameters (Tab.1 lists avg instances/task as ∞ for AndroidWorld vs 1 for OSWorld, 3.3 for WebArena). The baseline agent (M3A) and base models are off-the-shelf compositions — the value is the environment and the robustness methodology.

Research questions

Can a reproducible mobile environment provide durable, outcome-based reward across endlessly parameterized real-app tasks? How well do current LLM agents perform on it, do desktop web agents transfer to mobile, and how much does task-parameter variation move measured performance?

Methodology

Each task subclasses a TaskEval contract; parameters are drawn from a controlled seed so one task template yields endless valid variants. Reward is read primarily from system state (SQLite rows, files, settings) via adb, falling back to UI-element checks where state is not inspectable. Reproducibility is pinned (Pixel 6 / Android 13 emulator, clock fixed to 2023-10-15, fixed F-Droid app versions needing no login). MiniWoB++ is wrapped as MobileMiniWoB++ (92 web tasks in a WebView) so web tasks run through the same interface.

Method details

Task contract (TaskEval): NL goal template, generate_random_params, initialize_task (seeds device state), is_successful (returns reward, often a 0–1 score; composites give partial credit), teardown (Tab.2, §3.3). State-based checking makes tasks composable: two standalone tasks merge into a composite with partial reward (§3.5).

Observation: full-res screenshot (2400×1080×3), raw accessibility (a11y) tree, processed UIElement list; optional Set-of-Mark (SoM) annotated screenshot. No DOM (§3.2).

Action: human-like gestures (tap, long-press, swipe), text input, navigation, open-app, plus task-control (status/answer/wait) and limited high-level APIs; executed asynchronously via adb.

Baseline agent M3A: zero-shot ReAct-style action+reasoning, then a Reflexion-style self-reflection over before/after states. Multimodal (SoM) and text-only (a11y) variants. Verifier = the environment’s own reward, not the agent.

Setup: base models GPT-4 Turbo, Gemini 1.5 Pro, open Gemma 2 27B. Metric = task success rate (SR), seed 30, temperature 0. Human reference = two software engineers, one attempt each. Robustness = 20 trials/task on a subset, same-seed (fixed params) vs different-seed (varied params); plus an OS-variation test on Pixel 5 / Android 12.

Environment is lightweight: 2 GB RAM, 8 GB disk; runs on a laptop. Code released at github.com/google-research/android_world.

Main result

Best agent (M3A, GPT-4 Turbo, a11y tree) completes 30.6% of tasks vs human 80.0% (Tab.3, p.8), leaving ample headroom. The adapted desktop web agent SeeAct reaches only 15.5%, evidence that web agents transfer poorly to mobile. Multimodal SoM input does not help here (25.4% < 30.6% text-only on GPT-4). The 30.6% headline is a single seed; the 3-seed mean is 29.0% (27.6 / 26.3 / 33.2), which the paper keeps out of the main table “for consistency with literature” — so the headline is mildly optimistic. The robustness analysis shows task-parameter variation materially changes measured performance (significant for add-expense, edit-note; Fig.3 shows an agent can look “completely incapable” under one unlucky seed yet solve a task often under others).

Implications

Ports OSWorld-style execution-based reward to mobile and adds MiniWoB++-style unlimited parameterization to real apps — a combination no prior benchmark offered. The robustness analysis is itself a methodological lesson: single-seed agent leaderboard numbers can be illusory; report the multi-seed mean. The state-based-reward design and mandatory parameter variation are reusable patterns for trustworthy agent benchmarks far beyond Android, and the dynamic task generation is pitched as a hook for online RL and supervised data generation.

Limitations

App scope skewed to open-source / no-login F-Droid apps for reproducibility; trending commercial and authenticated/cloud workflows are out of scope (and open-source UIs can be harder, skewing difficulty). Headline 30.6% is a single seed; 3-seed mean is 29.0% with high intra-task variance (Fig.3), yet the single-seed figure stays in the main table. Human baseline is only two engineers, one attempt each — a thin 80.0% upper bound. ”Millions of goals / ∞ instances” is a by-construction assertion, not an enumerated or sampled count. Reward fallback to UI-element matching for non-inspectable tasks reintroduces the brittleness the paper otherwise avoids.

Critiques and Questions

  1. The agent (M3A) is incremental (ReAct + Reflexion + SoM); the contribution is the environment, so don’t over-read the absolute SR numbers.
  2. If single-seed numbers are demonstrably fragile (Fig.3), why keep 30.6% in the main table at all? A leaderboard that reports the mean ± variance would be the honest default the paper itself argues for.
  3. How much of the “real apps are hard” signal is the reproducibility constraint (open-source, no-login, possibly less-polished UIs) rather than intrinsic mobile difficulty?
  4. The UI-matching fallback reintroduces the brittleness the paper criticizes — how many of the 116 tasks rely on it, and does their reward survive parameter re-rolls?
  5. What I’d test next: compare AndroidWorld’s state checkers against OSWorld’s device/cloud-state checkers; and check the live repo for task/app counts beyond 116/20.
