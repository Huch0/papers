Motivation

LM agents applied to software engineering are typically dropped into interfaces built for other users — the Linux shell and Python interpreter, which were designed for humans, not LMs. In practice agents struggle to reliably take actions in a raw shell: they cannot easily edit a small file segment, get no feedback on invalid edits, and waste budget on inefficient exploration (chains of cd, ls, cat, or grep output they cannot interpret). Unlike humans, LMs cannot freely ignore irrelevant context — every token has a fixed memory/compute cost and distracting context harms performance. Prior non-interactive, retrieval-augmented systems on SWE-bench resolved only 3.8% of tasks.

Contribution (what’s new)

The paper introduces the agent-computer interface (ACI) as a first-class design object: a thin abstraction layer that shapes the agent’s action set, action documentation, and environment feedback specifically for an LM, with the model weights left fixed. The mechanistic argument: because a raw shell is granular, verbose, and silent on errors, an LM must compose many turns and track much state to make one edit, exhausting context and budget. The ACI must therefore consolidate high-order operations (search, view, edit) into single compact commands, return concise informative feedback after every action, and add guardrails (e.g. a linter) that block error-introducing edits before they cascade. The second contribution is SWE-agent, an open-source instantiation of this ACI for software engineering.

Research questions

Does interface design — independent of the underlying model — meaningfully change LM-agent performance on real software engineering, and which ACI design choices matter most?

Methodology

SWE-agent is an LM plus an ACI, run in a ReAct loop atop a Linux shell. The ACI has four components — search/navigation, a windowed file viewer, a single-command file editor with a linter guardrail, and context management — arrived at by manual inspection of agent trajectories plus a hyperparameter sweep.

Method details
  1. Search & navigation — Special commands find_file, search_file, search_dir return a summarized list of matches; results are capped at 50 and an over-broad query is rejected with a prompt to refine, rather than flooding context.
  2. File viewer — open shows a window of at most 100 lines with the file path, total line count, lines omitted above/below, and per-line numbers; scroll_up / scroll_down / goto move the window.
  3. File editor — edit takes start line, end line, and replacement text, replacing a multi-line range in one command and auto-redisplaying the updated window. A linter checks each edit; invalid edits (e.g. syntax errors) are discarded and the agent is asked to retry, shown the before/after snippet.
  4. Context management — Each step requires a thought + action; malformed outputs trigger a retry and all but the first error message are dropped. Observations older than the last 5 are collapsed to a single line, keeping plan/action history while shedding stale file dumps.

Evaluation is on SWE-bench (2,294 real GitHub-issue tasks across 12 Python repos; full set for main results, the 300-instance Lite split for ablations) and HumanEvalFix (short-form code debugging). The metric is % Resolved / pass@1 — all hidden unit tests pass after the generated patch is applied — plus average API cost per resolved instance, with a $4 per-instance budget. Base LMs are GPT-4 Turbo and Claude 3 Opus; weaker/smaller-context models (Llama 3, DeepSeek Coder) performed poorly in the agent setting. Baselines are the non-interactive RAG system from SWE-bench and a Shell-only interactive agent (InterCode-style).

Main result

With GPT-4 Turbo, SWE-agent resolves 12.47% (286/2,294) of the full SWE-bench test set vs 1.31% for RAG and a prior best of 3.79% (RAG + Claude 3 Opus) — a ~6.7x gain over RAG (Tab.1, p.6), at $1.59 average cost per resolved instance. The ACI is portable across models: Claude 3 Opus reaches 10.46% on the full set without any ACI re-tuning (Tab.1, p.6). On the Lite split, SWE-agent’s 18.00% beats Shell-only’s 11.00% — a 64% relative (+7pp) improvement over the same model in a raw shell (§2, Tab.1).

Key findings

  • The single-command editor is the highest-leverage component (Tab.3, p.6): removing it (forcing sed/redirection edits) drops Lite from 18.0% to 10.3% (↓7.7pp), the largest ablation effect; dropping the linter alone costs 3.0pp (15.0%).
  • A summarized search interface beats an iterative one — and a bad search tool is worse than none (Tab.3, §5.1): iterative search (show results one by one) scores 12.0% vs 15.7% for No search and 18.0% for summarized search, because agents exhaustively page through every match with next, burning budget.
  • Feedback granularity has a sweet spot (Tab.3, p.6): a 100-line file-viewer window (18.0%) beats both 30 lines (14.3%, ↓3.7) and showing the full file (12.7%, ↓5.3) — too little or too much context both hurt.
  • Editing remains hard for agents even with the linter (§5.2, p.8): 51.7% (1,185/2,294) of GPT-4 Turbo trajectories contain ≥1 failed edit; any edit eventually succeeds 90.5% of the time, but that recovery rate falls to 57.2% after a single failed edit — errors compound.
  • Most failures are reasoning, not interface, failures (Fig.8, §5.2): ~52% of unresolved Lite instances are Incorrect or Overly Specific Implementations, with cascading failed edits a further 23.4% (labels auto-assigned by GPT-4o, 87% agreement with authors).
  • Agents succeed quickly and fail slowly (§5.2, p.8): resolved runs finish at a median $1.21 / 12 steps vs a mean $2.52 / 21 steps for unresolved ones, and 93.0% of resolved instances submit before exhausting budget — implying a larger budget would add little.

Implications

ACI design becomes a tunable axis for agent performance that is orthogonal to model training, and the four design principles (simple actions, compact/efficient actions, concise informative feedback, guardrails) are reusable across digital agent domains. SWE-agent operationalizes the SWE-bench evaluation line (Jimenez et al.) into an interactive agent and seeds the broad lineage of repository-level coding agents and scaffolds that followed. Whether stronger reasoning models shrink the ACI’s relative advantage (since failures are now mostly implementation-correctness, not interface friction) and how later scaffolds tackle localization and edit-error recovery, the residual weak spots are the natural threads to chase.

Limitations

  • The ACI was hand-tuned on GPT-4 Turbo via qualitative trajectory inspection on a small dev set; portability is shown for only one other model (Claude 3 Opus) and weaker models were excluded for poor performance, so generality of the specific design is uncertain.
  • Absolute resolve rates are low (12.47%) and per-instance cost is real ($1.59 avg, $4 cap); the system resolves a small minority of tasks.
  • Evaluation is Python-only (SWE-bench’s 12 repos) for the main results; cross-language SWE generalization is untested beyond the short-form HumanEvalFix.
  • Failure-mode categorization relies on an LM (GPT-4o) labeler, validated at 87% agreement but not human-exhaustive.

Critiques and Questions

  1. The headline 12.47% is far from human-level; the contribution is the relative interface gain and the design principles, not absolute capability — easy to overstate.
  2. Ablations are on the 300-instance Lite split with single-run point estimates (Fig. 4 notes non-trivial per-instance variance); some ~2-4pp deltas may be within noise.
  3. The “increasing budget won’t help” claim is inferred from the success/fail cost split; it does not isolate whether a smarter stopping policy (not more budget) would recover the slow-failing runs.
Why does an iterative (one-result-at-a-time) search interface score worse (12.0%) than having no search tool at all (15.7%)?
Given many matches, agents tend to page through every result exhaustively with `next`, consuming their cost budget and context window before doing useful work — an interface that invites inefficient behavior is worse than no extra tool, illustrating the 'compact, efficient actions' principle.
