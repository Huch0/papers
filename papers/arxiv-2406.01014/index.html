Mobile-Agent-v2 splits a mobile-GUI operation agent into three collaborating roles — a text-only planning agent that condenses history into task progress, a decision agent backed by a focus-content memory unit, and a reflection agent that checks each operation — lifting task completion by over 30% versus the single-agent Mobile-Agent.

A foundational computer-use-agent paper showing that multi-agent decomposition, not a bigger backbone, is what unlocks long-horizon GUI navigation — directly relevant to the computer-use and general agent-methods tracks.

Motivation

Mobile device operation is a multi-step sequential task: the agent must navigate two things from a growing history — task progress (what sub-goals are already done) and focus content (task-relevant facts seen on earlier screens, e.g. a match result needed later to write sports news). In a single-agent architecture, the history is a lengthy, interleaved image-and-text sequence whose token length grows with each step, degrading navigation effectiveness (§1, Fig.1). Prior work either depends on UI/XML files (AppAgent, platform-limited) or, like Mobile-Agent, uses end-to-end GPT-4V screen recognition that still chokes on long interleaved context.

Contribution (what’s new)

The genuine delta is architectural decomposition. A separate text-only planning agent compresses the interleaved image-text history into a pure-text “task progress” summary, so the decision agent never re-reads the raw history — this must take the form of a text condensation because the failure mode is precisely the length and interleaving of the multimodal history. A memory unit, updated by the decision agent, preserves focus content that the text task-progress would otherwise drop, and a reflection agent compares the before/after screenshots of each operation to catch errors the decision agent cannot self-detect (it never sees the prior screen). The roles are complementary, not redundant: each addresses a distinct failure the single agent exhibits.

Research questions

Can a multi-agent decomposition solve the task-progress and focus-content navigation challenges that long, interleaved context makes intractable for a single-agent mobile operation assistant — and which of the three roles matters most?

Methodology

The system runs iteratively over screenshots. A visual perception module turns each screen into text+icon content with coordinates; the planning agent updates task progress from the last operation; the decision agent picks one action from a fixed operation space and updates focus-content memory; the reflection agent classifies the result and triggers recovery on failure.

Method details

Visual perception module (§3.1): three tools — text recognition (ConvNextViT-document OCR from ModelScope), icon recognition (GroundingDINO), and icon description (Qwen-VL-Int4) — produce screen content plus coordinates, Pt = VPM(St).

Models (§4.1): planning agent uses text-only GPT-4 (no screen perception needed); decision and reflection agents use GPT-4V. All via official APIs.

  1. Planning agent — Updates task progress from instruction, prior operation, prior progress, and focus content: TPt = PA(Ins, O[t-1], TP[t-1], FC[t-1]) (§3.3). Output is pure text, shortening the decision agent's context.
  2. Decision agent — Generates next action Ot = DA(Ins, TP[t-1], FC[t-1], R[t-1], St, Pt) and updates focus-content memory FCt = DA(Ins, FC[t-1], St, Pt) (§3.4). Operation space: Open app / Tap (x,y) / Swipe / Type / Home / Stop.
  3. Reflection agent — Compares pre- and post-operation screens, Rt = RA(Ins, FCt, Ot, St, Pt, S[t+1], P[t+1]) (§3.5), labeling the action erroneous / ineffective / correct.
  4. Recovery — Erroneous → page reverts to prior state; ineffective → page unchanged; both are excluded from operation history so the agent does not imitate them; correct → action and page are committed.

Evaluation (§4.2): dynamic evaluation on real devices via ADB, on Harmony OS (non-English) and Android (English). 5 system + 5 external apps per scenario, 2 basic + 2 advanced instructions each, plus 8 multi-app instructions — 88 instructions per language. A “knowledge injection” variant adds manual operation hints, but only for instructions Mobile-Agent-v2 failed unaided. Baseline is single-agent Mobile-Agent (GPT-4V, temperature 0, fixed seed). Mobile-Eval was dropped as too easy (Mobile-Agent-v2 scored 99%).

Metrics (§4.2): Success Rate (SR, full-instruction success), Completion Rate (CR, fraction of correct steps vs ground truth), Decision Accuracy (DA), Reflection Accuracy (RA).

Main result

On the harder advanced instructions in the non-English scenario, multi-agent Mobile-Agent-v2 reaches 55% success rate vs 20% for single-agent Mobile-Agent (p.7), and the abstract reports an over-30% improvement in task completion overall (abstract, p.1). In English the gain is also clear despite a stronger baseline: +27% average success rate (p.7).

Key findings

  • The planning agent is the single most impactful component (Tab.4): ablating it drops basic SR from 88.6% (full) to 59.1% and advanced SR from 61.4% to 29.5%, the largest degradation of any removal — confirming that long-sequence navigation is the core bottleneck.
  • Reflection agent and memory unit each contribute on top of planning (Tab.4): with planning+memory but no reflection, advanced SR is 45.5%; adding the reflection agent and memory unit progressively raises it to 54.5% and 61.4%.
  • Multi-app tasks benefit most from the architecture (p.8): vs Mobile-Agent, Mobile-Agent-v2 improves multi-app SR by 37.5% and CR by 44.2%, because these tasks lean hardest on retrieving history operations and focus content (the memory unit).
  • The agent scaffold is essential, not optional (Tab.3): GPT-4V used end-to-end (no agent) scores only 2.7% / 0.9% single-step SR on basic/advanced, vs 92.7% / 83.5% for GPT-4V inside the framework; Gemini-1.5-Pro (38.2/29.8) and Qwen-VL-Max (42.1/33.6) trail GPT-4V.
  • The multi-agent design removes a positional error pattern (Fig.4, §4.3.3): in Mobile-Agent, erroneous/ineffective operations cluster in the later stages of a task; Mobile-Agent-v2 shows no such pattern, evidence it handles long sequences better.
  • Manual operation-knowledge injection further boosts performance and is robust (p.7, p.8): it lifts several settings to 100% CR and even 100% SR/DA/RA on EN multi-app advanced, and authors argue injected natural-language procedures survive UI size/color changes — pointing toward mobile-app test automation.

Implications

This is a lineage paper for GUI/computer-use agents: it extends Mobile-Agent (Wang et al. 2024), which removed XML dependence via visual perception, and AppAgent (Zhang et al. 2023a), which used XML, and it is the first to bring multi-agent collaboration frameworks (CAMEL, MetaGPT, AgentVerse, AutoGen) into mobile device operation. The standout thread is automating high-quality operation-knowledge generation — if the gains from manual knowledge injection can be produced automatically, the same pattern could underpin language-driven mobile app testing. Also worth chasing: whether the planning/memory/reflection decomposition transfers to open-weight backbones, given how far Gemini and Qwen trailed GPT-4V here.

Limitations

  • Heavy reliance on closed GPT-4/GPT-4V; the framework underperforms sharply with Gemini-1.5-Pro and Qwen-VL-Max (Tab.3), so portability to open models is unproven.
  • Small, self-built evaluation: 88 instructions per language, with multi-app having only 2 per setting (e.g. 2/2 results), making those success rates statistically thin (Tab.1, Tab.2).
  • Knowledge injection is applied only to instructions the system already failed and was authored manually, so reported “+Know.” numbers reflect a favorable, human-in-the-loop setting (§4.2).
  • The custom benchmark replaces the easier Mobile-Eval and is not a standard public benchmark, limiting comparability with other agents (§4.3.1 footnote).

Critiques and Questions

  1. The headline “>30% improvement” mixes settings; reported gains range widely (e.g. +27% average SR in English, much larger on multi-app). Which aggregate does “>30%” refer to, and over which tables?
  2. With only 2 multi-app instructions per setting, the “100% / 2-of-2” multi-app results carry little statistical weight — does the multi-app conclusion hold at larger N?
  3. Reflection adds extra GPT-4V calls (two screenshots per step). The paper reports accuracy gains but no latency/cost accounting — what is the per-task overhead of the three-agent loop?
Why must the planning agent be a separate, text-only model rather than just giving the decision agent the full history?
The single-agent failure mode is that the history is a long, interleaved image-text sequence whose length grows each step and degrades navigation. The planning agent condenses that history into a short pure-text task-progress summary, so the decision agent operates on far less context — and because the summary is text, the planning agent needs no screen perception (GPT-4 text-only). Ablating it causes the largest performance drop (Tab.4).
