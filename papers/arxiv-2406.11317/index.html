GUICourse is a three-part training-data suite (GUIEnv, GUIAct, GUIChat) that turns general vision-language models into vision-only GUI agents by first fixing their OCR/grounding and then teaching GUI knowledge — yielding a 3.1B agent competitive with a 9.6B one.

A foundational data recipe for the computer-use-agent track: it isolates why off-the-shelf VLMs fail as GUI agents (perception, not just knowledge) and supplies the largest real-website single-step + multi-step navigation datasets to fix it.

Motivation

General VLMs look promising for GUI control but fail as practical agents for two distinct reasons. First, their OCR and grounding are too weak to locate web elements or designated text accurately in large, high-resolution screenshots with mixed fonts — a powerful open VLM like Qwen-VL-Chat returns wrong bounding boxes for a button like “Log in” (Fig.1). Second, they lack knowledge of GUI element functions and control actions, so they cannot translate an instruction like “accept the cookies” into the right action. Compounding this, existing GUI datasets are too simplistic/simulated, too narrow in domain, or too small to train versatile agents, especially for real-world websites.

Contribution (what’s new)

The genuine delta is a two-stage data-driven pipeline that decouples perception from knowledge: pre-train on a large OCR/grounding corpus (GUIEnv) to fix fundamental perception, then SFT on navigation (GUIAct) and conversation (GUIChat) to install GUI knowledge. The design must take this form because navigation requires accurate pixel-level grounding that the baseline VLM lacks — the ablation shows adding GUIEnv lifts both grounding and downstream navigation, so knowledge SFT alone would sit on a broken perceptual foundation. A second contribution is the unified 7-group action space and per-action scoring rules that let one agent and one evaluator span websites and smartphones with vision-only inputs and position-based actions.

Research questions

  1. Can general VLMs be converted into capable vision-only GUI agents purely via data, without bespoke architectures?
  2. Does improving OCR/grounding (perception) causally improve GUI navigation, or is GUI knowledge the bottleneck?
  3. Can a small (3.1B) agent match much larger agents on GUI tasks?

Methodology

Construct three datasets, train agents in two stages, and evaluate on held-out splits plus two external benchmarks (Mind2Web, AITW). The agent receives only the viewport screenshot (no DOM/accessibility tree) and emits actions with positions.

Method details

GUIEnv — OCR + grounding (Tab.1, §3.1). 4M URLs from C4 rendered with Playwright into 10M annotated screenshots. GUIEnv-global (10M) is a pre-training task: predict all page text + grounding + layout sequence for a full screenshot. GUIEnv-local (0.7M) is an SFT QA task: text2bbox (text → region) and bbox2text (region → text), built by cropping ~50k large screenshots to <1920×1080, dropping images with <10 elements, and sampling 10 elements each.

GUIAct — navigation knowledge (Tab.1–2, §3.2). Three partitions: (1) web-single — 67k single-step instruction–action pairs over 13k real websites (50 scenarios), auto-labeled by GPT-4V using two images per request (raw screenshot + element-index overlay), then human-verified, raising accuracy 55% → 92% (§A.3.1). (2) web-multi — 5,696 high-level multi-step instructions (avg 7.9 turns) over 121 websites in 8 scenarios, crowd-annotated via a browser-plugin recorder (→ ~44k training samples). (3) smartphone — 9,157 trajectories converted from AITW-General into the unified action space (→ 67k samples). The two web subsets are the largest real-website nav datasets with both single- and multi-step instructions (Tab.2).

GUIChat — interaction (§3.3). ~44k single-turn QA + 6k multi-turn dialogues over text-rich website screenshots with grounding boxes, GPT-4-generated across four question types; human eval on 500 pairs scored Helpfulness/Honesty/Harmlessness/Image-text-correlation at 0.954 / 0.814 / 0.958 / 0.886 (§A.3.3).

Action space & formats (§A.1, Tab.7–8). 7 groups (point, input, browse, log, submit, select-value, end), 11 used actions. Parse format JSON for Qwen-VL/Fuyu, CSV for MiniCPM-V; positions absolute for Fuyu, relative for Qwen-VL, relative-scaled-to-[0,1000) for MiniCPM-V. Per-action scoring: IoU + element-EM for click/hover, distance<0.14 for tap, F1>0.5 for input/answer, direction-match for scroll/swipe; StepSR requires all actions in a step to succeed.

  1. Stage 1 — Pre-train perception — Mix GUIEnv-global (10M) into MiniCPM-V pre-training data to lift OCR/grounding. ~2 days on 128×A100.
  2. Stage 2 — SFT knowledge + dialogue — Merge 10% GUIEnv-local + all GUIAct + all GUIChat for SFT. ~8h on 8×A100, 3 epochs. Qwen-GUI/Fuyu-GUI get SFT only (no pre-train access).
  3. Evaluate — Held-out web-single/web-multi/smartphone splits (Type EM, Cli.Acc, StepSR), then fine-tune + test on Mind2Web and AITW vs baseline VLMs.

Main result

A small MiniCPM-GUI (3.1B) reaches a mean StepSR of 57.1 across the three test sets, essentially matching Qwen-GUI (9.6B) at 57.2 (Tab.3, p.7) and far above GPT-4o-mini’s 32.0. All three trained agents beat their baseline VLMs on the external benchmarks:

Key findings

  • OCR/grounding causally drives navigation (Tab.6, p.8): scaling GUIEnv 0 → 10M lifts Text2Bbox IoU@0.2 from 9.87 → 68.02, Bbox2Text F1 from 30.49 → 64.78, and web-single StepSR from 52.84 → 70.57 — the biggest jump comes from the first 2.5M, confirming perception is the binding constraint.
  • Type EM is unaffected by GUIEnv (≈92 throughout) (Tab.6, p.8): predicting which action is position-free, so all the GUIEnv gains flow through grounding (Cli.Acc 49.8 → 74.9), isolating perception from action-naming.
  • High resolution matters a lot (Tab.3, p.7): the 1344×1344 MiniCPM-GUI raises mean StepSR 49.0 → 56.1 over the 448×448 version.
  • GUIChat helps even on navigation (Tab.3, p.7): adding the dialogue dataset nudges mean StepSR +1 (56.1 → 57.1) despite targeting conversation — though external experiments (§B.4.2) find its effect on action execution is otherwise small.
  • Scaling the LM compounds the recipe (Tab.9, §B.4.1): “MiniCPM-GUI + Larger LM” (≈Qwen-VL size) hits Mind2Web cross-task 33.2 StepSR, beating SeeClick (25.5) at similar size.
  • Half of the analyzed errors are not real failures (§4.5): of 50 web-multi errors, 13 are “false errors” (reasonable but mismatched the single golden answer), 22 action-type and 15 action-position errors — static evaluation undercounts true capability, and remaining true errors cluster on complex screenshots.

Implications

GUICourse operationalizes the lesson that SeeClick / CogAgent-style GUI grounding is the bottleneck for vision-only agents, and provides reusable data to fix it — making strong GUI agents reachable from small open VLMs rather than only from GPT-4V-class closed models. It also standardizes a cross-platform action space and a per-action evaluator the field can reuse. Whether RLHF / RL-from-interaction (authors’ stated next step) closes the gap to assistant-level agents; generalization beyond web + smartphone to desktop apps and professional software; how the “false error” rate distorts static GUI leaderboards and whether interactive eval changes rankings.

Limitations

  • Pre-training + SFT alone is acknowledged as insufficient for assistant-level agents; no RL/RLHF yet.
  • Coverage is web + smartphone only — no desktop OS or professional-software GUIs, the very settings a “versatile” agent most needs.
  • Static, single-golden-answer evaluation produces ~26% false errors on web-multi, so StepSR understates real performance and is hard to compare across action spaces.
  • GUIEnv screenshots come from C4 / Common Crawl, so personally identifying or offensive content cannot be fully guaranteed (§D); GUIChat quality also degrades on unloaded-page screenshots.
  • Smartphone data is a converted AITW-General subset, not independently collected, limiting how novel the mobile evidence is.

Critiques and Questions

  1. The headline “3.1B ≈ 9.6B” comparison mixes design axes — MiniCPM-GUI uses high resolution and a different parse/position format than Qwen-GUI, so the parity may reflect resolution and full-pipeline pre-training as much as parameter efficiency. Is a controlled same-resolution comparison available?
  2. GPT-4o-mini is evaluated on only 100 samples per set (Tab.3 note) and given no GUICourse fine-tuning, so the closed-vs-open gap is suggestive, not a fair baseline.
  3. With ~26% false errors, why not adopt interactive/simulated evaluation (used only as a small case study) as the primary metric? The static numbers may rank models by golden-answer overlap rather than capability.
  4. GUIAct labels lean heavily on GPT-4V/GPT-4-generated instructions; how much of the “GUI knowledge” learned is teacher-model bias rather than ground-truth web behavior?
Why does increasing GUIEnv data improve navigation StepSR but leave Type EM roughly flat?
GUIEnv only trains OCR/grounding (where things are), so it improves grounding-dependent metrics (Cli.Acc, StepSR) by enabling accurate element localization. Type EM measures predicting the action *name*, which is position-free and already learned from navigation SFT, so more grounding data doesn't change it (Tab.6).
