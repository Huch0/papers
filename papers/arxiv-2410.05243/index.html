UGround is a 7B LLaVA-derived visual grounding model trained on a 10M-element web-synthesized dataset that lets GUI agents act from pixels alone, beating prior grounding models by ~20% absolute and matching or beating agents that additionally read HTML/accessibility trees.

Directly central to computer-use / GUI-agent harness design: it argues for and empirically validates a pure-pixel observation + coordinate-action interface with a clean planner/grounder module split, and the grounder is a pluggable component for any MLLM planner.

Motivation

GUI agents overwhelmingly perceive via text representations (HTML, accessibility trees) and act by selecting from a candidate list rather than operating directly on pixels (p.2, §1). These crutches carry real costs: full HTML is noisy and a11y trees are widely wrong — WebAIM 2024 finds 95.9% of top-1M home pages have accessibility errors, avg 56.8 per page (p.2) — and HTML can take up to ~10x more tokens than the equivalent screenshot (cited p.2). Prior pixel-only attempts exist but are “rarely adopted in state-of-the-art solutions”; the diagnosed reason is that grounding — mapping a textual plan to precise on-screen coordinates — is not yet accurate, cross-platform, and pluggable enough (p.2). The problem is therefore: text input is a leaky, costly proxy, and the missing piece for removing it is a good grounding model.

Contribution (what’s new)

The genuine delta is a constructive result, not the critique above: a web-only synthetic grounding dataset built from the HTML↔pixel dual representation, plus a resolution-adapted LLaVA, produces a single grounder accurate and general enough that a planner/grounder-split agent runs on pixels alone and still beats agents that consume HTML/a11y trees. Why the design must take this form:

  • Pixel-only is principled, not just cheaper: the rendered screen is the information-complete channel; text is a lossy proxy. Removing it is a net gain provided grounding is solved — and the paper is a constructive proof that it can be.
  • A separate grounding module (SeeAct-V), not a monolithic agent: the environment space (1B+ sites, idiosyncratic icons) is too large for one model; grounding benefits from a dedicated, independently fine-tunable, reusable module (Appendix B, p.19).
  • Web data generalizes: webpages uniquely offer full HTML + rendering + exact element↔bbox correspondence, making them an automatic, scalable source of ⟨screenshot, referring-expression, coordinate⟩ triplets that transfer to desktop/mobile (p.4, §2.2).

Research questions

Can a pixel-only GUI agent — no HTML, no a11y tree — match or beat text-augmented agents, if given a sufficiently strong universal grounding model? And can such a grounder be trained largely from synthetic web data and still transfer across platforms?

Methodology

A planner/grounder split (SeeAct-V): per step an MLLM planner sees only the screenshot + task and emits {action, element description, value}; UGround maps the element description to (x,y) coordinates; the executor performs pixel-level mouse/keyboard actions. No HTML or a11y tree enters observation, planning, grounding, or execution.

Method details
  1. Data: Web-Hybrid — Render Common Crawl pages with Playwright at varied resolutions (~1/3 mobile-friendly), ~773K screenshots. For each element, synthesize referring expressions via a hybrid pipeline: primary descriptors from HTML attributes or MLLM (LLaVA-NeXT-13B condensed by Llama-3-8B); positional REs (absolute/relative/contextual) from rule-based neighbor analysis. Center-point coords are the target. EasyOCR gates trivial text; frequent aria-labels downsampled. Supplemented by Web-Direct (GPT-4o, 408K) + Android sets. Total 10M elements / 1.3M screenshots, ~90% from Web-Hybrid (Tab.1, p.5).
  2. Model: UGround — 7B LLaVA-NeXT (Vicuna-1.5-7b-16k LM, frozen CLIP-ViT-L-14@224px), modified AnyRes up to 36 grids (1344x1344 / 896x2016) for >1000px legibility, output un-normalized pixel coordinates as text, no low-res global thumbnail, trained with LoRA. UGround-V1 variants (2B/7B/72B) rebuild on Qwen2-VL with the same data.
  3. Agent: SeeAct-V — MLLM planner (GPT-4 / GPT-4o) emits an element description; UGround grounds it to (x,y); a Click is auto-inserted before Type. Long web pages split into 1280x1000 viewport blocks with simulated scroll.

The referring-expression taxonomy is itself a conceptual contribution: real planners emit visual, positional, and functional REs (and composites), so the synthesis pipeline deliberately manufactures all three types to make the grounder robust to whatever a planner says (p.4).

Main result

On the grounding benchmark, UGround averages 73.3 on ScreenSpot Standard vs SeeClick 53.4 (+19.9 abs) and CogAgent 47.4 (Tab.2, p.6); GPT-4o scores 18.3. The “up to 20% absolute” headline matches this gap. Under the agent setting the lead widens: UGround 81.4 vs SeeClick 52.3 (+29.1 abs) with a GPT-4o planner (Tab.3, p.6). As a full agent, vision-only SeeAct-V+UGround reaches 46.8 element accuracy on Multimodal-Mind2Web, beating the HTML-Choice SeeAct baseline (42.3) (Tab.4, p.7) despite using strictly less input.

Key findings

  • Data, not architecture, is the main driver (Tab.C.1, p.20): with the same Qwen-VL backbone and only Web-Hybrid, UGround-Qwen beats SeeClick by +10.1 abs; the model-design changes add a further +14.5. The same-backbone ablation cleanly isolates the two contributions.
  • Web-only training transfers cross-platform (p.6, p.8): UGround sees zero desktop screenshots yet leads on ScreenSpot-Desktop and OmniACT (32.8 vs DetACT 17.0), validating the central transfer hypothesis.
  • The remaining bottleneck is the planner, not grounding (Fig.4, p.9): planning errors dominate everywhere (e.g. Multimodal-Mind2Web 18.2% planner vs 8.7% grounding; ScreenSpot-Desktop 46.5% vs 9.3%). UGround is no longer the weak link.
  • Grounding scales with data but saturates early (Fig.5, p.9): accuracy rises with Web-Hybrid size, diminishing after ~100K screenshots; at just 50K screenshots (~600K elements) UGround already beats SeeClick by >10% despite SeeClick’s ~3M elements.
  • Positional REs and dynamic resolution both help (Tab.9, p.10): Web-Hybrid alone (76.9) > supplementary “Others” alone (73.4); combined 81.4. Dynamic resolution beats fixed (Tab.C.2); positional REs add +3.9 standard avg (Tab.C.4).
  • Vision-only does not win every online cell (Tab.7, p.9): on Mind2Web-Live the GPT-4o SeeAct-V success rate (19.2) trails text GPT-4o (22.1), so “beats text-input agents” is true in aggregate but not literally universal.

Implications

Validates a pure-pixel observation + coordinate-action interface and a modular planner/grounder split as a practical harness design, with a reusable, fully-automated data-synthesis recipe exploiting the web’s HTML↔pixel duality. It extends SeeAct’s planner/grounder decomposition into a fully vision-only one and supersedes SeeClick / CogAgent as the grounding model, while being distinct from Set-of-Mark prompting (no labels) and OmniParser-style detectors (one generative grounder). Threads worth chasing: monolithic end-to-end GUI MLLMs (Aguvis, ShowUI, Qwen2.5-VL grounding) on the same ScreenSpot and the UGround-V1 (Qwen2-VL 2B/7B/72B) follow-ups, partly folded in here.

Limitations

  • No variance or significance anywhere — all numbers are single runs; AndroidControl uses only 500 random test steps.
  • The efficiency motivation (latency/cost) is argued via cited figures, never measured for SeeAct-V itself — the 10x token and a11y-latency claims are not re-measured.
  • No desktop training data; long-tail idiosyncratic icons remain a grounding-error source (author-stated, p.10).
  • Vision-only forces action-space compromises: on Mind2Web-Live, API-based Select is disabled and some widgets “cannot be easily operated with only Click” (Appendix G.5); on AndroidControl coordinates are post-hoc snapped to the smallest containing element (Appendix G.3).
  • UGround is not a standalone agent — it depends entirely on an external planner.

Critiques and Questions

  1. The efficiency premise is asserted, not demonstrated — measured end-to-end latency/cost would strengthen the pixel-only case considerably.
  2. “Vision-only matches/beats text-input agents” is a controlled fixed-planner, grounding-isolated claim; operation scores are omitted on Mind2Web-Live as “orthogonal” (p.7), narrowing it from full end-to-end superiority.
  3. Center-point = correct action breaks for elements whose actionable region differs from their visual center; the AndroidControl coordinate snapping is a benchmark-specific crutch.
Why does the paper argue removing HTML/a11y input is a net gain rather than a sacrifice, and what makes that argument hold empirically?
The rendered screen is information-complete (it contains exactly what a human user needs), whereas text representations are a lossy, noisy, token-expensive proxy. So removing text is only a sacrifice if grounding can't be solved — and the controlled same-backbone ablation (+10.1 from data) plus zero-shot cross-platform transfer to desktop show the grounder is strong enough that vision-only agents match or beat text-augmented ones.
