Agent S is an MLLM-based desktop GUI agent that layers retrieval over self-evaluated, abstraction-stratified memory plus live web search onto a bounded language-level Agent-Computer Interface (ACI), nearly doubling OSWorld success over the OSWorld-agent baseline.

A foundational OSWorld-era computer-use agent and the first to port SWE-agent’s “ACI” abstraction into the desktop GUI setting. Its Manager/Worker/Self-Evaluator decomposition, narrative-vs-episodic memory split, and planning/grounding/execution error taxonomy are directly reusable harness patterns.

Motivation

Automating real desktop tasks via direct mouse/keyboard control faces three pressures (§1): domain knowledge — apps and websites change constantly, so the agent needs up-to-date knowledge it cannot hold statically in weights; long task horizons — desktop tasks are multi-step with order-sensitive, interdependent actions; and dynamic, non-uniform interfaces — the agent must parse large visual+textual state and pick the right element from a vast action space. Underneath: GUIs are built for humans (real-time perception) or scripts (APIs), but an MLLM is a third user type that acts in slow discrete steps, lacks an internal coordinate system, and cannot process per-pixel feedback (§3.3). Prior GUI agents are reported bottlenecked specifically on grounding (§3.3, citing Xie et al. 2024; Zheng et al. 2024a).

Contribution (what’s new)

The genuine delta: a GUI agent improves most not from a smarter single policy but from layering retrieval of self-evaluated, abstraction-stratified experience onto a bounded language-level action interface. Two design commitments follow mechanistically:

  • Memory is stratified by abstraction and matched to stage: high-level full-task “narrative” memory (actions stripped) feeds the planner; low-level grounded “episodic” memory feeds the executor. A single memory would either pollute planning with brittle coordinates or starve execution of step detail (§3.1.1–3.1.2).
  • Grounding is delegated to the accessibility tree, not asked of the MLLM as coordinates: elements get integer IDs, and rather than augmenting the image with tree marks (prior Set-of-Marks work), Agent S augments the tree with OCR text blocks so missing elements become referenceable (§3.3).

This is distinct from prior RAG-for-agents (RAP/AutoGuide/RADA: single-level exemplars or guidelines) and from SWE-agent’s ACI (code, not GUI dual-input). Note the abstract’s framing of “extensive experiments / SOTA” as a contribution is evidence, not novelty.

Research questions

Can a frozen MLLM control a full desktop GUI well by (1) retrieving stratified self-built experience and live web knowledge rather than fine-tuning, and (2) acting through a bounded, ID-grounded interface — and which components actually drive the gains?

Methodology

A closed-loop hierarchical architecture (Fig.3): a Manager plans by fusing web search and narrative memory into a subtask queue; sequential Workers execute each subtask using episodic memory plus in-episode reflection, emitting one grounded action per step; a Self-Evaluator summarizes completed episodes into textual “rewards” written back to memory. Backbones are frozen — “learning” is retrieval over a continually-updated memory, bootstrapped by self-supervised exploration on synthetic tasks.

Method details
  1. Manager: plan from fused experience — Receives task T_u and initial observation O_0 (annotated accessibility tree + screenshot), forms an observation-aware 'How to do X' query Q, then (a) web-searches via Perplexica and (b) retrieves abstractive full-task experience from Narrative Memory (success AND failure, actions stripped). Experience Context Fusion combines them; the Subtask Planner emits a topologically-sorted subtask queue with per-subtask context C_si. (§3.1.1)
  2. Worker: execute with episodic memory + reflection — Each subtask is run by a Worker that retrieves grounded subtask experience from Episodic Memory (keyed by query+subtask+context, only from DONE subtasks). A Trajectory Reflector gives in-episode advice to avoid repeated actions. The Action Generator emits a structured CoT (prior-action status, observation analysis, semantic action, grounded action) yielding one action. DONE ends the subtask; FAIL triggers Manager replanning. (§3.1.2)
  3. Self-Evaluator: experience as textual reward — Completed subtasks become episodic summaries; completed full tasks become narrative summaries — all without human feedback or ground truth. Framed as a hierarchical-RL-like loop where retrieval is the learning mechanism. (§3.1.3)
  4. Memory construction — Bootstrapped by self-supervised exploration on synthetic tasks (top-50 common app tasks = environment-independent; tasks generated from benchmark initial environments = environment-aware), run with only web knowledge; then continually updated during inference. (§3.2)
  5. ACI — Dual input (screenshot for salient changes; integer-tagged, OCR-augmented accessibility tree for grounding via IOU match); bounded space of ~11 primitives (click, type, scroll, hotkey, hold_and_press, drag_and_drop, save_to_buffer, switch_applications, wait, done, fail — Table 5); one action per step, compiled to Python. (§3.3, App. A.1)

Main result

On the full OSWorld test set, Agent S w/ GPT-4o reaches 20.58% vs the GPT-4o baseline 11.21% — +9.37 pts, an 83.6% relative gain and a new SOTA (Tab.1, p.8); Agent S w/ Claude-3.5 reaches 20.48%. Per category it dominates the GPT-4o baseline, but GPT-4V is a surprisingly strong baseline (24.50% on Daily), so “nearly doubling” holds only against the GPT-4o row.

Key findings

  • Web knowledge is the single most impactful experience component (Tab.2, p.9): on the 65-task testsub, removing it drops Agent S from 26.15% to 16.80% (vs w/o Episodic 18.46%, w/o Narrative 21.41%, w/o All 13.85%, baseline 10.77%). But on OS, “w/o Web Knowledge” falls below baseline (16.60 vs 33.33) — web guidance can mislead in some categories.
  • Retrieval and ACI are synergistic, not additive (Fig.6, p.9): baseline 10.77% → +Retrieval 12.31% → ACI-only 12.31% → ACI+Retrieval 20.00% → full Agent S (adds hierarchical planning) 26.15%. Retrieval barely helps without ACI but jumps once ACI is present; hierarchical planning adds the final 20.00→26.15.
  • Self-supervised exploration is the most load-bearing memory mechanism, and summaries beat raw trajectories (Fig.7, p.9): full 26.15% → w/o Continual Update 23.18% → w/o Self-Evaluator 18.46% → w/o Exploration 15.38%. Replacing summaries with full trajectories (w/o Self-Evaluator) costs ~8 pts.
  • Execution and grounding errors dominate failures (Tab.3, p.10): of failed testsub tasks, execution errors 79.59%, grounding 53.06%, planning 34.69% (overlapping). Grounding stays the practical bottleneck the ACI only partially fixes.
  • Generalization to Windows is real but uneven (Tab.4, p.10): 18.2% vs NAVI 13.3% with no adaptation, driven by Windows System (29.2→45.8) and Coding (9.1→29.2), but Agent S is worse on Web Browser (20.0→13.3) and Media&Video (25.3→19.1), and 0.0% on Office.
  • Per-app gains are concentrated, and one app regresses (Tab.8, p.16): large jumps in VS Code (30.43→52.17), Thunderbird (6.67→40.00), GIMP (0.00→23.08), and OS, but LibreOffice Calc regresses (4.26→2.13) — the OS-category headline is dominated by one app’s leap.

Implications

Establishes that retrieval-as-learning over stratified memory plus a bounded ACI is a strong, fine-tuning-free recipe for desktop control, extending the SWE-agent ACI line (Yang et al. 2024) from code to GUIs and the RAG-for-agents line (RAP, AutoGuide, RADA) with two-level, self-evaluated experience. It also recombines the broader cognitive-architecture lineage (Reflexion, ReAct, Voyager, Generative Agents; contemporaneous OS-Copilot, Synapse, Cradle) rather than introducing a new learning mechanism. Worth chasing: cost/step-count Pareto-optimality, explicitly flagged as future work; whether the grounding bottleneck is fixed by later grounding-specialized models or Agent S2; head-to-head against the contemporaneous GUI agents it names but never compares to.

Limitations

  • No variance, seeds, or significance anywhere — all numbers appear single-run at low absolute success (~20%), so a few tasks can swing a category.
  • Cost, latency, and step-count are unmeasured (authors flag this, p.10) — a major omission for practical computer-use given the slow one-action-per-step loop.
  • No head-to-head with the contemporaneous GUI agents (Synapse, OS-Copilot, Cradle) it names as closest work.
  • Heavy dependence on accessibility-tree quality limits applicability to canvas-heavy / custom-rendered apps (consistent with GIMP, Calc, and Windows web-browser weaknesses).
  • Abstract and intro miscite OSWorld as “(OpenAI, 2023)” — it is Xie et al. 2024 (minor but notable).

Critiques and Questions

  1. Self-evaluation labels success/failure with no ground truth; mislabeled successes would silently poison memory, and this is never measured.
  2. Self-supervised exploration tasks are generated from the benchmarks’ own initial environments — possible leakage between exploration and test tasks.
  3. The testsub of 65 is the authors’ own resampling (the official 39-task “small” set is called too imbalanced, footnote p.8); ablations run only on GPT-4o and only on this subset.
  4. The Trajectory Reflector does not reliably break documented grounding→repeated-action→execution-error loops (Figs.14,16).
Why does Agent S use two separate memories (narrative and episodic) instead of one?
Because the two stages need experience at different abstraction levels: the Manager plans and needs transferable strategy with concrete actions stripped (narrative memory), while the Worker executes and needs concrete grounded action sequences (episodic memory). A single memory would either pollute planning with brittle low-level coordinates or starve execution of step-level detail.

Personal notes

The narrative/episodic split and “self-evaluator turns episodes into textual rewards, retrieval is the learning step” are the transferable ideas. Watch the unmeasured step/latency cost and the accessibility-tree dependence when reusing this pattern.
