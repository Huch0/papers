DistRL is an asynchronous, decoupled (centralized-training / decentralized-collection) RL fine-tuning pipeline for on-device Android control agents, paired with an off-policy algorithm (A-RIDE) that delivers ~3× faster training and a ~20% relative success-rate gain over the prior state of the art (DigiRL).

Foundational for the computer-use / device-control agent track: it is the first deployable, scalable system for online RL fine-tuning of mobile GUI agents in the wild, and shows that asynchronous distributed RL — not just bigger base models or static SFT — is what closes the gap on dynamic real apps.

Motivation

On-device MLLM control agents must operate real, frequently-updated mobile apps, but existing fine-tuning relies on static offline datasets (AitW, AndroidControl) that fail to capture distribution shift from app updates, ads, and new UI layouts, leading to in-the-wild failures. Online RL would fix this, but prior on-device RL (DigiRL) uses fully synchronous multi-machine data collection, so fast workers idle waiting for slow ones — impractical when task durations vary by up to 100× (seconds to >10 min). Asynchronous collection in turn creates non-stationary, off-policy data and policy-lag that destabilize standard RL convergence.

Contribution (what’s new)

Two coupled deltas. (1) A decoupled host–worker architecture that fully separates GPU-side centralized policy training from CPU-side decentralized, asynchronous trajectory collection across heterogeneous Android workers — necessary because synchronous designs waste the throughput of fast workers when task durations are wildly uneven. (2) A-RIDE, an off-policy algorithm built for that asynchronous data: importance-sampling-corrected updates, Retrace(λ) value correction, and Distributed Prioritized Experience Replay (DPER) that prioritizes by TD-error + IS-ratio + entropy — necessary because asynchronous collection makes data non-stationary and off-policy, so naive on-policy methods (PPO/A2C) and even IMPALA-style V-trace are unstable or wasteful here.

Research questions

Can an online RL system fine-tune mobile control agents efficiently and stably under asynchronous, heterogeneous, distributed data collection — and does doing so beat synchronous SOTA on both training speed and final success rate on real Android tasks?

Methodology

Model the task as a finite-horizon MDP over GUI screenshot states and touch/type/navigation actions, with sparse rewards from a VLM auto-evaluator (Gemini-1.5-Pro) that scores task completion from the final screenshot, turning it into a POMDP. A central Host Learner trains on GPUs from a circular replay buffer fed by a FIFO queue; heterogeneous Workers run Android emulators/devices, execute the latest policy, and stream trajectories back asynchronously. The policy is optimized off-policy with A-RIDE.

Method details (A-RIDE: architecture + algorithm)

System (host–worker, §4). Host Learner holds a circular replay buffer and a FIFO trajectory queue; it trains on GPUs and pushes updated LoRA weights to workers via SCP-over-SSH (~100 MB transfers in <2 s, negligible vs ~120 s/update and 6–10 trajectories/worker-thread/min — Tab.2, A.3.2). Workers use environment snapshots to reset emulators and run many threads independently, absorbing variance in task duration. With two 96-vCPU machines it runs up to 32 emulators concurrently and scales near-linearly.

A-RIDE = Advantage-based Retrace Improved by Distributed Prioritized Experience Replay (§5).

  1. Two value heads — A trajectory-level estimator V_traj (MLE/binary-cross-entropy on final-step reward) labels & filters high-value trajectories; a state-value network V(s) is trained as binary classification — predicting P(Monte-Carlo return G_t > 0) rather than regressing scalar returns.
  2. Advantage — One-step estimate A(s_t,a_t) = r(s_t,a_t) + γV(s_{t+1}) − V(s_t), where r includes the MC success/failure signal plus hard penalties for repetition/invalid actions (§A.4.2).
  3. Policy loss (Eq.1) — L = −E[ρ_t A log π] − β E[H(π)] + λ E[P_invalid]: an importance-sampling ratio ρ_t for off-policy stability, entropy term for exploration, and a Gemini-judged invalid-action penalty.
  4. Retrace(λ) off-policy correction (§5.2) — Corrects V(s_t) with truncated IS ratios c_i = λ·min(1, ρ_i), exploiting λ-returns when behavior/target policies agree and clipping variance when they diverge.
  5. DPER (§5.3, A.4.3) — Trajectory priority p(τ) = w1·|δ| + w2·ρ + w3·H (each normalized to [0,1]); sampled with P(τ) ∝ p(τ)^α, α=0.5; weights chosen by grid search. High-TD-error, high-IS-ratio, high-entropy trajectories are replayed more, with priorities refreshed against the current policy.

Backbone model (A.5.1). A 1.3B T5-based encoder–decoder MLLM (AutoUI-style), decoder initialized from GPT-2 (not pretrained T5), handling vision+language inputs.

Main result

On the AitW General test set, DistRL reaches 73.2% ± 1.1 success vs 61.2% ± 2.4 for DigiRL-multi and 59.9% ± 2.1 for DigiRL-single — a ~19.6% / ~22.2% relative improvement (Tab.1, p.10), while also showing the lowest variance. It wins across all four settings:

Key findings

  • ~3× training efficiency and 2.4× faster data collection vs leading synchronous multi-machine methods (§abstract, Fig.6): by 6k s, DistRL is 30%/40% (absolute) above DigiRL multi/single; even DigiRL’s algorithm dropped into DistRL’s async framework trails DistRL by ~10% — so gains come from both the framework and A-RIDE, not the framework alone.
  • Collection throughput dominates the baseline (Fig.6c, Fig.6d): ~800 trajectories in 6k s vs DigiRL’s ~300 (multi-machine); ~7.7 trajectories/min on 192 CPUs with near-linear scaling close to the ideal no-overhead upper bound.
  • Both algorithmic components are load-bearing (Fig.7b, §6.6): removing DPER costs −8% success; ablating Retrace costs −6% and induces sharp mid-training instability — DPER drives convergence speed, Retrace drives stability.
  • Strong zero-shot-ish generalization (Fig.12, §A.6.2): trained only on General tasks, DistRL still scores 68.5% (Web Shopping), 73.5% (Install), 70.2% (Google Apps) on unseen subsets, well above DigiRL/AppAgent.
  • Lowest failure rates across categories (Fig.11, §A.6.1): on General, 4% fail-to-recover, 12% stuck-midway, 2% wrong-goal — at least ~3× lower than AutoUI/DigiRL; Web Shopping shows 3% / 7% / 4%.
  • The VLM auto-evaluator is reliable but context-sensitive (Fig.5, §6.3): using the last screenshot + last two actions, evaluator-vs-human discrepancy is <2%; feeding longer trajectories hurts accuracy (Gemini over-predicts success and hallucinates with many images).

Implications

DistRL extends the lineage from IMPALA / IMPACT distributed RL and Ray RLlib and from DigiRL’s on-device online RL, but is the first to make heterogeneous, in-the-wild asynchronous collection practical for mobile agents. It reframes the bottleneck for device-control agents from base-model quality toward online training throughput and stability. Whether the V_traj/V(s) “is-the-return-positive?” binary-classification value formulation transfers to denser-reward or longer-horizon agent settings; porting the framework beyond Android (the authors sketch iOS via XCUITest/simctl) and to open-world benchmarks; using DistRL as a backbone to RL-fine-tune larger/other MLLMs rather than a 1.3B T5.

Limitations

  • Reward and the invalid-action penalty both depend on a proprietary VLM judge (Gemini-1.5-Pro); its <2% human-agreement is measured only on General AitW, and the paper itself shows the judge is brittle to context length.
  • The backbone is a small 1.3B T5-based MLLM; it is unclear whether the efficiency/stability gains hold for larger modern MLLMs.
  • Evaluation is Android + AitW/AndroidWorld (≈400–600 train, 128 test per domain) only; cross-OS and broad app generalization is acknowledged as future work, and DistRL still early-terminates on some web-shopping tasks (case studies, A.7).
  • Headline efficiency claims rest on wall-clock curves (Fig.6) with informally-read offsets (e.g. “30%/40% higher by 6k s”); some comparisons give DigiRL ~2× tuning time, but exact compute-normalized parity is not fully reported.

Critiques and Questions

  1. The “20% relative improvement” headline blends absolute (~12 pts) and relative (~19.6%) framings; reported deltas should be read carefully against Tab.1.
  2. With the evaluator both defining the reward and grading the test, success rates risk measuring agreement with Gemini’s idiosyncrasies rather than true task completion — an independent or human-held-out test would strengthen the claims.
  3. The DPER priority weights (w1,w2,w3) and α=0.5 are grid-searched on a validation set; how sensitive are results to these, and do they transfer across task domains?
Why does DistRL need an off-policy algorithm like A-RIDE at all, rather than a standard on-policy method such as PPO?
Because data is collected asynchronously by many heterogeneous workers running slightly stale policies, the training data is non-stationary and drawn from behavior policies that differ from the current target policy. On-policy methods (PPO/A2C) assume fresh, synchronous on-policy data, so they idle waiting for collection and become unstable here; A-RIDE adds importance-sampling correction, Retrace(λ), and prioritized replay to learn stably from this off-policy stream.
