Ferret-UI 2 is a single multimodal LLM for fine-grained UI referring, grounding, and user-centered interaction across five platforms (iPhone, Android, iPad, Webpage, AppleTV), built from adaptive grid-based high-resolution encoding plus GPT-4o set-of-mark visual prompting for training-data generation.

A foundational computer-use-agent perception model: it shows how to make one MLLM understand UIs at native resolution across heterogeneous device types, and that better visually-grounded training data — not just architecture — drives the gains.

Motivation

Prior UI MLLMs are locked to one or two platform types — the original Ferret-UI handles only mobile (iPhone/Android). They also use a fixed grounding resolution (336×672 / 672×336), which distorts the diverse native resolutions and aspect ratios of iPad, Web, and (landscape) AppleTV screens. Finally, training data was generated with text-only GPT-4 prompting where bounding boxes are pure text, stripping out the visual and spatial relationships between widgets and capping the quality of the resulting supervision.

Contribution (what’s new)

Three coupled deltas. (1) Adaptive N-gridding: instead of unbounded any-resolution tiling, search for the grid (Nw, Nh) that minimizes aspect-ratio distortion times relative pixel change under a cost cap Nw + Nh ≤ N — preserving native resolution within a fixed inference budget. (2) GPT-4o with set-of-mark (SoM) visual prompting to author advanced-task data: each widget is drawn with a corner-style, color-coded numbered box on the actual screenshot, so the generator reasons over real spatial layout rather than text coordinates. This is needed because text-only prompting cannot perceive widget appearance or nesting, and plain GPT-4o has poor grounding. (3) A uniform 13-class label space and multi-platform dataset built largely from human/HTML annotations rather than model-detected boxes.

Research questions

Can one MLLM deliver fine-grained referring, grounding, and user-centered single-step interaction across structurally different platforms; how much of the gain comes from architecture vs. data quality; and does the model transfer zero-shot to unseen platforms and benchmarks?

Methodology

Build on Ferret-UI’s CLIP ViT-L/14 + Visual-Sampler + LLM architecture (tried Gemma-2B, Llama3-8B, Vicuna-13B). Collect screenshots per platform (human boxes for Apple devices, parsed HTML hierarchy for Web/WebUI, RICO for Android; OCR fills missing text). Construct elementary tasks (3 referring + 3 grounding) directly from annotations, and 3 advanced task types via GPT-4o+SoM, then train one joint model.

Method details
  1. Raw collection + filtering — Human-annotated boxes for iPhone/iPad/AppleTV; HTML-parsed boxes for Web (WebUI dataset); RICO for Android. Screen-wide OCR (conf ≥ 0.5) substitutes for text labels. Filter out-of-bound/empty boxes and screenshots with >5% non-ASCII text (no multilingual support).
  2. Uniform label space — Map heterogeneous source labels to 13 classes: Checkbox, Button, Container, Dialog, Icon, PageControl, Picture, SegmentedControl, Slider, TabBar, Text, TextField, Toggle (GPT-4 suggestion + human review).
  3. Elementary tasks (6) — Referring: OCR, widget classification, tappability. Grounding: widget listing, find-text, find-widget. Converted from annotations into conversations.
  4. Advanced tasks (3) via GPT-4o+SoM — Comprehensive description, multi-round perception QA, multi-round interaction QA. SoM draws corner-style numbered, class-colored boxes on the screenshot so GPT-4o can reference and spatially reason about widgets. Interaction QA uses user-centered intents ('confirm submission') not 'click [bbox]'.
  5. Data balancing — Distribution is highly skewed (iPad 19k, AppleTV 16k vs Web 321k). Mitigate via per-platform loss weights and generating all 3 advanced-task types for iPad/AppleTV but only 1 for others.
  6. Adaptive N-gridding — Grid 336×336. Search Nw,Nh in [1,N] minimizing Δaspect × Δpixel subject to Nw+Nh ≤ N; total grids ≤ ⌊N²/4⌋. Paper sets N=8 (max 16 grids). CLIP encodes a low-res global view + high-res local grids; features flattened into the LLM.

Key gridding objective (Algorithm 1): pick (Nw, Nh) minimizing Δaspect × Δpixel where Δpixel = |Nw·Nh − Nw0·Nh0| / (Nw0·Nh0), bounding cost while least distorting native resolution.

Main result

On the authors’ constructed benchmarks (45 = 9 subtasks × 5 platforms, averaged), Ferret-UI 2 with Llama3-8B reaches an advanced-task GPT-4o score of 89.73, beating Ferret-UI’s 45.81 by 43.92 points and GPT-4o’s 77.73 by 12.0 points (Tab.2), plus the best GUIDE next-action IoU of 55.78 vs 26.91 for Ferret-UI (Tab.2). GPT-4o is strong only when given SoM prompting on referring (87.91) but collapses on grounding (12.14 elementary, 7.06 advanced Multi-IoU).

Key findings

  • Data quality matters more than architecture (Tab.5): on the harder iPhone-v2 test set, the new dataset lifts the GPT-4o score from 85.97 to 89.7 (+3.73), whereas Ferret-UI 2’s architecture adds only +1.9 on the same data; on the easy iPhone-v1 set both contribute marginally — the improved visually-grounded data is the dominant lever.
  • Strong zero-shot generalization (Tab.3): on GUI-World, Ferret-UI 2 averages 2.948, beating GUI-Vid (2.767, the benchmark’s own model) and Ferret-UI (2.638) on iOS/Android/Web without overfitting (still below Gemini-1.5-Pro 3.295 and GPT-4o 3.619).
  • Cross-platform transfer follows resolution/aspect-ratio similarity (Tab.4): iPhone↔iPad↔Android transfer well (≈65–72), but AppleTV and Web (landscape) transfer poorly to mobile, and every other domain scores ≤~43 when tested on AppleTV — a large content/orientation domain gap.
  • Data diversity aids generalization (Tab.4): iPhone, trained on the largest set (~112k images), is the best donor platform, transferring at ≥68.1 (refer) / ≥63.1 (ground) to iPad and Android.
  • Grounding transfers to unseen data within supported platform families (Tab.7): on ScreenSpot (unseen), Ferret-UI 2 (8B) hits 54.0% avg, topping CogAgent (47.4%) and SeeClick (53.4%), strong on Mobile/Web but weak on the unseen Desktop split (e.g. 21.7% icon/widget).
  • OCR, widget listing, and comprehensive description are the hardest subtasks (Tab.6) within their task types (e.g. OCR 69.3–80.5 across platforms vs find-widget ≥86), and even Gemma-2B (6× smaller than Vicuna-13B) stays competitive across the board.

Implications

Establishes a recipe for universal UI perception: cost-bounded native-resolution encoding plus visually-grounded synthetic supervision. Extends the Ferret / Ferret-UI / Ferret-v2 referring-and-grounding lineage and connects to the broader computer-use-agent and multi-platform-benchmark line (CogAgent, SeeClick, GUI-World, GUIDE). Single-step interaction only — chaining into a full multi-step planning agent is the obvious next step; the AppleTV/desktop domain gap suggests landscape-heavy and OS-desktop data are the next frontier; SoM-prompted GPT-4o as a data engine could generalize beyond UIs.

Limitations

  • Single-step interaction only: the model responds to one screen at a time; multi-step navigation is shown only qualitatively (Appendix H/Fig.15), not benchmarked.
  • Severe platform imbalance (Web 321k vs AppleTV 16k, iPad 19k) patched with loss weights/task-count heuristics rather than balanced collection.
  • Advanced-task evaluation is GPT-4o-as-judge, and the same family (GPT-4o+SoM) generates the training data — a potential circularity and judge-bias risk.
  • English-only by construction (>5% non-ASCII screenshots discarded); no multilingual UI support.
  • Text labels come from OCR (conf ≥ 0.5) rather than ground truth on Apple platforms, so OCR errors propagate into supervision; OCR is itself the weakest subtask.
  • Weak transfer to unseen Desktop/landscape (ScreenSpot desktop, AppleTV) — “universal” holds within trained platform families, less so outside them.

Critiques and Questions

  1. The headline 89.73 is a GPT-4o-judged score on the authors’ own benchmark with GPT-4o-generated data; how much of the +12 over GPT-4o reflects genuine capability vs. distribution match to the judge?
  2. Adaptive N-gridding’s gain is small in isolation (Tab.5) — is the costlier search worth it over plain AnyRes given that data is the real driver?
  3. Numbers appear to be single runs with no variance/error bars reported; the cross-platform transfer table especially would benefit from repeats.
  4. Tappability is ”-” for AppleTV (Tab.6) and SoM/GPT-4o rows have gaps (Tab.2) — coverage is uneven across the claimed 5 platforms.
Why does Ferret-UI 2 use set-of-mark visual prompting when generating advanced-task data with GPT-4o, instead of describing bounding boxes as text?
Text-only prompting (as in Ferret-UI) hides widget appearance and spatial/nesting relationships, and GPT-4o grounds poorly from raw screenshots. Drawing corner-style, class-colored numbered boxes on the actual screenshot lets GPT-4o reliably reference and reason about each widget's location and relationships, yielding higher-quality, more spatially-aware training data — which the ablation (Tab.5) shows is the dominant source of the model's gains.
