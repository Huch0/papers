AndroidLab is a unified environment plus a reproducible 138-task benchmark for Android agents that puts text-only (XML) and multimodal (SoM) models in one shared action space, and shows that fine-tuning small open-source models on its Android Instruct dataset lifts average success rates from 4.59% to 21.50% (LLMs) and 1.93% to 13.28% (LMMs).

A foundational computer-use-agent artifact: it is one of the first frameworks to systematically compare open- and closed-source Android agents in the same action space and to ship an instruction-tuning dataset, making open-source mobile agents competitive with proprietary ones.

Motivation

Training and evaluating Android agents lacked systematic study spanning both model families. Prior mobile benchmarks were largely static — agents predicted actions from screenshots without real interaction (AITW, MetaGUI, PixelHelp). The interactive ones still fell short: almost all recent benchmarks tested only closed-source models with prompt-based tweaks, so behavior could not be analyzed, insights integrated, or RL run. Evaluation metrics relied on fixed operation sequences or single device states, which restrict valid task paths and the set of expressible tasks. And there was no unified benchmark comparing open- vs closed-source models across modalities.

Contribution (what’s new)

A single action space shared by text-only and multimodal models: XML mode (compressed UI tree) for LLMs and SoM mode (set-of-mark numbered screenshot) for LMMs are aligned so the same elements and actions are selectable in both — enabling fair cross-modality comparison. Sub-goal-based task verification: each task is decomposed into required page states checked by UI-tree element matching, rather than matching one canonical operation sequence or final device state; this admits many valid paths and near-arbitrary task types. The design must take this form because sequence/state-matching metrics fail to credit alternative-but-correct trajectories and cannot express query tasks. AndroidLab also ships the Android Instruct instruction-tuning dataset, closing the open-source gap.

Research questions

  • Can open- and closed-source LLMs and LMMs be benchmarked fairly under one action space and one reproducible Android environment?
  • How far behind are open-source models, and can fine-tuning on a purpose-built dataset close the gap to proprietary models?
  • Do reasoning frameworks (ReAct, SeeAct) actually help Android agents, and at what efficiency cost?

Methodology

A standard operation environment (action space + four modes) drives a deterministic 138-task benchmark over preloaded Android virtual devices; an annotation pipeline builds the Android Instruct dataset used to fine-tune six small open-source models.

Method details

Action space. Built on AppAgent and AndroidEnv: four core operations — Tap, Swipe, Type, Long Press — plus Home/Back shortcuts and a Finish action (§3.1).

Four operation modes. Two base modes: XML mode (text-only) and SoM mode (multimodal); selections are aligned so both modes act on identical elements. Two add-on frameworks: ReAct and SeeAct (§3.1, App.B).

Benchmark. 138 tasks across 9 offline apps (Settings, Bluecoins, PiMusic, Clock, Maps.me, Zoom, Contacts, Calendar, Cantook): 93 Operation Tasks and 45 Query Tasks (§3.2). Each task is a 4-tuple Task(E, I, F, M) — environment (pre-packaged AVD image), instruction, framework, backbone model. Reproducibility: ADB fixes evaluation time and geolocation, apps run offline with preloaded records, query answers are pre-computed for unique ground truth (§3.2.2).

Metrics (§3.2.3, higher = better). Success Rate (SR); Sub-Goal SR (Sub-SR); Reversed Redundancy Ratio (RRR); Reasonable Operation Ratio (ROR).

Android Instruct dataset (§4). Three steps: (1) task derivation/expansion seeded from academic datasets + hand-written instructions, expanded by LLMs; (2) self-exploration (a reward model replaced manual selection after 500 traces); (3) manual annotation with cross-verification, using a custom ADB + Accessibility-Service tool that captures screenshot + page XML before each action. Yields 10.5k traces / 94.3k steps overall; the 726 traces / 6208 steps from benchmark apps are used for training, in aligned XML and SoM formats.

Training (§4.3). Backbones: Llama-3.1-8B-Instruct, GLM-4-9B-Chat, Qwen2-7B-Instruct (LLMs); Llama-3.2-11B-Vision-Instruct, Qwen2-VL-7B-Instruct, CogVLM2 (LMMs). Base versions fine-tuned where available (Qwen2-VL has no open base); batch size 32, max length 4096, 5 epochs, LR 1e-5. Eval: tasks start inside the target app, max 25 steps, 3s per step, greedy decoding (§5.1).

Main result

The benchmark is hard: the leading closed-source models top out near 31%. In XML mode GPT-4-1106-Preview leads at 31.16% SR (38.21% Sub-SR); in SoM mode GPT-4o leads at 31.16% SR (35.02% Sub-SR) (Tab.1, p.9). Fine-tuning on Android Instruct raises average SR from 4.59% to 21.50% for LLMs and from 1.93% to 13.28% for LMMs (Tab.1, §5.3), bringing the best tuned open-source LLMs near GPT-4o and the best tuned LMMs past Gemini-1.5-Pro.

Key findings

  • Fine-tuning closes most of the open-vs-closed gap (Tab.1, §5.3): Qwen2-7B’s SR jumps 4.35→23.91 (XML), Qwen2-VL-7B 3.62→18.12 (SoM), and CogVLM2’s ROR leaps 17.97→85.58 — tuned models’ effective-action rate (ROR) surpasses most closed-source models.
  • ReAct helps mainly in XML mode (Tab.2, §5.3): GPT-4o XML SR rises 25.36→33.33 and Gemini-1.5-Pro 18.84→31.16 with ReAct; in SoM ReAct barely moves GPT-4o (31.16→31.88) and SeeAct is inconsistent, attributed to weaker multimodal reasoning.
  • Reasoning frameworks cost tokens (Tab.3, §5.3): average generated tokens grow from 4.96 (post-SFT) and 23.56 (base XML/SoM) to 67.89 (ReAct) and 129.12 (SeeAct), hurting efficiency for marginal or no SR gain.
  • Screen size matters (Fig.6, §5.3): SoM agents perform best on common phone sizes (Pixel 7 Pro / 8 Pro) and degrade on smaller (Pixel 3a, more scrolling) and tablet-like (Pixel Fold, landscape) screens.
  • RRR (efficiency) is decoupled from SR (Tab.1, §5.3): Claude-3.5-Sonnet has the best SoM RRR (113.41) despite lower SR than GPT-4o, and fine-tuning consistently lifts both RRR and ROR.
  • Launch-failure is a real confound (§5.1): agents often failed simply because they could not open the specified app, so evaluation starts inside the target app to isolate task-execution ability.

Implications

AndroidLab makes open-source mobile agents a viable research target: a shared action space plus an instruction-tuning dataset turns ~2-5% baseline models into ~20% agents. It extends the AndroidEnv / B-MOCA / AndroidWorld line of interactive mobile benchmarks and the AppAgent / CogAgent / SeeAct agent lineage, and its SoM tasks are reused as the VAB-Mobile component of VisualAgentBench. Whether the sub-goal UI-tree matcher generalizes to apps with dynamic or non-standard UI trees and whether RL (now feasible on these open models) beats SFT are natural next threads, as is reducing redundancy to improve real-world deployability.

Limitations

  • Scope is 9 offline, login-free, no-internet apps chosen for reproducibility — excludes the network-dependent, account-gated, dynamic apps where real users spend most time.
  • All results are single-run greedy decoding with a 25-step cap; no variance or error bars are reported, so small SR differences may not be significant.
  • RRR is unreported whenever SR < 5 and can exceed 100 because human paths are not optimal, making it a noisy efficiency proxy for weak models.
  • Tasks start pre-launched inside the target app, removing the app-discovery/launch step that real agents must handle.
  • Query-task grading relies on an advanced LLM judge, which introduces an unquantified evaluator-reliability risk.

Critiques and Questions

  1. The headline “21.50% / 13.28%” averages are computed only over the open-source models fine-tuned here; with single-run numbers and a 25-step ceiling, how much of the gain is robust vs. variance or distribution-match between the 6208 training steps and the same-app eval tasks (potential overlap/contamination)?
  2. Is the sub-goal UI-tree matcher robust to UI changes across app versions, or does it silently age as the AVD images drift — and how brittle is it on apps with dynamic components (the very case that broke ADB XML capture)?
  3. SeeAct’s failure is blamed on multimodal reasoning limits, but it was the most token-heavy mode (129.12 tokens); could the two-round protocol itself, not modality, be the culprit?
Why does AndroidLab align the XML and SoM modes onto the same elements and action space?
So text-only (LLM) and multimodal (LMM) models can be compared fairly: both modes select the identical underlying UI elements and emit the identical action calls, isolating the modality/model effect rather than differences in the interface. The SoM serial numbers map to the same elements as the compressed XML list.
