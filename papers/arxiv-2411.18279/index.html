Motivation

Traditional GUI automation relied on script-based and rule-based methods (e.g. RPA) that are rigid, narrow, and need constant manual updates when layouts or workflows change. The arrival of multimodal LLMs enabled flexible, language-driven GUI control, triggering a surge of frameworks, datasets, models, and benchmarks — but no systematic review existed, leaving the field fragmented with no shared vocabulary, taxonomy, or roadmap. The survey also argues GUI agents beat pure API-based agents because APIs are often not exposed or accessible, whereas GUIs are a universal, non-intrusive control surface for nearly any application.

Contribution (what’s new)

The genuine delta is organizational, not empirical: it imposes a single end-to-end pipeline / taxonomy that decomposes every GUI agent into five reusable components and then walks the full build path — frameworks → data → models → evaluation → applications → challenges. The framing matters because the field’s prior surveys were either about LLM agents in general or about classical GUI automation; neither gave practitioners a component-level blueprint specific to LLM-brained GUI agents. The survey deliberately doubles as a constructive “cookbook” rather than a pure literature catalog.

Research questions

The survey is structured around eight RQs: (RQ1) the historical trajectory of GUI agents; (RQ2) essential components and advanced techniques; (RQ3) the principal frameworks and their characteristics; (RQ4) existing datasets and how to collect more; (RQ5) how to train purpose-built LAMs; (RQ6) evaluation metrics and benchmarks; (RQ7) real-world applications; and (RQ8) challenges, limitations, and future directions.

Methodology

A literature survey: it collects and categorizes research and industry systems across web, mobile, and desktop platforms, then narrates each category with representative exemplars and comparison tables. The companion paper list is maintained on GitHub and a searchable site.

The 5-component reference architecture (§5.1)

The survey’s spine is a workflow that runs from user request to task completion:

  1. Operating Environment — Defines the platform (mobile / web / desktop OS). The agent perceives state via screenshots, widget/accessibility trees, or UI element properties, and reads environment feedback after each action.
  2. Prompt Engineering — Assembles instruction + processed visual data + UI layout/properties + context into a structured prompt aligned to the current GUI state.
  3. Model Inference — A general-purpose LLM or fine-tuned LAM produces plans, actions, and complementary outputs (reasoning/insights).
  4. Action Execution — An executor translates inferred intent into concrete UI operations (mouse/keyboard/touch/gesture/voice), native API calls, or AI tools.
  5. Memory — Short-term memory tracks prior steps, progress, and states for coherence; long-term/external memory enables continuous learning and knowledge access.

Actions span three executor families (Tab.5): UI Operations (click, type, swipe, gesture, voice), Native API Calls (shell, application/system/web APIs), and AI Tools.

Key findings

The survey’s value is its organizing insights — the cross-paper trends it distills per section. Per-system metrics are largely Not reported in summary form; the survey instead reports directional trends.

  • Six advanced enhancements are layered on top of the base five components (§5.7): CV-based GUI grounding (e.g. OmniParser), multi-agent frameworks, self-reflection, self-evolution, and reinforcement learning — and these are composable, not mutually exclusive (e.g. self-reflection inside a multi-agent loop).
  • Frameworks are trending toward multi-agent role specialization, multimodal (screenshot + text) inputs, expanded action sets (UI ops + API + AI tools), and world-model / search-based decision-making (MCTS, best-first) (§6.5), all converging on cross-platform, increasingly pure-vision agents (e.g. AGUVIS) that act from screenshots alone.
  • Data is the bottleneck: high-quality trajectory data needs heavy human annotation (§7.6); the field is moving to large-scale, cross-platform datasets and AI-driven automated collection pipelines to scale beyond manual effort.
  • Purpose-built LAMs trend small (≈1B–7B params) for on-device, low-latency, privacy-preserving inference (§8.7), lean on visual grounding to reduce reliance on structured UI metadata, use RL (e.g. DigiRL) to bridge static training to dynamic environments, and adopt unified function-calling plus inference-time reasoning (“o1-style”, UI-R1/GUI-R1).
  • Evaluation spans step/task success-rate metrics and is shifting from synthetic to “live”, interactive environments (simulators, Docker, real apps), toward cross-platform and multi-turn/human-in-the-loop benchmarks, with growing emphasis on safety, privacy, and compliance (§9); building realistic tasks and evaluators remains a scalability pain point.
  • GUI grounding is treated as a first-class capability with its own benchmarks (§5.7.1): ScreenSpot covers 5 platforms with 600+ screenshots and 1,200 instructions; ScreenSpot-Pro extends to 1,581 expert-annotated high-resolution professional tasks (CAD, dev tools, office).
  • Two application clusters dominate: GUI testing (general testing, text-input generation, bug replay, verification) and virtual assistants (research, open-source, production) (§10), the latter exemplified by Power Automate and Microsoft Copilot.

Implications

This is the lineage map for the computer-use field. It positions and connects SeeAct (web), AppAgent (mobile), UFO (Windows), WebVoyager, OmniParser, AGUVIS, DigiRL, xLAM, and benchmarks like Mind2Web and ScreenSpot into one taxonomy. Threads worth chasing next: pure-vision agents and how grounding quality gates them; RL and inference-time reasoning (UI-R1 / GUI-R1 / InfiGUI-R1) as the current frontier for long-horizon control; the GUI-vs-API debate and hybrid agents that mix UI operations with native API calls; and standardizing a unified cross-platform interface to make benchmarking tractable.

Limitations

  • Survey, not measurement: it catalogs and narrates trends but runs no head-to-head experiments, so cross-system claims rest on each cited paper’s own (often non-comparable) numbers.
  • Fast-aging by construction — the field moves monthly; the static PDF is a snapshot (the extracted text corresponds to a later v8/v12 update, May 2025), and even the GitHub list lags. Treat any “state of the art” as dated.
  • Breadth over depth and a Microsoft-authored vantage point: heavy coverage of in-house systems (UFO, Power Automate, Copilot, OmniParser) and limited critical/quantitative comparison between frameworks.
  • The forward-looking challenges (§11) are well-enumerated but mostly identify problems and “potential solutions” rather than evaluate them: privacy, latency/resource constraints, safety/reliability, human-agent interaction, customization, ethics/regulation, scalability/generalization.

Critiques and Questions

  1. The “GUIs are universal, APIs aren’t always exposed” argument is persuasive, but the survey’s own §6 trend toward hybrid (UI + API) agents undercuts a pure-GUI stance — is GUI control a transitional surface until apps expose agent-first APIs?
  2. With no unified evaluation, how much of the reported per-system progress is real vs. benchmark-shopping? The survey flags non-standard metric naming but cannot resolve it.
  3. The 5-component architecture is clean, but where do world models and search live — inside “inference”, or as a distinct planning component the taxonomy underweights?
What are the five core components of the survey's reference GUI-agent architecture, in workflow order?
Operating Environment (perception) → Prompt Engineering → Model Inference → Action Execution → Memory, iterated until the task completes (§5.1).
