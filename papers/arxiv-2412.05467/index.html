BrowserGym is a unified Gym-style environment that exposes six fragmented web-agent benchmarks behind one observation/action API, paired with AgentLab for reproducible large-scale experiments; using it, the authors run the first multi-benchmark comparison of six LLMs/VLMs.

Foundational infrastructure for the computer-use / web-agent track: it standardizes how web agents are built and evaluated, and provides the first apples-to-apples cross-benchmark leaderboard, with Claude-3.5-Sonnet emerging as the strongest general web agent.

Motivation

The web-agent field has produced an abundance of benchmarks, but each ships diverging evaluation code, incompatible implementations, and inconsistent practices — “research silos.” This fragmentation blocks fair cross-benchmark comparison and forces every researcher to re-do the cumbersome work of installing diverse setups, adapting data formats, and manually wiring agents into each separate benchmark. The result is unreliable, hard-to-reproduce comparisons of web-agent capabilities.

Contribution (what’s new)

The genuine delta is an ecosystem that exposes every existing benchmark through one standard Gym-like observation/action interface, so an agent written once runs unchanged on all of them — and a benchmark added once is immediately evaluable by all existing agents. The design must take the form of a thin POMDP-style environment wrapper (<Term def="OpenAI/Farama Gym reset()/step() loop where env returns observation+reward and agent returns action">gymnasium</Term> API over Chromium + Playwright) precisely because the baseline failure was per-benchmark glue code; only a shared interface with a standardized observation space (DOM, AXTree, screenshot, bid-tagged elements) and a pluggable action_mapping removes that glue. AgentLab adds the complementary tooling (parallel Study runner, AgentXRay trace viewer, reproducibility journal, dynamic prompting, unified LLM API). Note this paper extends BrowserGym from Drouin et al. (2024); the new benchmarks it integrates are WebLINX, VisualWebArena, and AssistantBench.

Research questions

For this system/infrastructure paper the central questions are: (1) Can a single unified interface meaningfully absorb the heterogeneity of existing web-agent benchmarks without losing benchmark-specific fidelity? (2) When you finally evaluate the same agent across all of them, how do today’s state-of-the-art LLMs/VLMs actually compare?

Methodology

Define a web-agent interaction as a <Term def="Partially Observable Markov Decision Process — agent sees partial observations, picks actions, receives reward">POMDP</Term> and expose it through the gymnasium reset()/step() loop. The environment provides a rich observation (task goal/chat, open tabs, and per-page DOM + <Term def="accessibility tree — structured semantic representation of a page used by screen readers">AXTree</Term> + raw screenshot, with each element enriched by a unique bid, bounding box, visibility ratio, and a <Term def="overlay element boxes/IDs on the screenshot to help a VLM ground actions">Set-of-Marks</Term> flag). The native action space is raw Playwright/Python code, optionally constrained to a safe HighLevelActionSet (e.g. click(bid), fill(bid, text), mouse_click(x,y)). New tasks need only a setup() and validate() method.

Method details: observation, action, AgentLab
  • Observation space (§3.1): goal_object/chat_messages, open_pages_urls/titles/active_page_index, dom_object, axtree_object, screenshot, plus extra element properties (bid, bbox as (left, top, width, height), visibility in [0,1], set_of_marks). last_action_error feeds Playwright exceptions back to the agent for self-correction.
  • Action space (§3.2): raw Python over a Playwright page plus send_message_to_user() / report_infeasible(); an optional action_mapping converts structured actions (JSON/command) into trusted Python, mitigating arbitrary-code risk. The default HighLevelActionSet covers bid-, coordinate-, keyboard-, and tab-level primitives (Tab.3).
  • Extensibility (§3.3, §5.5): a task = setup(page)→goal + validate(page, chat_messages)→(reward, done, message); an agent = bgym.Agent with a get_action(obs)→(action, agent_info), plus AgentArgs for picklable config.
  • AgentLab (§5): Study orchestrates parallel runs (ray/joblib, up to ~20 tasks/laptop, 50–100/server; auto-relaunch failed tasks up to 3×, but (Visual)WebArena dependencies cap parallelism at 2–4). AgentXRay is a Gradio trace inspector. Reproducibility features log package/benchmark versions, commit hash, OS; a reproducibility_journal.csv tracks drift; a ReproducibilityAgent replays an action sequence and diffs prompts. DynamicPrompting shrinks prompts component-wise to fit context (AXTree can exceed 100k tokens, raw DOM >1M).
  • Benchmark scale (Tab.1): MiniWoB++ 125 templates (max 10 steps), WebArena 812, VisualWebArena 910, WorkArena L1 33 / L2 341 / L3 341, WebLINX 31,586 (single-step, static), AssistantBench 214 (open web). Multi-tab: WebArena, VisualWebArena, WorkArena L2/L3, AssistantBench.

Main result

On the first unified, multi-benchmark sweep, Claude-3.5-Sonnet wins the majority of benchmarks and reaches an unprecedented 39.1%±3.2 on WorkArena L2, versus 8.5%±1.8 for second-place GPT-4o (Tab.2, p.16) — a previously near-unsolved enterprise benchmark. The authors credit Claude’s dedicated computer-use training as a likely cause.

Key findings

  • Claude-3.5-Sonnet is the strongest general web agent, with o1-Mini second (Tab.2): Claude leads on MiniWoB (69.8%), WorkArena L1 (56.4%), L2 (39.1%), WebLINX (13.7%), and WebArena (36.2%).
  • Vision is the exception: GPT-4o beats Claude on the visual benchmark (Tab.2, §6.2): 26.7% vs 21.0% on VisualWebArena, the only benchmark where Claude is not on top.
  • Even the best model collapses on the hardest enterprise tasks (§6.2, p.15): WorkArena L3 is essentially unsolved (Claude 0.4%, all others 0.0%), showing how far current agents are from complex compositional workflows.
  • Open-weights are closing the gap (§6.2, p.16): Llama-3.1 70B roughly matches GPT-4o Mini, and Llama-3.1 405B significantly surpasses GPT-4o Mini on several benchmarks — “a promising omen for the open-source community.”
  • Web benchmarks may be aging / contaminating (§6.2, p.16): a newer GPT-4o checkpoint jumped from 23.5%→31.4% on WebArena and 3.8%→8.5% on WorkArena L2 versus Boisvert et al. (2024) with a near-identical agent — attributable to better reasoning or benchmark content leaking into training data.
  • The generic agent underperforms on retrieval-heavy tasks (§6.2, p.16): best AssistantBench score is only 6.9% vs the leaderboard’s ~27%, because GenericAgent prioritizes generality over information-retrieval specialization.

Implications

This is plumbing the whole subfield can stand on: it lowers the cost of cross-benchmark evaluation, makes new benchmarks instantly testable against existing agents, and seeds an online leaderboard plus reproducibility journal that track benchmark drift over time. It builds directly on Drouin et al. (2024) BrowserGym and the WorkArena line and unifies the WebArena / VisualWebArena, WebLINX, AssistantBench, and MiniWoB lineages under one roof. Threads worth chasing: why WorkArena L3 is unsolved and what scaffolding (planning/search, memory) moves the needle, whether the GPT-4o jump reflects real gains or benchmark contamination, and using BrowserGym’s logged traces as fine-tuning data for web agents.

Limitations

  • Reproducibility on live sites remains fragile: localization (time zone, language, region), OS/browser rendering differences, and dynamic content (ads) inject non-determinism the framework can only partly mitigate.
  • Open-web safety and robot detection: agents acting on the live web (AssistantBench, future GAIA) face CAPTCHA, IP rate-limiting, and behavior-based blocking, plus genuine risk of harmful autonomous actions — mitigated only by URL protection and warning notices.
  • Agent collisions and forced serialization: shared mutable backends (WebArena/VisualWebArena use one VM per agent) mean only one agent can run at a time, badly slowing evaluation.
  • Synchronous sync-loop bottleneck: rapid sequential actions can outrun the browser sync loop, causing delays.
  • Error analysis is only qualitative: §6.3 offers a category taxonomy (navigation, form-handling, task-understanding, stuck behavior, extraction, external errors) but no quantitative breakdown, given experiment scale.

Critiques and Questions

  1. The headline 39.1% framing flatters a single best model; absolute numbers (L3 at 0%, WebArena ~36%) show web agents are still far from reliable — the “ecosystem” contribution is more durable than the leaderboard snapshot.
  2. The GPT-4o improvement is explicitly ambiguous between genuine reasoning gains and training-data contamination; without held-out or freshly-authored tasks, public web benchmarks have a real and unmeasured aging problem.
  3. Single-run success rates with standard error σ/√N are reported, but several benchmarks are non-deterministic and some models were skipped (WorkArena L3 reuses prior numbers, VisualWebArena multimodal-only), so cells aren’t strictly comparable.
  4. Is a single GenericAgent the fair instrument for cross-model comparison, when per-benchmark specialization (e.g. on AssistantBench) clearly matters?
Why must BrowserGym's contribution take the form of a shared Gym-style observation/action interface rather than per-benchmark adapters?
Because the baseline failure mode was per-benchmark glue code that created research silos. Only a single standardized interface (one observation space — DOM/AXTree/screenshot with bid-tagged elements — and a pluggable action_mapping over a POMDP reset/step loop) lets an agent written once run unchanged on every benchmark and a new benchmark be evaluated by every existing agent, eliminating the glue.
