Mobile-Agent-E is a hierarchical multi-agent mobile GUI assistant that separates high-level planning from low-level action and adds a self-evolution module — persistent Tips and reusable Shortcuts learned across tasks — yielding a 22.1% absolute Satisfaction-Score gain over the prior SOTA on a new long-horizon benchmark.

A foundational computer-use-agent design point: it shows that an explicit planning/action hierarchy plus a cross-task memory of natural-language Tips and executable Shortcuts measurably improves both accuracy and efficiency on realistic, multi-app mobile tasks.

Motivation

Current LMM-based mobile agents face two gaps. First, existing mobile benchmarks are short, goal-oriented, single-app tasks (e.g. “navigate to a gas station”) whose performance is already saturated, so they fail to reflect real human needs that require reasoning over multiple constraints, long-horizon planning across apps, and active exploration of vague instructions. Second, existing agents cannot learn from prior experience — they treat every task as a first attempt, spend the same compute each step, and repeat the same mistakes even on recurring subroutines like searching or note-taking.

Contribution (what’s new)

The genuine delta is twofold. (1) An explicit agent hierarchy that disentangles high-level planning (a Manager that decomposes a task into subgoals) from low-level action (an Operator that emits concrete tap(x,y)-style actions) — the prior SOTA Mobile-Agent-v2 conflates “what to do next” and “where to tap” in one decision module, which limits long-horizon planning and error recovery, so the design must split these roles. (2) A self-evolution module with a persistent long-term memory of two knowledge types: Tips (natural-language lessons, akin to episodic memory) and Shortcuts (reusable executable atomic-operation sequences with explicit preconditions, akin to procedural knowledge), updated after each task by two Experience Reflectors and consumed by the Manager and Operator on later tasks.

Research questions

  1. Does an explicit planning/action hierarchy improve performance on reasoning-intensive, long-horizon, multi-app mobile tasks over flat multi-agent designs?
  2. Can a mobile agent self-evolve across tasks — accumulating Tips and Shortcuts — to improve both task performance and inference efficiency?
  3. How should such complex, exploratory tasks be evaluated when there is no binary success flag or single ground-truth trajectory?

Methodology

Mobile-Agent-E runs a per-step agent loop over a real Android device (controlled via ADB) and a cross-task self-evolution loop over a shared long-term memory. All reasoning agents are instantiated from a frozen LMM (default GPT-4o); only the Perceptor is a separate vision pipeline.

Method details

Hierarchical agent loop (per step t).

  1. Perceptor (A_P) — Vision-only perception (no XML): OCR (DBNet), icon grounding (GroundingDINO), icon captioning (Qwen-VL-Plus) produce a list of texts/icons with coordinates W_V. The raw screenshot is still passed to downstream agents as holistic context.
  2. Manager (A_M) — High-level planning: from the query, screenshot, previous plan/subgoal/progress/notes and available Shortcuts, it outputs an updated overall plan and the next subgoal. It does NOT see fine-grained perception, to avoid noise in high-level planning.
  3. Operator (A_O) — Low-level action decision: from the query, screenshot, perception W_V, plan/subgoal/progress/notes, the last m=5 actions and errors, plus Shortcuts and Tips, it emits the next action a_t. The action space includes atomic operations AND Shortcuts.
  4. Action Reflector (A_R) — Compares before/after screenshots to label the outcome: A successful/partially successful, B failed (wrong page), or C failed (no change). On B/C it records an error description with suggested causes/fixes; on A it updates action history and progress.
  5. Notetaker (A_N) — Aggregates task-relevant information (prices, phone numbers, etc.) into persistent notes W_N after each step.

Error escalation. When the Operator first hits an error it tries to fix it; after k=2 consecutive failures an Error Escalation Flag is raised and the Manager re-plans from a higher level (revising the plan or subgoal). A task is force-exited (counted as a termination error) on max 40 iterations, 3 consecutive errors, 3 repeated identical actions (excluding Swipe/Back), or any parsing error.

Atomic operation space (Tab.8): Open App, Tap, Swipe, Type, Enter, Switch App, Back, Home, Wait.

Self-evolution module. At the end of each task (final step τ), two Experience Reflectors update memory from the full action/error history (and, in the evolution setting, the queries of remaining tasks): A_ET updates Tips (natural language) and A_ES proposes new Shortcuts (JSON with name, arguments, description, precondition, and an atomic-action sequence). The Operator must verify a Shortcut’s precondition before invoking it; a Shortcut collapses several atomic ops into one decision step.

Benchmark — Mobile-Eval-E (Tab.2). 25 manually crafted tasks across 5 scenarios (Restaurant Recommendation, Information Researching, Online Shopping, What’s Trending, Travel Planning), spanning 15 apps; 19/25 (76%) are multi-app vs <10% in prior benchmarks; 14.56 avg ops/task (364 total) vs 5.55–6.31 in Mobile-Eval / Mobile-Eval-v2 / AppAgent.

Metric — Satisfaction Score (SS). Per task, humans write rubrics mixing milestones (“Opened Tripadvisor”) and exploratory criteria (“Viewed multiple attractions”); SS = fulfilled rubrics / total rubrics, judged by a human. Also reported: Action Accuracy (AA), Reflection Accuracy (RA), Termination Error rate (TE), and a Satisfaction-Score-vs-Steps (SSS) curve for efficiency (steeper/higher = better).

Main result

With an identical GPT-4o backbone, Mobile-Agent-E beats the prior multi-agent SOTA (Mobile-Agent-v2) on every metric, and enabling self-evolution widens the gap: Satisfaction Score rises from 53.0% (v2) → 75.1% (Mobile-Agent-E) → 86.9% (+Evo) (Tab.3), i.e. +22.1 absolute from the hierarchy and +33.9 absolute total over prior SOTA, while Termination Error drops from 52.0% → 32.0% → 12.0% (Tab.3) — much better robustness/error recovery.

Key findings

  • Gains hold across three backbones: average absolute SS gain is +15.6% (no evolution) and +22.1% (with evolution) over Mobile-Agent-v2 across GPT-4o, Claude-3.5-Sonnet, and Gemini-1.5-pro (Tab.4): the architecture, not one model, drives the improvement.
  • Self-evolution helps stronger backbones more (Tab.4): on GPT-4o SS jumps 75.1→86.9 and on Claude 75.5→83.0 with Evo, but on Gemini it barely moves (70.9→71.2), suggesting evolution depends on the LMM’s reflection quality.
  • Self-evolution compounds over time (Fig.6): tasks later in a scenario’s sequence (which inherit more accumulated Tips/Shortcuts) show larger gains, though not strictly monotonic since task difficulty varies by index.
  • Shortcuts cut compute, recovering speed lost to the deeper hierarchy (Tab.5): a Shortcut executes multiple ops in one perception+reasoning iteration; with Evo, GPT-4o reaches 27.4s/op (perception+reasoning), faster than Mobile-Agent-v2’s 43.5s, with Shortcut usage at ~12–15% of all actions.
  • Tips help on their own, isolated from Shortcuts (Tab.6): restricting to trajectories with no newly generated Shortcuts, evolved Tips still raise SS (e.g. GPT-4o 79.7→87.5, Claude 75.6→85.2), so episodic-style guidance contributes independently of procedural Shortcuts.
  • Mobile-Eval-E is genuinely harder (Tab.2): prior SOTA reaches only ~50–70% of human satisfaction, and the benchmark has 2× the ops/task and far more multi-app tasks (76% vs <10%) than predecessors.

Implications

This consolidates the Mobile-Agent line (v1, v2) and the broader LMM GUI-agent lineage (AppAgent, AndroidWorld, OSWorld, Cradle) by importing two ideas: a planning/action hierarchy and a self-evolving skill/lesson memory. Unlike Cradle’s skill curation, which adds skills directly to the atomic action space, Mobile-Agent-E attaches explicit preconditions to Shortcuts, which the authors argue is critical for multi-app, layout-varying environments. Threads worth chasing: the closed-loop variant with Experience Retriever agents that select relevant Tips/Shortcuts from a large memory (58 Tips, 7 Shortcuts after 25 tasks) — needed once memory outgrows the context window, and whether agent-generated Shortcuts can be self-revised to fix the precondition/parameter errors observed.

Limitations

  • Shortcuts are misused when the Operator misperceives the phone state and violates a precondition (e.g. invoking a search Shortcut before the search bar’s app is foregrounded; App.C.1, Fig.10).
  • Self-generated Shortcuts can be imperfect or erroneous — a redundant Tap, or a missing Tap needed to enter an app — and such errors propagate to later tasks (App.C.2, Fig.11).
  • Evaluation is small-scale and human-dependent: only 25 tasks, on a single device (Samsung Galaxy A15), with Satisfaction Score / Action Accuracy / Reflection Accuracy all judged by human raters against hand-written rubrics; inter-rater reliability is Not reported.
  • Numbers appear to be single-run; variance / error bars are Not reported.

Critiques and Questions

  1. Satisfaction Score is a human-rated rubric ratio with no reported inter-annotator agreement, and the same authors design both the benchmark and the rubrics — the headline “+22.1%” should be read with that subjectivity in mind.
  2. In the evolution setting the Experience Reflectors are fed the queries of remaining tasks in the scenario, so a task partly benefits from foreknowledge of upcoming tasks; this is a generous notion of “self-evolution” versus a strictly online agent.
  3. With only 25 tasks and apparently single runs, the per-cell differences (especially Gemini’s flat 70.9→71.2) sit within plausible noise; replication and variance reporting would strengthen the claims.
  4. Shortcuts/Tips are evaluated within the same 5 scenarios they were learned in; transfer to genuinely unseen app domains is shown only qualitatively in one case study (Fig.7), not measured.
Why does the Manager deliberately not receive the Perceptor's fine-grained perception output?
High-level planning only needs the task, current screen, and prior plan/progress/notes; the detailed icon/text list would add noise to subgoal decomposition. Fine-grained perception is instead given to the Operator, which needs precise coordinates for actions like tap(x,y).
