A compositional computer-use agent that routes each action’s grounding to a specialist expert (visual / OCR / spreadsheet-API) and re-plans subgoals after every step, setting new SOTA on OSWorld, WindowsAgentArena, and AndroidWorld.

The lineage successor to Agent S from an open-code lab; its “compose cheap specialists to beat the best monolith” thesis is directly relevant to GUI-agent harness design (planner/worker/grounder separation and routing).

Motivation

Computer-use agents must do three cognitively distinct things at once — long-horizon planning, action generation, and pixel-precise grounding — yet prior work pushes all three through a single model. That monolith fails on two fronts: it imprecisely grounds GUI elements to coordinates and struggles on long-horizon tasks amid pop-ups and evolving screens. A structural reason: fine-tuning one model to specialize on a sub-skill erodes its other abilities (catastrophic forgetting), so a generalist is bottlenecked by its weakest sub-skill. The best agents still trail humans by ~40% on OSWorld (p.1). (Problem only — not the fix.)

Contribution (what’s new)

Two transferable mechanisms layered on the inherited Agent S Manager/Worker + knowledge base:

  • Mixture-of-Grounding: treat grounding as a routing problem — the Worker gates each action to a specialist tool because a single visual grounder misses fine targets (exact word edges, resized cells) that non-MLLM tools handle better (OCR for spans, a spreadsheet API for cells).
  • Proactive Hierarchical Planning: re-plan after every subgoal, not just on failure — a reactive plan cannot absorb information learned from a successful subgoal, so proactive re-planning enables backward correction and adaptive exploration as the partial screen is revealed.

The core empirical claim: composing individually-suboptimal specialists can beat the best monolithic model.

Research questions

Can splitting cognition across a generalist planner/worker and specialist grounders — plus per-subgoal re-planning — outperform monolithic native agents on real computer-use tasks, and does each component contribute?

Methodology

A Manager decomposes the instruction into subgoals; a Worker executes the top subgoal as atomic actions, each tagged with a language descriptor of its target, and routes that descriptor to a grounding expert rather than localizing itself. After each subgoal returns SUCCESS/FAILURE, the Manager regenerates the remaining subgoals from the latest observation. Screenshot-only input; no fine-tuning (off-the-shelf backbones + pretrained grounders).

Method details

Three grounding experts:

  • Visual — description → ⟨x,y⟩ pixel coordinate, screenshot-only (UI-TARS-72B-DPO in their setup).
  • Textual — Tesseract OCR returns start/end coordinates of a word span given two boundary phrases (highlight_text_span).
  • Structural — UNO interface programmatically writes spreadsheet cells from a {cell:value} dict (set_cell_values), assuming a UNO-controllable app (LibreOffice).
  1. Plan — Manager decomposes instruction into high-level subgoals using the Agent S knowledge base (task/web experience).
  2. Execute — Worker emits an atomic action with a language descriptor of the target location.
  3. Route (MoG) — Worker acts as a gate, selecting the visual / textual / structural expert to localize the target.
  4. Re-plan (PHP) — On subgoal SUCCESS/FAILURE, Manager regenerates remaining subgoals from prior subgoals + latest observation + original instruction.

Backbones (Manager/Worker): Claude-3.7-Sonnet (best, thinking on), Claude-3.5-Sonnet (new), GPT-4o. POMDP framing; no explicit tree search; AndroidWorld uses Worker-only mode (short horizons).

Main result

New SOTA across three benchmarks. On OSWorld with Claude-3.7: 27.0% @15-step and 34.5% @50-step (Tab.1, p.6), vs UI-TARS-72B-DPO 22.7% / OpenAI CUA 32.6% and Claude-3.7 Computer Use baselines. Also WindowsAgentArena 29.8% (+52.8% rel.) beating a11y-tree-using NAVI 19.5% (Tab.3, p.7) with screenshots only, and AndroidWorld 54.3% (+16.5% rel.) vs UI-TARS-72B-SFT 46.6% (Tab.4, p.9). A direct novelty test: an in-framework UI-TARS-7B-DPO grounder (29.23%) beats the Claude-3.7 generalist grounder (24.61%) (Fig.6, p.8). All numbers appear to be single runs with no variance; the 50-step OSWorld margin over OpenAI CUA is only 1.9 pts; ablations use a 65-task subset; the abstract’s “+18.9%/+32.7%” relative framing uses different baselines/variants than p.2, so it is easy to misread.

Implications

A testable case that modular composition of off-the-shelf models beats monolithic native agents on computer use, and that grounding is better framed as per-action routing than as one skill. For harness design: separate planner/worker/grounder, route grounding to the cheapest tool that works, and re-plan per subgoal. Lowers the adoption barrier by reusing commodity OCR/UNO and open code.

Limitations

  • No variance/error bars; headline and ablation numbers appear single-run, so small SOTA margins (e.g. +1.9 pts on 50-step OSWorld) could be within noise.
  • Ablations on a 65-task subset, not the full 369.
  • Cost/latency of per-subgoal re-planning and per-action routing is never quantified.
  • Structural grounding is tied to UNO-controllable office apps — not a general mechanism.
  • By the authors’ own error analysis, planning is now the dominant failure (41.0%) — the framework moves the bottleneck rather than removing it; still ~40% below humans.

Critiques and Questions

  1. With single-run numbers and a 1.9-pt 50-step margin, how much of the SOTA is noise? Re-run with seeds.
  2. Routing is itself an LLM decision — what is the misroute rate, and how often does a wrong expert choice cause failure?
  3. What is the latency/cost multiplier of re-planning after every subgoal vs reactive? The obvious price of the design is unmeasured.
  4. Does the compositional advantage survive when grounders are also weak, or does it depend on a strong UI-TARS visual expert?
  5. How does the v1→S2 delta isolate against the original Agent S? Check the repo for Manager/Worker/router prompts and the 65-task subset selection.
