Under rigorous human evaluation on a new online benchmark (Online-Mind2Web, 300 realistic tasks across 136 sites), frontier web agents are far weaker than their self-reported numbers suggest — most solve only ~30% of tasks — and the gap traces to flawed benchmarks rather than real progress.

Foundational reality-check for the computer-use / web-agent track: it shows headline success rates on WebVoyager are inflated by easy tasks and unreliable LLM judges, and offers a better benchmark plus a scalable LLM-as-a-Judge (WebJudge) for honest evaluation.

Motivation

Recent web agents claim close to 90% success on the WebVoyager benchmark, fueling optimism that web automation is nearly solved. But most existing benchmarks are offline (cached site snapshots) or sandboxed, so they cannot reflect how real users interact with live, dynamic websites. Even the most-used online benchmark, WebVoyager, has serious flaws: it covers only 15 websites with limited task diversity, many tasks have shortcut solutions, and its automatic evaluation agrees poorly with humans. These issues together inflate reported competence and risk short-sighted decisions about the field’s true state.

Contribution (what’s new)

The genuine delta is a rigorous, human-grounded online benchmark that minimizes the simulation-to-reality gap, plus a scalable judge that makes such evaluation repeatable. Online-Mind2Web is built from crowdsourced, validated, real-world tasks (not synthetic prompts) spanning 136 popular sites, so it resists the shortcut-and-similarity biases of synthesized benchmarks. WebJudge mitigates the core failure of prior LLM judges by identifying key points and selecting only key intermediate screenshots — this design is forced because looking only at the final screenshot misses critical intermediate steps, while feeding all screenshots overloads the context window.

Research questions

  1. How capable are current frontier web agents really, when measured under conditions that approximate real users on live websites?
  2. Why do self-reported benchmark numbers (e.g., WebVoyager) overstate competence?
  3. Can an automatic evaluator reach human-level reliability while remaining scalable across agents with very different output formats and trajectory lengths?

Methodology

Construct a realistic online benchmark from validated real tasks, manually evaluate six frontier agents as ground truth, then develop and validate an LLM-as-a-Judge against that ground truth.

Method details

Benchmark construction (Online-Mind2Web). From 650 randomly sampled original Mind2Web tasks, 47% were found invalid or to have outdated ground-truth trajectories (§2.2, p.4). The final set of 300 tasks combines 167 selected from Mind2Web, 24 rewritten Mind2Web tasks, 34 from Mind2Web-Live, and 75 newly authored on high-traffic sites. Tasks are graded by reference length Nstep (steps a human needs): easy (Nstep ≤ 5, 83 tasks), medium (6 ≤ Nstep ≤ 10, 143 tasks), hard (Nstep ≥ 11, 74 tasks). Agents are given a start URL and prompted not to use Google Search to block shortcuts.

Shortcut diagnosis. A naive search agent solved 51% of a 100-task WebVoyager sample but only 22% of Online-Mind2Web (≈50% easy, 18% medium, 3% hard), confirming WebVoyager’s tasks skew easy and shortcut-solvable.

Evaluation. Every trajectory of all six agents was human-annotated (≥2 annotators, a third resolves conflicts) to form the reference standard.

WebJudge (LLM-as-a-Judge), 3 steps:

  1. Key Point Identification — Prompt the model to extract the key requirements K = (k1..km) a task needs (e.g., location, rating).
  2. Key Screenshot Identification — Score each screenshot's relevance 1–5; keep those above threshold δ, discarding uninformative frames to preserve intermediate evidence without token overload.
  3. Outcome Judgement — Given task, key points, key screenshots, and action history, output binary Success/Failure: O = Evaluator(T, A, I).

WebJudge deliberately avoids the agent’s final response (often hallucinated) and judges environment observations rather than LLM-generated text, reducing self-preference bias. A lightweight WebJudge-7B (Qwen2.5-VL-7B) fixes cost at two API calls per trajectory.

Main result

Under human evaluation, OpenAI Operator (61.3%) and Claude Computer Use 3.7 (56.3%) lead, while every other agent — including Browser Use (30.0%), SeeAct (30.7%), Claude Computer Use 3.5 (29.0%), and Agent-E (28.0%) — clusters around 30% (Tab.2, p.6). This stands in stark contrast to WebVoyager self-reports: agents drop by 23–59 percentage points (e.g., Browser Use 89%→30%, Agent-E 87%→28%), and most recent agents fail to beat the simple SeeAct agent released in early 2024 (Fig.1, p.1).

Key findings

  • WebJudge (o4-mini) reaches 85.7% agreement with humans at only a 3.8% success-rate gap; WebJudge-7B reaches 87% agreement at a 3.9% gap with just two API calls per trajectory (Tab.2, p.6): a scalable judge can substitute for human evaluation for rapid iteration.
  • Success collapses with difficulty — average SR drops 31.6 points from easy→medium and a further 15.4 from medium→hard (Fig.4, p.7): even the best agents (Operator 83.1%, Claude 3.7 90.4% on easy) struggle on hard tasks, pointing to weak long-horizon planning and exploration.
  • Prior judges degrade as trajectories lengthen — WebVoyager suffers token overload from too many screenshots, and Autonomous Evaluation (final-screenshot only) misses intermediate steps (Fig.5, §4.3, p.7): WebJudge stays accurate up to ~80 action steps, explaining its edge on long Operator runs (often >100 screenshots).
  • Hallucinations in agents’ final responses inflate WebVoyager-style scores via false positives (§4.3, p.7): a likely contributor to previously over-optimistic reported numbers.
  • WebJudge generalizes out-of-domain — 73.7% (GPT-4o), 75.7% (7B), 82.0% (o4-mini) overall precision across 1,302 trajectories on 5 benchmarks in AgentRewardBench (Tab.4, p.8): suggests use as a reward model for RFT/RL/reflection.
  • Failed tasks take far more steps than successful ones (≈2× for Browser Use, Claude 3.7, Operator), and Operator trades efficiency for exploration (2.6× human reference, up to 44 min) (Fig.6, §5.1, p.9): an exploration-vs-exploitation trade-off where greedy agents are efficient (≈1.0× human) but get stuck at dead ends.

Implications

The paper reframes web-agent “progress” as partly an artifact of weak evaluation, building on the Mind2Web / WebVoyager / Mind2Web-Live lineage and the LLM-as-a-Judge tradition (MT-Bench, AlpacaEval) and complementing AgentRewardBench, which it uses to test generalization. WebJudge’s high out-of-domain precision opens using it as a reward signal for Rejection-sampling Fine-Tuning, Reflection, and RL on web agents. The difficulty cliff motivates better long-horizon planning and exploration, and handling numerical/temporal constraints that Operator routinely violates.

Limitations

  • WebJudge’s cost and latency scale with the number of screenshots per trajectory, making evaluation of long-trajectory agents (e.g., Operator) expensive — partly addressed by WebJudge-7B’s fixed two calls.
  • WebJudge (o4-mini) is overly conservative (low recall), widening its success-rate gap on some benchmarks despite high precision.
  • Benchmark durability depends on ongoing manual maintenance — live websites change, so tasks must be replaced over time to stay valid.
  • Detailed error analysis focuses mainly on Operator; failure taxonomies for other agents are coarser.
  • Some residual evaluator randomness remains even at temperature 0 (avg SR std 1.1%, max 1.8%).

Critiques and Questions

  1. Disabling Google Search to block shortcuts is reasonable for measuring navigation, but it also diverges from how real users let agents act freely — is the “real-user-approximating” claim partly in tension with this constraint?
  2. WebJudge is validated against the same human annotations used as ground truth; its 85% agreement is reassuring, but how much does residual judge bias propagate when WebJudge is later used as a training reward (the proposed RFT/RL use)?
  3. The “illusion of progress” framing rests heavily on comparing to self-reported WebVoyager numbers from different harnesses/prompts — how much of the 23–59pp drop is benchmark difficulty vs. reproduction differences in agent setup?
Why does WebJudge select 'key screenshots' rather than using either the final screenshot alone or all screenshots?
Using only the final screenshot (as in Autonomous Evaluation) misses crucial intermediate steps and drops agreement on long trajectories; feeding all screenshots (as in WebVoyager) overloads the context window. Scoring and keeping only relevant intermediate frames preserves the evidence needed to judge completion while staying within token limits — which keeps WebJudge accurate even past ~80 action steps.
