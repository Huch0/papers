GUI-Actor replaces text-based coordinate generation for GUI grounding with an attention-based action head: a dedicated <ACTOR> token attends directly to relevant visual patches, proposing one or more click regions in a single forward pass, with a lightweight verifier picking the best candidate.

Grounding (mapping a plan to a screen location) is the perception bottleneck for computer-use agents. This is a foundational rethink of the grounding head itself — coordinate-free, patch-native, and verifier-augmented — directly relevant to my computer-use and general-agent tracks.

Motivation

GUI agents must perform visual grounding: locating the right screen region for an action from a screenshot plus a textual plan. Most prior methods formulate this as text-based coordinate generation — the VLM emits positions as language tokens (e.g. x=0.125, y=0.23). This inherits three structural weaknesses. First, weak spatial-semantic alignment: mapping pixels to numeric strings through a language head carries no spatial inductive bias, making it data-hungry and error-prone. Second, ambiguous single-point supervision: many points inside a button are valid, but point-prediction penalizes all deviations from one labeled point. Third, a granularity mismatch: coordinates are continuous and dense while ViT features are coarse patch-level, so the model must infer pixel-precise targets from coarse tokens, hurting generalization across resolutions and layouts. Even bounding-box variants still emit raw coordinate strings detached from visual features, lacking ROI-pooling or spatial-attention machinery to bridge intent and visual action.

Contribution (what’s new)

The genuine delta is grounding by attention rather than by token generation: a dedicated <ACTOR> token serves as a contextual anchor, and an attention-based action head aligns it with visual patch tokens to produce a spatial activation map over the screen. The design must take this form because the baseline fails at the interface between vision and action: emitting numbers through a language head discards the patch structure the vision encoder already provides. Attending directly over patches grounds actions at the backbone’s native patch resolution, removing the granularity mismatch. The same mechanism enables multi-patch supervision (all patches overlapping the ground-truth box are positives), which absorbs the natural ambiguity of valid click points, and lets the head propose multiple candidate regions in one forward pass at no extra inference cost. A lightweight grounding verifier then re-ranks candidates by marking each on the image and asking a small VLM whether the mark satisfies the instruction.

Research questions

  1. Can coordinate-free, attention-based grounding match or beat text-based coordinate generation, especially on out-of-distribution high-resolution interfaces?
  2. Does grounding at native patch resolution with multi-patch supervision improve sample efficiency and generalization?
  3. Can grounding be imparted to a frozen backbone VLM by training only the action head, without sacrificing its general-purpose abilities?

Methodology

Insert three special tokens (<ACTOR_START>, <ACTOR>, <ACTOR_END>) where a coordinate span would normally go. Take the final-layer hidden state of <ACTOR> as a query, run self-attention over the vision encoder’s patch features, project both query and patches into a shared space via two MLPs, and compute scaled dot-product attention to yield a per-patch score map. Train with next-token-prediction loss plus an action-attention loss (KL between predicted attention and a normalized multi-patch target mask). At inference, decode normally, extract the <ACTOR> attention map, take the top-K patches, and use the verifier to select the first candidate exceeding a confidence threshold.

Method details
  1. ACTOR token as anchor — Replace the coordinate span {x_i..x_j} in the pyautogui-style action string with <ACTOR_START>, <ACTOR>, <ACTOR_END>. The final-layer hidden state h_<ACTOR> is the contextual anchor for action attention (Eq. 1, §3).
  2. Attention-based action head — Visual patch features v_1..v_M from the vision encoder pass through a self-attention layer so patches of the same UI element share representations. Project h_<ACTOR> via MLP-T to z and each patch via MLP-V to z_i; attention weight a_i = softmax(z·z_i / sqrt(d)) over all M patches (Eqs. 2-4, §3).
  3. Spatial-aware multi-patch supervision — Convert each normalized GT box to a binary mask over the W×H patch grid (floor on left/top, ceil on right/bottom so all partially-covered patches count). Positives y_i=1, else 0. Loss is KL between attention {a_i} and normalized target p_i = y_i / (sum_j y_j + eps) (Eqs. 5-6, §3; Eqs. 8-9, App. B).
  4. Grounding verifier — Fine-tuned from UI-TARS-2B-SFT on 730K balanced examples built from OS-Atlas: positives place a hollow red circle at the box center; negatives use a wrong box center or a random outside point. Score s = P(True)/(P(True)+P(False)). 4-connected patch clustering yields weighted sub-patch centers to catch small icons between patches (§4, App. F).
  5. Inference — K=20 candidate patches (filtered below 20% of max attention), confidence threshold gamma=0.95 for ScreenSpot-Pro and 0.8 for ScreenSpot/-v2; return first candidate above threshold to save compute (§5 Implementation, App. F.2).
  6. Training recipe — Backbone Qwen2-VL-7B-Instruct (also Qwen2.5-VL, 2B/3B). ~1M screenshots / 9.6M elements from UGround Web-Hybrid, GUI-Env/Act, AndroidControl, AMEX, Wave-UI, 1 epoch. Warm-up trains only the action head (~20M/~100M params for 2B/7B) with backbone frozen, then full fine-tune (Tab. 7, §5).

Main result

On the hard, out-of-distribution ScreenSpot-Pro, GUI-Actor-7B (Qwen2-VL) scores 40.7 avg, beating UI-TARS-72B (38.1) with far fewer parameters and less data; the verifier lifts it to 44.2 (Tab.1). With a Qwen2.5-VL backbone, GUI-Actor-7B reaches 44.6 (no verifier) / 47.7 (verifier) and the 3B model reaches 42.2 / 45.9 (Tab.4). Notably, GUI-Actor-2B (36.7) already surpasses several 7B baselines and beats UI-TARS-2B (27.7) by +9.0 (Tab.1).

Key findings

  • Largest gains on the OOD benchmark (Tab.1, §5): GUI-Actor beats prior SOTA UI-TARS by +9.0 (2B) and +5.0 (7B) on ScreenSpot-Pro, the benchmark with the biggest domain shift — attributed to explicit spatial-semantic alignment at native patch resolution. On in-distribution ScreenSpot/-v2 the margin is smaller (e.g. 88.3 vs 89.5 vs UI-TARS-7B on ScreenSpot, Tab. 2).
  • Better sample efficiency (Fig.3, §5): GUI-Actor reaches its final ScreenSpot/-v2 accuracy using only ~60% of the training data, while the Aguvis coordinate-based baselines plateau only after 80–90%. On OOD ScreenSpot-Pro, GUI-Actor recovers from early dips and keeps improving, whereas baselines decline after an early peak (no sustained overfitting).
  • Frozen-backbone LiteTrain works (Tab.5, §5): training only the ~19M/~103M action-head params (backbone frozen) plus a verifier yields 34.0 (2B) and 35.8 (7B) on ScreenSpot-Pro — comparable to fully fine-tuned coordinate models — implying the backbone VLM already perceives UI semantics and coordinate-generation training mostly learns coordinate mapping, not understanding.
  • Verifier helps most when grounding is hard (Tab.5, §5): it adds ~4 points to GUI-Actor-7B and ~13 points to GUI-Actor-7B-LiteTrain on ScreenSpot-Pro, with smaller in-distribution gains.
  • Multi-region prediction is real, not redundant (Fig.4, §5): GUI-Actor shows a large Hit@1→Hit@3 jump (e.g. 40.7→45.4 on ScreenSpot-Pro for 7B) because the attention map yields genuinely distinct, mutually-exclusive candidates, whereas resampling coordinate baselines mostly returns near-identical points.
  • Ablation against coordinate supervision (Tab.6, §5): Aguvis-7B with bbox supervision (13.8 on ScreenSpot-Pro) does no better than point supervision (15.6) and both trail GUI-Actor-7B (40.7) — more spatial info in string form does not help without an architectural spatial mechanism.
  • Online OSWorld-W (Tab.10, §5): with GPT-4o as planner, GUI-Actor-7B reaches a 12.2% task success rate (6/49) vs NAVI/OmniAgent (10.2%) and Aguvis-7B point-sup (4.0%) — a small absolute number on a hard live benchmark.

Implications

This recasts GUI grounding as a detection/attention problem aligned with the vision backbone rather than a language-generation problem, extending the Aguvis / UGround / UI-TARS / OS-Atlas coordinate-generation lineage and connecting to attention-driven, training-free grounding (Xu et al., 2025) and classic ROI pooling / spatial attention from detection. The frozen-backbone result is the most provocative thread: if a small action head plus verifier suffices, grounding can be bolted onto general VLMs without eroding their other skills — a modular path to computer-use agents. The verifier also transfers to other grounding models (it boosts Aguvis too) at ~5% of Aguvis’s sampling cost, suggesting a reusable decision-refinement layer. Worth chasing: offset-based spatial refinement or higher-resolution encoders to fix the small-icon failure mode.

Limitations

  • Inherits the backbone’s fixed 28×28 patch size (Qwen2-VL Naive Dynamic Resolution): icons smaller than ~10×10 px are under-represented, pronounced in high-precision pro software like CAD (App. A). The clustering refinement only partially mitigates this.
  • Trained with supervised fine-tuning only — no continual pre-training, annealing, or DPO; the authors note UI-TARS-7B still wins on some metrics partly due to far larger (incl. proprietary) data and a heavier pipeline (§5).
  • The online evaluation is narrow: OSWorld-W is just 49 Windows tasks with GPT-4o as a fixed planner, and absolute success (12.2%) is low, so end-to-end agentic gains are only weakly demonstrated.
  • Verifier thresholds are hand-tuned per benchmark (gamma 0.95 vs 0.8), and best results lean on the extra verifier pass rather than the grounding head alone.

Critiques and Questions

  1. The headline “beats UI-TARS-72B” pairs GUI-Actor-7B with its strongest backbone (Qwen2.5-VL) or counts the bare Qwen2-VL number (40.7) against a 72B model trained very differently — fair on params, but the comparison mixes backbones and training regimes; the like-for-like same-backbone gains (vs Aguvis-7B) are the cleaner evidence.
  2. How robust is the verifier out of distribution? It is trained on OS-Atlas (desktop/mobile/web) but contributes the biggest lift exactly on the most OOD benchmark — is it generalizing or exploiting the red-circle marking artifact?
  3. Multi-patch supervision rewards covering the whole box; does this bias clicks toward element centers and hurt on elements where only a sub-region (e.g. a checkbox within a row) is actionable, as in the Fig. 4b “shopping cart” example?
  4. Single-run numbers throughout — no variance/error bars reported, and several margins on ScreenSpot/-v2 are within a point or two of strong baselines.
Why does attending over visual patches remove the 'granularity mismatch' that coordinate-generation suffers from?
Coordinate generation forces the model to emit continuous, pixel-level positions through a language head, while the vision encoder only provides coarse patch-level features — a resolution gap. GUI-Actor instead predicts an attention distribution directly over those same patches, so grounding happens at the backbone's native patch resolution with no need to invent sub-patch precision from coarse tokens.
