GTA1 pairs a click-rewarded RL grounding model (no chain-of-thought) with per-step test-time scaling over action proposals, reaching state-of-the-art on GUI grounding and OSWorld task execution.

Directly relevant to the computer-use-agents track: it shows a two-stage (planner + grounding) agent can beat native end-to-end agents, and that “thinking” is unnecessary — even harmful — for static GUI grounding.

Motivation

Two-stage GUI agents face a planning challenge: many valid action-proposal sequences exist for the same task, and GUI environments lack “lookahead” because actions have irreversible state effects, so committing to a single plan invites cascading failures. They also face a grounding challenge: SFT-based grounding trains models to hit the exact center of a target element, which misaligns with the task — any coordinate inside the element is valid — hurting generalization to high-resolution, professional interfaces. A further gap: prior RL grounding work copies CoT “thinking” from math/reasoning domains, even though it is unclear whether explicit reasoning or bounding-box rewards help grounding at all.

Contribution (what’s new)

Two complementary mechanisms. (1) A grounding model trained purely with a binary click reward — +1 if the predicted (x,y) lands inside the annotated bounding box, 0 otherwise — and NO chain-of-thought; this design follows because the reward must match the task’s true success criterion (any point in the box), which SFT’s center-only objective violates. (2) Per-step test-time scaling: sample L candidate proposals from the planner, let an MLLM judge pick the best, then ground only the chosen action; this sidesteps the missing-lookahead problem by exploring short-term alternatives concurrently instead of rolling out full irreversible sequences. A supporting piece is OmniParser-based data cleaning that drops training samples whose annotated box has IoU below 0.3 with any detected element, removing misaligned labels.

Research questions

  1. How can a GUI agent plan robustly despite no lookahead and many plausible action sequences?
  2. Is explicit “thinking” (CoT) or an auxiliary bounding-box reward necessary for effective GUI grounding?

Methodology

A two-stage agent: at each step the planner sees the task, trajectory, and current screenshot and samples L proposals; an MLLM judge selects the best; if it is coordinate-based, the grounding model predicts the click point, otherwise the action executes directly. Grounding is optimized with GRPO using only the click reward.

Method details
  1. Proposal sampling — At each step the planner (o3 or GPT-5) is given the user instruction, trajectory so far, and current screenshot, and samples L candidate action proposals (default L=8).
  2. Judge selection — An MLLM judge (which can be the planner itself) scores the L candidates on correctness, effectiveness, alignment, planning quality, and appropriateness, and returns the best index as JSON (§D.3, Tab.12-13).
  3. Grounding — If the chosen action is coordinate-based (e.g. a click), the grounding model maps the action description + screenshot to a precise (x,y); non-coordinate actions (key presses, text) execute directly (§3.1, Fig.2).
  4. Data cleaning — OmniParser detects element boxes; a training sample is discarded if max IoU between its annotated box and any detected box < τ=0.3, removing render/timing-misaligned labels (Eq.1, §3.2).
  5. RL training — GRPO samples G responses per input; reward r=1 iff predicted (x,y) is inside the annotated box (Eq.2); rewards are Z-score normalized into advantages (Eq.3) and optimized with a clipped surrogate (Eq.4). No CoT prompt — the model outputs coordinates directly.

Training detail (Tab.7, §B): GTA1-7B is initialized from UI-TARS-1.5-7B, GTA1-32B from OpenCUA-32B; rollout G=8, batch 256, ~250 iterations, ~70K samples drawn from Aria-UI-Web, OmniACT, UI-Vision, Widget Caption, and OS-Atlas-Desktop. 7B trains on 16 H100 (~2 days), 32B on 32 H200 (~1 day). The click reward (Eq.2):

r_i = \begin{cases} 1, & x_{min} \le x_i \le x_{max} \text{ and } y_{min} \le y_i \le y_{max} \\ 0, & \text{otherwise} \end{cases}

Main result

On the hardest grounding benchmark, ScreenSpot-Pro, GTA1-7B scores 50.1% — beating UGround-72B’s 34.5% despite ~10x fewer params — and GTA1-32B reaches 63.6% (Tab.1, p.6). On agent task execution, GTA1-32B w/ GPT-5 hits 63.4% on OSWorld-Verified and GTA1-7B w/ o3 reaches 45.2% on the original OSWorld, beating CUA o3’s 42.9% while using a 100-step (vs 200-step) horizon (Tab.4, p.9).

Key findings

  • Click reward alone works best for grounding (Tab.6, p.9): 50.1 / 92.4 / 67.7% on ScreenSpot-Pro / V2 / OSWorld-G, vs 44.5 / 89.3 / 59.9% when adding both IoU and format (thinking) rewards — the auxiliary rewards generally hurt (except marginally on ScreenSpot-V2).
  • “Thinking” helps only in dynamic environments (§4.3, p.7-8): in static grounding it gives near-zero gain, but on AndroidWorld a 7B model’s task success rose 39%→44% with CoT, attributed to richer textual context (task + trajectory + instruction).
  • Test-time scaling generalizes across horizons and to other agents (§4.3, p.9): with L=8, OSWorld success climbs from 41.3→43.4% (50 steps) and 43.4→45.2% (100 steps) vs the L=1 baseline; the strategy also boosts UI-TARS-1.5-7B (Fig.4).
  • Scaling proposals can substitute for execution length (Fig.4, p.8): UI-TARS-1.5-7B at 15 steps with L=32 already beats the un-scaled 100-step baseline, and because the L proposals are sampled concurrently this also cuts wall-clock time.
  • A 50-step horizon gives the largest gain from scaling (Fig.4, p.9): 15 steps is too few to finish some tasks, while 100 steps adds excessive slack that dilutes the benefit.
  • A two-stage agent matches/beats native end-to-end agents (§4, p.6): the paper claims to be the first to show a modular planner+grounding agent is competitive on dynamic benchmarks like OSWorld, challenging the assumption that end-to-end is inherently superior.

Implications

GTA1 reinforces the two-stage GUI-agent lineage (Aria-UI, UGround, Aguvis, Jedi, Agent S/S2) against the native end-to-end line (UI-TARS, OpenAI CUA, Claude Computer Use), and extends the R1-Zero-style RL-for-grounding thread (GUI-R1, UI-R1, InfiGUI-R1, GUI-G1) by showing CoT is unnecessary for static grounding. Why does test-time scaling peak at a 50-step horizon, and how does the optimal L scale with task difficulty? Whether a learned judge (vs the planner judging itself) would further reduce cascading failures, and how the approach transfers to mobile/dynamic settings where “thinking” already helps.

Limitations

  • Grounding still fails in some scenarios, e.g. selecting custom foregrounds/backgrounds in image editing (GMIP), echoing prior work (§5).
  • Test-time scaling and the strongest agent results depend on heavy proprietary planners (o3, GPT-5), and the agent’s headline numbers conflate grounding quality with planner quality (Tab.4) — the paper does not report agent success with an open planner at scale.
  • “Thinking” results in dynamic settings rest on a single AndroidWorld 39%→44% comparison the authors themselves attribute partly to training instability (§4.3); generalization is asserted, not broadly demonstrated.
  • The data-cleaning IoU threshold τ=0.3 and several '∗'-marked baseline numbers are self-evaluated and may change with improved eval scripts (Tab.1-3 footnotes).

Critiques and Questions

  1. The OSWorld comparison mixes step budgets and planners across rows (15/50/100/150/200 steps; GPT-4o/o3/GPT-5/Claude), so “SOTA” claims are only loosely apples-to-apples; the cleanest controlled signal is the L=1 vs L=8 ablation.
  2. The judge can be the planner itself — is the gain from genuine candidate diversity, or just from sampling-then-rerank temperature 1 noise? An entropy/diversity analysis of the L proposals is missing.
  3. How much of the 7B grounding win over 72B baselines is the click reward vs the OmniParser data cleaning? Tab.6 isolates the reward but not the cleaning.
Why does GTA1 reward any click inside the target box rather than the box's center, and what does Tab.6 show about adding a 'format' (thinking) reward?
A click anywhere inside the element is a valid interaction, so a center-only (SFT-style) objective misaligns with the task; the binary in-box click reward matches the true success criterion. Tab.6 shows click reward alone (50.1/92.4/67.7%) beats adding format and/or IoU rewards (e.g. 44.5/89.3/59.9% with all three), so explicit thinking is unnecessary and generally hurts static grounding.
