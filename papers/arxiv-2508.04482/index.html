Motivation

Virtual assistants (Siri, Cortana, Alexa, Google Assistant) hinted at autonomous device use but were held back by weak contextual understanding and limited model capability (§1). With (M)LLMs and products like Anthropic Computer Use, Apple Intelligence, AutoGLM, and Project Mariner, the field has exploded — but the research is fragmented across modalities, platforms, training strategies, and evaluation protocols, with no consolidated view of how OS Agents are built and assessed (§1, §6). The survey exists to unify that scattered body of work.

Contribution (what’s new)

The genuine delta is an integrated framework that decomposes OS Agents along two orthogonal axes — fundamentals (three key components × three core capabilities) and construction (foundation models vs. agent frameworks), then maps the evaluation landscape and open challenges onto it (§1). The framing must take this form because the two construction paths are complementary: domain-specific foundation models inject GUI knowledge via training, while non-tuning agent frameworks add reasoning/memory on top of general models — surveys that pick only one miss half the field.

Research questions

What are OS Agents, what components and capabilities define them, how are they constructed (model + framework), how are they evaluated, and what are the unsolved challenges?

Methodology

A literature survey, not an empirical study. The authors define OS Agents, build a taxonomy, and tabulate recent work along the taxonomy’s dimensions (Tab.1 foundation models, Tab.2 frameworks, Tab.3 benchmarks). A live GitHub repository tracks new work. Because it is a survey, per-system success-rate numbers are Not reported in a unified table — the contribution is the organization, not a leaderboard.

The taxonomy in detail

Fundamentals (§2). Three key components: Environment (desktop / mobile / web), Observation space (screenshots, screen text/descriptions, HTML/DOM), Action space (input / navigation / extended ops). Three capabilities: Understanding (parse cluttered high-res GUIs and HTML), Planning (decompose + adapt), Grounding (translate plans into executable actions with coordinates/values).

Construction — Foundation models (§3.1). Architectures: existing LLMs (HTML-only, e.g. WebAgent’s HTML-T5 + Flan-U-PaLM), existing MLLMs (LLaVA/Qwen-VL/InternVL/CogVLM backbones), concatenated MLLMs (LLM + vision encoder), and modified MLLMs (high-res encoders, e.g. CogAgent’s 1120×1120 EVA-CLIP cross-attention; Ferret-UI any-resolution; MobileFlow’s LayoutLMv3 GUI encoder). Training: pre-training (publicly-available + synthetic data; tasks = screen grounding, screen understanding, OCR), SFT (collect trajectories for planning, render GUIs for grounding, unify action spaces), RL (early “LLM as feature extractor” e.g. WebShop/MiniWob++, now “LLMs as agents” e.g. AutoGLM’s self-evolving online curriculum RL, AGILE).

Construction — Agent frameworks (§3.2). Four modules: Perception (textual-description vs. GUI-screenshot input; GUI grounding split into visual / semantic / dual — SoM prompting, SeeAct’s HTML-as-semantics, AppAgent/OSCAR dual grounding); Planning (global one-shot vs. iterative — CoT/ReAct/Reflexion/CoAT, task-driven replanning); Memory (internal / external / specific sources; optimization via management, growth-from-experience, retrieval — AWM workflow reuse, PeriGuru KNN); Action (input / navigation / extended-code-and-API ops).

Evaluation (§4). Protocols: objective (rule-based, exact/fuzzy/semantic match) vs. subjective (human, LLM-as-judge). Metrics: step-level (operation/element accuracy, F1, step SR) vs. task-level (overall SR, accuracy, reward; efficiency = step ratio, API cost, execution time, peak memory). Benchmarks (Tab.3): platform (mobile/desktop/web), setting (static vs. interactive→simulated/real-world), task (GUI grounding / information processing / agentic).

Main result

This is a survey; there is no single headline metric. The deliverable is the taxonomy plus three reference tables and a maintained repository. Per-system metrics: Not reported (the survey does not aggregate success rates into one comparison).

Key findings

The survey’s main organizing insights and aggregated claims:

  • Resolution is the dominant architectural bottleneck for GUI foundation models (§3.1, Tab.1): standard MLLM encoders run at ~224×224 while GUIs are ~720×1080, so small text and icons are lost — driving “modified MLLM” designs (CogAgent high-res cross-attention, Ferret-UI any-resolution, UI-Hawk shape-adaptive cropping).
  • Pre-training and SFT are bottlenecked by data, not architecture (§3.1.2, §3.1.3): public web/HTML data alone fails to generalize across platforms, so the field relies on synthesizing cross-platform grounding data (simulators, A11y trees, rendered webpages) and unifying heterogeneous action spaces.
  • RL has shifted from “(M)LLM as feature extractor” to “LLMs as agents” (§3.1.4): early work paired RL with behavior cloning (MiniWob++ human-level via behavioral priors); recent work makes the LLM the policy and uses RL to align it with task objectives (AutoGLM’s self-evolving online curriculum RL for error recovery).
  • Perception is converging from textual-DOM input toward GUI-screenshot input (§3.2.1), which better matches human behavior; grounding bifurcates into visual (SoM/bounding boxes), semantic (HTML/A11y descriptions), and dual grounding.
  • Planning splits cleanly into global (one-shot, CoT-based) vs. iterative (ReAct/Reflexion/CoAT) (§3.2.2), and iterative planning is essential because OS environments mutate after every action.
  • Memory is the lever for self-improvement (§3.2.3): internal/external/specific memory plus optimization (self-summarization, context clipping, backtracking, KNN/workflow retrieval) is how agents reuse experience and personalize.
  • Benchmarks are migrating from static (cached pages, one-step) to interactive real-world environments (§4.2.2) (OSWorld VMs across Windows/Linux/macOS, AndroidWorld emulators) because static eval is less authentic and cannot test multi-step adaptation.
  • OS Agents introduce attack surfaces that conversational LLM safety does not cover (§5.1): web indirect prompt injection (WIPI), environmental/pop-up injection, and the finding that refusal-trained LLMs lose their guardrails when deployed as browser agents.
  • Personalization and self-evolution remain largely aspirational (§5.2): academia lacks real user data, so most work optimizes task performance rather than long-horizon adaptation, though memory mechanisms are the validated path forward.

Implications

The survey consolidates the computer-use-agent lineage spanning OS-Atlas, OS-Copilot, AutoGLM, CogAgent, Ferret-UI, AppAgent, WebVoyager, SeeAct and the WebArena/OSWorld/AndroidWorld benchmark families into one map, and positions itself against concurrent GUI-agent / generalist-virtual-agent surveys (§6). Threads worth chasing: OS-Agent-specific defenses, which the survey flags as scarce relative to the catalogued attacks (§5.1.2); multimodal memory (image/voice) and its management/retrieval as the route to genuine self-evolution (§5.2); and “LLMs as agents” RL with self-evolving online curricula (AutoGLM) as the emerging training paradigm (§3.1.4).

Limitations

  • No unified empirical comparison — the survey tabulates which techniques each system uses but reports no consolidated success-rate leaderboard, so relative effectiveness is hard to judge.
  • “Grounding” is overloaded across three sections (capability in §2.2, visual grounding in §3.1.2, GUI-screenshot grounding in §3.2.1), each with a different definition — the authors footnote this but it remains a source of confusion.
  • Fast-moving field with hard recency limits — model rankings are pinned to Dec 2024 (Chatbot Arena) and coverage skews to 2023–2024; the maintained GitHub repo is the intended mitigation.
  • This arXiv version is the extended form of a 9-page ACL 2025 paper; depth is uneven and safety/personalization sections are comparatively thin.

Critiques and Questions

  1. Without any normalized metric across the surveyed systems, the taxonomy tells you what designs exist but not which win — readers must still go to OSWorld/AndroidWorld leaderboards themselves.
  2. The two-axis construction split (foundation model vs. framework) is clean but real systems blend both; the survey could say more about the foundation-model + framework co-design that top agents actually use.
  3. Safety coverage is attack-heavy and defense-light, accurately reflecting the literature but leaving the most practically urgent question (how to deploy these safely) open.
Why does the survey treat domain-specific foundation models and agent frameworks as two distinct construction paths rather than one?
They are complementary, not substitutes: foundation models inject GUI/OS knowledge through training (architecture, pre-training, SFT, RL), while agent frameworks add reasoning, memory, and action handling on top of general models without tuning. Strong systems combine both, so a taxonomy must cover each axis to map the field (§3).
