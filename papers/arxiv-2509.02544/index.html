UI-TARS-2 is a 230B-total / 23B-active MoE native GUI agent that scales online multi-turn RL via a model–data co-evolution loop, a stabilized PPO recipe, and a unified GUI+terminal+file+tool sandbox.

The most detailed public account so far of scaling online multi-turn RL for a production GUI agent and of unifying GUI with terminal/file/tool actions — directly relevant to my computer-use agent track even though the weights stay closed.

Motivation

Native GUI agents stall on four scaling problems, not a missing idea: (1) long-horizon trajectories pairing reasoning + action + environment state are far rarer and costlier than text/code corpora; (2) multi-turn online RL is unstable — rewards are sparse/delayed and credit assignment over long action sequences is hard; (3) GUI-only operation is unnatural for real workflows (data processing, software dev, sysadmin) better done via files/terminals/tools; (4) rollout environments (browsers, VMs, simulators) are fragile and crash-prone at the scale of millions of episodes. This is what breaks — not the fix.

Contribution (what’s new)

A GUI agent improves stably at scale when data and model co-evolve, long-horizon PPO is stabilized, and GUI/terminal/file/tools share one sandbox. Concretely: (a) a data flywheel that recycles every generated trajectory to the stage it best fits; (b) a stabilized multi-turn PPO recipe (value pretraining + decoupled & length-adaptive GAE + clip-higher) with asynchronous stateful rollouts; (c) a containerized sandbox with shared file system unifying GUI, terminal, files, and tools under one action space. The mechanistic why: human GUI traces are off-policy so the model never self-corrects (fixed by on-policy annotation + flywheel); plain PPO’s value estimates were found negatively correlated with rewards over long sequences (fixed by value pretraining + decoupled GAE); a shared file system lets a browser-downloaded file be processed by shell in the same instance, which GUI-only environments cannot express. The delta is integration and scaling over UI-TARS-1.5, not a new learning principle (RL machinery is imported from VAPO/VC-PPO/DAPO).

Research questions

Can a native GUI agent be scaled by (i) co-evolving training data with the model, (ii) stabilizing long-horizon online RL, and (iii) unifying GUI with system-level actions — and does the resulting RL transfer out-of-distribution?

Methodology

A single end-to-end ReAct policy with hierarchical memory acts in the sandbox over a unified action space (GUI clicks/types + SDK terminal/MCP-tool calls). It is trained through a CT → SFT → RL flywheel from a Seed-thinking-1.6 checkpoint, then per-domain RL models are fused by convex parameter interpolation.

Method details
  • Model: 532M vision encoder + MoE LLM, 23B active / 230B total params, init from Seed-thinking-1.6 (§3.1).
  • Flywheel (§2.3): each RL model generates trajectories scored by a validator V(s)∈{0,1}; high-quality → appended to SFT, low-quality → routed to CT, so no data is wasted. SFT data is collected on-policy via interactive human-in-the-loop annotation (annotators accept/override the live model’s proposed thought+action); CT uses an in-situ think-aloud protocol.
  • Multi-turn RL (§2.5): RLVR across GUI-Browsing (LLM-as-Judge reward), GUI-General (690 websites, scored by UI-TARS-2 as a generative ORM on the last 5 screenshots), and Gameplay (JS verifiers). Optimizer = PPO + value pretraining + Decoupled-GAE + Length-Adaptive GAE (α=0.05) + clip-higher.
  • Merging (§2.6): θ(merge)=Σ αₖθ(k), convex blend of per-domain RL models from a shared SFT init, exploiting linear mode connectivity.
  • Infra (§2.2): thousands of cloud VMs (Windows/Ubuntu/Android, PyAutoGUI+ADB) + a hardware-accelerated browser sandbox with fake-clock determinism; asynchronous server-based inference with streaming partially-filled rollout pools and stateful environments; W4A8 quantization for deployment.

Main result

Strong, consistent gains over its own predecessor across computer/mobile/browser: OSWorld 47.5 vs UI-TARS-1.5 42.5; WindowsAgentArena 50.6 vs 42.1; AndroidWorld 73.3 vs 64.2; Online-Mind2Web 88.2 vs 75.8 (Tab.1, p.15). RL transfers out-of-distribution even though browser tasks dominated RL: SFT→RL lifts OSWorld 43.0→47.5 and AndroidWorld 64.6→73.3 (§3.2, p.14). On games it reaches 59.8 mean normalized score (~60% of human), +35.0 over OpenAI CUA and +38.2 over Claude Computer Use (Tab.2, p.15). Cross-model comparisons are largely not head-to-head: in Tab.1 Claude/OpenAI have no OSWorld/AndroidWorld/Online-Mind2Web cells (”-”/✗), and headline † numbers (Terminal Bench 45.3, SWE-Bench 68.7, BrowseComp) use a GUI-SDK action space the baselines lack. It actually trails the best proprietary models on SWE-Bench Verified (68.7 vs Claude-4-Sonnet 72.7) and BrowseComp-en (29.6 vs o3 49.7). All numbers are single-run with no error bars except LMGame’s 3 runs; training figures lack axis values.

Implications

A concrete, reusable systems blueprint for scaling online multi-turn RL on GUI agents — asynchronous stateful rollouts, streaming pools, value pretraining, decoupled/length-adaptive GAE — plus the practical finding that RL trained mostly on browser tasks transfers OOD to desktop/mobile. Unifying GUI with terminal/file/tool actions in one sandbox is a design pattern beyond pure click-simulation agents. Empirical observations (PPO > GRPO here, rising not collapsing entropy, an imperfect ORM that still trains well) are usable even though weights, code, and data stay closed.

Limitations

Tech-report rigor: single runs, no ablations isolating flywheel vs sandbox vs RL-recipe contributions. Cross-model comparisons are not head-to-head — many Claude/OpenAI cells are missing and † numbers use a GUI-SDK action space baselines lack; it trails the best proprietary models on SWE-Bench and BrowseComp-en. Fully closed: no weights, training code, or data; built on closed Seed-thinking-1.6; figures lack axis scales so quantitative training claims are unverifiable. Self-judged rewards (ORM / LLM-as-Judge) on open-ended GUI-General tasks risk reward hacking; high ORM false-positive rate noted. Long-horizon planning ceiling (weak on Tetris/Sokoban) attributed to the backbone, not addressed.

Critiques and Questions

  1. The headline “outperforms Claude/OpenAI” rests partly on non-comparable cells and a GUI-SDK action space the baselines lack — what do the numbers look like restricted to GUI-only, apples-to-apples?
  2. No ablation isolates the three contributions (flywheel vs sandbox vs stabilized RL); which one actually drives the gains?
  3. Self-judged ORM rewards with a high false-positive rate plus rising entropy are a setup where reward hacking could surface later in training — how stable is this past the reported horizon?
  4. The “value estimates negatively correlated with rewards” finding is the most interesting mechanistic claim; is it GUI-specific or a general long-horizon PPO pathology?

Glossary

OSWorld, AndroidWorld, Online-Mind2Web, and BrowseComp are the main eval suites. GRPO and GAE appear in the RL recipe. stateful rollout and GUI-SDK are core system terms.
