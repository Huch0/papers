Motivation

Computer-use agents (CUAs) remain unreliable on long-horizon, complex tasks where small step-level errors compound and produce high outcome variance — the same agent often succeeds on one attempt and fails catastrophically on another (p.1). Prior single-rollout scaling (e.g. step-wise best-of-N) yields only limited gains because it commits the rollout to one agent plan and can over-commit to a harder route (§2.2). Scaling wide — many parallel rollouts — is more promising, but exposes a bottleneck: reliably evaluating and selecting among information-dense, multimodal, long-horizon trajectories is hard, and many tasks admit multiple valid solutions that scripted evaluators cannot judge (p.1–2).

Contribution (what’s new)

The genuine delta is treating wide scaling as a trajectory-selection problem and making selection tractable via representation. Behavior Judge (BJudge) first converts each raw trajectory into a “behavior narrative” — a per-transition sequence of facts describing what the agent did and how it changed the environment — then runs a single comparative (multiple-choice) judgment over all candidate narratives at once (§3.1–3.2). The design must take this form because the baseline failure is evaluation: raw trajectories are too dense to judge directly (most pixels are irrelevant), and independent scoring of candidates plateaus — so the judge must (a) see a compact, faithful action-effect representation and (b) compare candidates against each other rather than rank them in isolation. A second contribution is Agent S3, an improved base framework that adds a natively-integrated coding agent and replaces hierarchical planning with a flat (worker-only) policy (§3.3).

Research questions

How can test-time scaling be made effective for long-horizon CUAs? Concretely (§4): (1) how does BJudge compare to CUA baselines; (2) how do judging strategies fare under equal rollout budget; (3) do behavior narratives beat other trajectory representations; (4) how accurate is the judge and what are its failure modes; (5) how should compute be split across workers; (6) how to pick a mixture-of-models ensemble; (7) does it generalize to other OSes.

Methodology

Generate multiple independent rollouts from one or more base policies (Agent S3 instances), compress each into a behavior narrative, and have a VLM judge comparatively select one winner. The judge never sees raw trajectories — only narratives plus the first/last screenshot.

Method details
  1. Generate candidates — Sample M base policies with stochastic decoding to get a candidate set C = ∪ {τ_m^(n)}, capturing both within-model variance (n) and cross-model diversity (m). Objective: select argmax over R(τ,I) (§3.2).
  2. Behavior Narrative Generation — For each transition (s_i, a_i, s_{i+1}) a VLM generator G derives facts φ_i = G(s_i, a_i, s_{i+1}). Pointer actions get visual augmentation: a marker at the pre-action coordinate and a zoomed, outlined crop at the post-action coordinate for pixel-level verification; the after-screenshot is taken 3s later to catch delayed changes (§3.1).
  3. Assemble narrative — Narrative τ̃ = (s_0, φ_0, …, φ_{T-1}, s_T) keeps only task-relevant changes plus initial/final screenshots for grounding (§3.1).
  4. Comparative Behavior Evaluation — A VLM evaluator E receives all candidate narratives and selects one best via a single-round multiple-choice (MCQ) prompt — more informed than independent ranking, faster/cheaper than tournament-style pairwise. The prompt requires citing and contrasting facts (§3.2, App. H).

Agent S3 base framework (§3.3, App. D): a flat GUI policy π(a_t | I, h_t) that can either emit a GUI action or invoke a coding agent. A code call runs a bounded inner loop (budget B); at inner step k the coding agent conditions on (I, o_t, F_{1:k-1}) where each feedback tuple F_k = (status, return_code, stdout, stderr) comes from executing Python/Bash in a sandboxed VM, terminating on DONE/FAIL or k=B. On termination a summary plus a verifiable inspection checklist is appended to the GUI agent’s history. Unlike CoAct-1, the coding agent is natively in the action space (no AutoGen, no orchestrator). The flat policy replans online instead of committing to a subgoal list.

Setup: OSWorld 361-task subset (8 Google-Drive tasks omitted), 100-step budget. Best config = 5 rollouts each from GPT-5 and Opus 4.5, Opus generates facts, GPT-5 selects (§4.1).

Main result

On OSWorld 100-step, Agent S3 + BJudge with a GPT-5 + Opus 4.5 ensemble (N=10) reaches 72.6% SR, surpassing prior SoTA GTA1 at 63.4% (+9.2% absolute) and human-level 72.36% (Tab.1, Fig.1). The base framework alone is already strong: Agent S3 with GPT-5 hits 62.6% (averaged over 10 runs), vs Agent S2 at 48.8% (Tab.1). BJudge lifts single-model rollouts substantially: +7.3% for GPT-5 (62.6→69.9%) and +10.4% for GPT-5 Mini (49.8→60.2%) at N=10 (Tab.1).

Key findings

  • Behavior narratives beat all other trajectory representations: 60.2% vs naive captioning 56.8%, screenshot-only 56.0%, and trajectory-summary 55.0% (10 GPT-5 Mini rollouts, +3.4% over best baseline) (Tab.3): per-transition action-effect facts carry more signal than raw screenshots or holistic summaries.
  • Comparative selection is what scales — WebJudge (independent ranking) gives little benefit over the rollout average and plateaus around N=4, dropping by N=10, whereas BJudge keeps improving (Fig.4): trajectories must be compared against each other, not scored in isolation.
  • Judge accuracy is 78.4% on the 159-task “Judge Subset” (tasks with at least one correct and one incorrect rollout) against OSWorld scripts, rising to 92.8% under human evaluation because OSWorld scripts only accept one predefined solution (Tab.4): a chunk of apparent judge errors are actually flawed benchmark graders.
  • Remaining 12 judge failures split into 8 behavior-narrative hallucinations (VLM misses fine-grained text, e.g. a negative sign) and 4 Code–GUI handoff failures (GUI agent overwrites the coding agent’s correct changes; rich GUI narratives bias the judge toward the failing run) (§4.5): the judge is vulnerable to “reasonable-sounding” narratives (App. G).
  • Model diversity drives coverage and the best SR: GPT-5 + Opus 4.5 gives the top SR (71.6% at N=4) while the most diverse “All” mix gives the highest Pass@N coverage (80.5%) (Tab.5): pair strong, capability-diverse models for rollouts.
  • Compute allocation is budget-dependent — at small total budgets a single agent is best, but larger N pays off as budget grows (N=10 gives +6.38% at budget 1000) (Fig.5): moderate N for tight budgets, large N when resources allow.
  • Agent S3’s design choices are individually efficient: removing hierarchy gives +9.1% SR with −43.8% LLM calls / −52.1% time, and adding the coding agent reaches +13.8% SR, −52.3% calls, −62.4% time vs Agent S2 (Tab.2): the flat policy is both better and cheaper.

Implications

Reframes CUA reliability as a wide-scaling selection problem and gives a practical recipe (compress to narratives, compare via MCQ). It builds directly on Agent S2 (Agashe et al., 2025b) as the base framework and CoAct-1 (Song et al., 2025) for coding-as-actions, and contrasts with single-rollout step-wise scaling (GTA1, Yang et al., 2025b) and web-only LLM/VLM judges like WebJudge (Xue et al., 2025) and Mind2Web 2 (Gou et al., 2025). It generalizes zero-shot: BJudge (N=3) adds +6.4% on WindowsAgentArena (50.2→56.6%, 100-step) and +3.5% on AndroidWorld (68.1→71.6%) over Agent S3 (Tab.6, Tab.7). Threads to chase: cheap-rollout/expensive-judge economics — the judge modules cost ~5× less than rolling out, and a pure open-source Qwen3-VL pipeline still gains +7.6% (App. B–C); infrastructure for snapshot/duplication so independent parallel rollouts are cheap and side-effect-isolated; fixing the Code–GUI handoff verification failure that biases the judge.

Limitations

BJudge assumes access to an agent that can produce multiple independent rollouts from the same initial state — standard in benchmarks and snapshot-capable VMs, but violated when running on a user’s live desktop without isolation (concurrent rollouts interfere) (§5). Even with separate VMs, tasks touching shared online state (email, cloud storage, shopping carts) can introduce cross-run interference (§5). Judge quality is bounded by VLM visual understanding — narrative-generation hallucinations on fine-grained details (e.g. a negative sign) cause selection errors that zooming doesn’t fix (§4.5, App. G). Wide scaling multiplies inference cost: each task needs N full rollouts plus narrative generation and judging (App. B).

Critiques and Questions

  1. The headline 72.6% barely clears human 72.36% (~0.24%) and uses a two-model N=10 ensemble; the single-config robustness/variance of that exact number is not reported, so “surpasses human” is a thin margin.
  2. The most honest accuracy figure (92.8% human-aligned) implies OSWorld scripts mis-grade some tasks — which also casts uncertainty on the comparability of the 72.6% vs prior SoTA numbers, since all share the same imperfect scripted evaluator.
  3. Coverage (Pass@N up to 80.5%) is well above realized SR, so selection — not generation — is now the ceiling; how much of the remaining gap is judge limitation vs benchmark-grader noise is Unclear.
  4. Cost-normalized comparisons against single-rollout baselines are relegated to the appendix; a fair SR-per-dollar curve vs GTA1 would clarify whether wide scaling is practically preferable.
Why does BJudge compress trajectories into behavior narratives instead of judging raw screenshots, and why use comparative (MCQ) selection rather than independent scoring?
Raw long-horizon trajectories are information-dense and mostly task-irrelevant, so a judge struggles to interpret them directly; per-transition action-effect facts isolate what actually changed (Tab.3: narratives 60.2% vs screenshot-only 56.0%). Comparative MCQ selection lets the judge contrast candidates against each other, which keeps scaling with N, whereas independent ranking (WebJudge) plateaus near N=4 and drops by N=10 (Fig.4).
