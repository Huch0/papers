Surfer 2 shows that a purely vision-based, hierarchical orchestration of off-the-shelf frontier models — planner, executor, and validator — reaches state-of-the-art across web, desktop, and mobile computer-use benchmarks with no task-specific fine-tuning.

Central data point for the computer-use-agent track: it argues that system design and orchestration — not bigger or fine-tuned models — can deliver human-level cross-platform GUI control from screenshots alone.

Motivation

Building agents that generalize across web, desktop, and mobile is unsolved because prior systems lean on environment-specific interfaces — DOM parsers for web, accessibility trees for mobile, specialized APIs for desktop, which limits cross-platform deployment. The dominant alternative path — training ever-larger models with minimal scaffolding to push agentic capability — couples progress to expensive model training and domain-specific data rather than reusable system design.

Contribution (what’s new)

The genuine delta is the claim, backed by results, that careful orchestration of fixed, general-purpose models substitutes for model-level learning: coordination and self-correction at the system level, with no gradient updates, reach SOTA. The design must take this form because the failures it targets are systemic, not perceptual — a hierarchical planner/executor split, persistent shared environment state, and a VLM-as-a-Judge validation loop that catches premature termination address error propagation and long-horizon drift that a single monolithic model call cannot. Critically, the agent operates purely from screenshots with raw pixel-coordinate actions (no DOM/a11y tree), making one architecture portable across all three environments.

Research questions

Can a single, vision-only, hierarchical agent architecture — assembled from existing frontier models without fine-tuning — match or beat specialized, trained systems across web, desktop, and mobile computer-use benchmarks? And which system-level mechanisms (planning, persistent context, validation, test-time scaling) actually drive that performance?

Methodology

Surfer 2 is a three-component agent over an unaltered-screenshot interface. An optional Orchestrator plans and replans; a Navigator (an upgraded Surfer-H, ReAct loop) executes; a Localizer (Holo1.5) grounds textual element references to pixel coordinates; a Validator (VLM-as-a-Judge) gates answers. Architecture is flexibly toggled per task: the Orchestrator is enabled for long-horizon web tasks and bypassed for OSWorld/AndroidWorld.

Method details

Design philosophy (§3.1) — five principles: separation of concerns (Orchestrator plans / Navigator executes); automatic hierarchical context (each component sees relevant global state — goal, plan, completed work, current subtask); shared environment state (browser sessions / open apps persist across subtasks); explicit multi-stage validation; chain-of-thought in all modules except the Localizer.

  1. Orchestrator (optional, §3.2) — Plans, coordinates, communicates. Decomposes the goal into verifiable subtasks, delegates to the Navigator, evaluates outcomes + Validator feedback, and replans on failure. Maintains persistent memory (objective, plan, status, full interaction history, latest screenshots). Four actions: create plan, replan, delegate, answer.
  2. Navigator (§3.3) — Upgraded Surfer-H, now web+desktop+mobile. ReAct loop emitting structured output: a note (info from latest observation), a thought (reasoning), and an action. Perceives only via screenshots; issues constrained keyboard/mouse/touch controls.
  3. Localizer (§3.4) — Holo1.5 model that converts textual element descriptions (e.g. 'blue submit button') into clickable (x,y) coordinates — visual grounding, the critical reliability bottleneck.
  4. Validator (§3.5) — VLM-as-a-Judge inspecting the full trace (task spec, reasoning, actions, proposed answer, last k screenshots). Operates at two levels: inside the Navigator (gate each answer; on fail, resume with feedback) and at the Orchestrator (accept / refine / replan).

Model configuration (Tab.1, p.9): mixed frontier models per benchmark. WebVoyager: Orchestrator o3, Navigator policy Claude Sonnet 4.5, Judge GPT-4.1, Localizer Holo1.5 7B. WebArena: o3 / Sonnet 4.5 / o3 / Holo1.5 72B. OSWorld: no Orchestrator, Sonnet 4.5 / o3 / Holo1.5 72B. AndroidWorld: no Orchestrator, o3 / o3 / Holo1.5 72B.

Evaluation (§4): four benchmarks, no task-specific fine-tuning. WebVoyager (590-task Magnitude subset, 15 live websites, 3x GPT-4o majority-vote judge); WebArena (812 tasks, 6 self-hosted sites, 5x GPT-4.1 majority-vote judge plus URL/HTML checks, with 71 manually corrected tasks); OSWorld (369 Ubuntu tasks, strict Foundation E2E GUI category — human-performable GUI ops only, no code/API shortcuts; programmatic checks); AndroidWorld (116 tasks, screenshot-only — deliberately not using the available accessibility tree; ADB state checks scored 1). Step budgets: web Orchestrator ≤20 / Navigator ≤50; OSWorld 15–100 (memory cleared at cap without env reset); AndroidWorld 150.

Main result

Surfer 2 sets a new SOTA on all four benchmarks at pass@1 without fine-tuning, and with multiple attempts exceeds the human baseline on every benchmark where one is reported (abstract, §5, Fig.1).

(Note: for context, the looser OSWorld “All” category — which permits code/API shortcuts — is led by Agent S3 at 69.9%; Surfer 2 reports only the stricter E2E regime.)

Key findings

  • The Localizer is the dominant performance lever (§5.1, §5.3, Fig.7): swapping Holo1.5 for UI-TARS 7B drops WebVoyager 97.1→94.7%, OSWorld 60.1→56.9% (7B Holo1.5: 58.4%), and AndroidWorld 87.1→81.9% (−5.2%) — small-target/icon grounding errors cascade into wrong actions.
  • Test-time scaling mainly expands task coverage, not single-run reliability (§5.2, §5.4): WebArena 69.6→84.9% (pass@10) and AndroidWorld 87.1→93.1% (pass@3); the diversity of successful paths suggests residual failures are local exploration traps / stochastic perception, not systematic reasoning errors.
  • Persistent context across subtasks cuts navigation steps by 30–40% (§6), and multi-stage validation intercepts 15–20% of errors before they propagate (§6) — direct evidence that the system-level mechanisms, not just the base models, carry the gains.
  • Prompt engineering is surprisingly impactful — minor wording changes swing accuracy 5–10% (§6), and judge model variance stays substantial even at temperature 0, necessitating multi-sampling.
  • Performance degrades sharply with task difficulty on mobile (§5.4, Fig.6): AndroidWorld Easy 98.4% / Medium 86.1% / Hard 52.6%, with multi-app (37.5%), memorization (50%), and transcription (50%) the hardest categories — long-horizon temporal memory and cross-app reasoning remain weak.
  • Surfer 2 solved several tasks humans labeled infeasible (§5.3, App.B.1) (e.g. switching Chrome’s UI language to Korean by opening a terminal and running system commands via visual interaction), showing adaptive reasoning beyond predefined task boundaries.

Implications

The result reframes computer-use progress as an orchestration problem: a fixed-model, modular system generalizes across web/desktop/mobile, extending the plan-and-act and ReAct lineage and building directly on Surfer-H / Holo1.5 and Agent S / Agent S3 hierarchical-agent work. It positions specialized localization models (Holo1.5, UI-TARS) as first-class system components rather than afterthoughts. The authors flag cost ($1–5 per complex Orchestrator task with frontier reasoning models), variance, and speed as the binding constraints, and announce a planned family of smaller specialized models to reach Pareto-optimal cost-efficiency — the natural next read alongside how the high pass@k coverage could seed RL fine-tuning of these behavioral patterns.

Limitations

  • Stochastic outputs force expensive multi-sampling for reliability; Orchestrator cost reaches $1–5 per complex task (§6).
  • Even SOTA localizers fail on 5–8% of UI elements due to dynamic content / ambiguous descriptions, and tasks beyond ~50 steps hit context-window limits and compounding errors (§6).
  • The LLM-based evaluation itself carries 5–10% error rates, complicating benchmarking — and the team’s own WebArena harness changes (71 corrected tasks, custom judge ensemble) make cross-paper comparison harder.
  • Long-horizon temporal perception and cross-app workflows remain weak (AndroidWorld Hard 52.6%, multi-app 37.5%; §5.4).

Critiques and Questions

  1. The pass@1 headline mixes different model stacks per benchmark (Tab.1), so it measures the best-tuned configuration per environment rather than one fixed system — the “unified architecture” claim is about the framework, not a single deployed agent.
  2. WebArena’s 69.6% relies on a custom judge ensemble plus 71 hand-corrected tasks; the un-corrected, LLM-judge-only ablation is 67.4%, so part of the headline gap over prior SOTA (65.4%) is methodology, not capability. How comparable is this to numbers reported by others?
  3. “Exceeds human performance” holds only at pass@3–pass@10 multi-attempt budgets; at pass@1 the agent is still below human on WebArena (69.6 vs 78.2) and OSWorld (60.1 vs 72.4).
  4. Cost ($1–5/task) and the heavy multi-sampling make the impressive pass@k numbers economically distant from the pass@1 numbers a real deployment would see.
The paper claims SOTA without fine-tuning — so what is doing the work, and what is the single biggest component lever they identify?
System-level orchestration of fixed frontier models (hierarchical planner/executor split, persistent shared context worth a 30–40% step reduction, and a validation loop catching 15–20% of errors) substitutes for model training. The single biggest lever is the Localizer: replacing Holo1.5 with UI-TARS drops accuracy by up to 5.2% (AndroidWorld), because pixel-grounding errors on small/icon targets cascade into wrong actions.
